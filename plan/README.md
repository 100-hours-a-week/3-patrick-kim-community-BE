## 방법론

모든 작업은 동일한 사이클을 따른다:

```
① 질문    — "우리 서비스는 ___에서 버티는가?"
② 측정    — k6 + Grafana로 현재 상태 정량화
③ 병목 발견 — 어디서, 왜 터지는가?
④ 가설    — "___를 하면 ___가 개선될 것이다"
⑤ 실험    — 가설 기반 코드/인프라 변경
⑥ 재측정  — 동일 조건 Before/After 비교
⑦ 결론    — 수치로 증명, 실패 시 원인 분석 후 새 가설
```

## 현재까지 달성한 수치 (Phase 10-3 완료)

| 항목 | 수치 | 조건 |
|------|------|------|
| 읽기 API p95 | 178ms | 300 VUs, **읽기만** |
| 투표 API p95 | 658ms | 50 VUs, **Hot Spot** |
| RPS | 308 | 300 VUs, 읽기만 |
| 에러율 | 0% | 300 VUs |

**핵심 문제**: 위 수치는 전부 **읽기 따로, 쓰기 따로** 측정한 것. 실제 서비스는 동시에 발생한다.
