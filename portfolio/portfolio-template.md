# SP1. 동시성 → 비동기 투표 아키텍처 진화

## 슬라이드 1: 문제 — Hot Spot 동시성

```
[문제] 50명 동시 투표 시 순차 Lock 대기 → p95 1.73s
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Pessimistic Lock 동작:
  t1  [User 1] ████ Lock 획득 + 처리
  t2  [User 2]      ████ Lock 획득 + 처리
  t3  [User 3]           ████ Lock 획득 + 처리
  ...
  t50 [User 50]                                    ████ ← 49명 기다린 후 처리

  → 마지막 사용자는 49명 × 35ms = 1.73s 대기


3가지 전략 비교 (50 VUs 동시 투표):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
            성공률      p95       특징
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Pessimistic  100%      1.73s     느리지만 확실
Optimistic    16%      885ms     충돌 다발
Atomic+Retry  94%      2.98s     데드락 + 백오프
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ Hot Spot에서는 재시도보다 순차처리가 효율적
→ 그러나 1.73s는 UX 한계 → 아키텍처 변경 필요
```

## 슬라이드 2: 해결 — 비동기 아키텍처 + MQ 선택

```
[해결] DB Lock 제거 → Redis 즉시 반영 + SQS 비동기 DB 저장
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Before (동기 Pessimistic Lock):
  Client → API → [SELECT FOR UPDATE] → INSERT → COMMIT → Response
                      ↑ Lock 대기 (병목)             p95 = 1,730ms


After (Hybrid Async):
  Client → API → DB 중복체크 → Redis ZINCRBY → SQS 전송 → Response
                 (HikariCP 30)  (Pipeline 1RTT) (Fire&Forget) p95 = 658ms
                                                    ↓
                                    SQS Queue → Consumer → DB 저장 (비동기)


MQ 선택 (SQS Standard Queue):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
          Kafka     RabbitMQ    SQS ✅
처리량    수백만/초  수만/초     무제한
운영비용  $200+/월  EC2 관리    $1/월
운영부담  높음      중간        없음 (완전관리형)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
"100byte 투표 메시지에 분산 로그 시스템은 불필요"
```

# SP2. 외부 서비스 장애 대응 — Graceful Degradation

## 슬라이드 1: 문제 — 외부 서비스 의존 리스크

```
[문제] Redis/SQS 장애 시 투표 전체가 중단될 위험
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

정상 흐름:
  Client → API → DB 중복체크 → Redis → SQS → Response
                                 ↑        ↑
                              장애 지점  장애 지점

"챌린지 마감 직전에 Redis가 죽으면?"
→ 투표 불가 → 사용자 이탈 → 콘테스트 무효화 위험

목표:
  ❌ "외부 서비스 죽으면 서비스도 죽는다"
  ✅ "외부 서비스 죽으면 느려지지만 계속 동작한다"
```

## 슬라이드 2: 해결 — 5단계 방어선

```
[해결] 5단계 Fallback 체인
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Layer 5  ┌─ 정합성 스케줄러 (5분 주기 Redis↔DB 비교) ─ 최종 안전망
Layer 4  ├─ DLQ 재처리 (30분 주기, 실패 메시지 자동 복구)
Layer 3  ├─ Circuit Breaker (Redis 실패율 50%→OPEN→Fallback)
Layer 2  ├─ SQS 동기 Fallback (전송 실패→syncWriteToDb)
Layer 1  └─ DB Unique (entry_id, member_id) ─────── 최종 방어선


시나리오별 Fallback 경로:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
정상:     API → Redis → SQS → Response (p95 138ms)
SQS 장애:  API → Redis → DB 직접 저장 (p95 337ms)
Redis 장애: API → Circuit Breaker → Pessimistic Lock (p95 1,022ms)
둘 다 장애: API → DB-only 모드 (p95 792ms)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

핵심 설계: Pessimistic Lock 동기 경로를 삭제하지 않고
         → Fallback 자산으로 재활용 (이미 100% 정합성 검증 완료)
```

## 슬라이드 3: 결과 — 장애 시나리오 부하 테스트 실증

```

시나리오       총 요청   성공률   avg      p95       Fallback
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
정상           2,738    94.7%    52ms    138ms     Redis + SQS

※ 실패분은 중복 투표(400 에러)이며, 서버 에러 아님


응답시간 비교 (p95):

  정상        ██                                   138ms
  SQS 장애    █████                                337ms  (2.4배)
  동시 장애   ████████████████                     792ms  (5.7배)
  Redis 장애  █████████████████████                1,022ms (7.4배)

  → 최악의 경우에도 서비스 유지 실증
  → "느려지지만 죽지 않는다"를 수치로 증명


설계 원칙:
  Source of Truth = DB (Redis 유실 → DB에서 재구축)
```

# SP3. 데이터 기반 성능 최적화 — 측정 → 분석 → 개선

## 슬라이드 1: 읽기 최적화 — 4단계 순차 적용

```
[읽기 API 최적화] p95: 4.76s → 130ms (97% 개선)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

4단계 최적화 (누적 개선):

  1. 커넥션 풀 튜닝                      → 획득 대기 89%↓
     HikariCP max 10→30
     Grafana에서 풀 포화 발견

  2. Pageable 적용                        → p95 81%↓
     Stream.limit() → DB LIMIT
     100K row → 10 row 로드

  3. Fetch Join (N+1 해결)                → p95 73%↓
     p6spy에서 61개 쿼리 발견
     QueryDSL JOIN FETCH → 1개 쿼리

  4. 복합 인덱스                          → p95 38%↓
     EXPLAIN filesort 발견
     (challenge_id, vote_count DESC)


최종 결과:
  p95:  4.76s  →  130ms   (97% 개선)
  RPS:  32.9   →  308     (9.4배 증가)
```

## 슬라이드 2: 혼합 부하 — 가설 실패에서 배운 것

```
[혼합 부하 최적화] 가설 검증 실패 → 데이터 분석 → 재도전 성공
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Round 1: 병목 발견
  읽기 70% + 투표 30% 혼합 부하
  → 100 VUs에서 HikariCP 포화 (30/30)
  → Pending 170개, 커넥션 대기 최대 12.4초
  → 전체 p95 = 5,439ms

Round 2: 가설 실패 ✗
  가설: "투표 DB 호출을 줄이면 개선된다"
  변경: DB exists → Redis SADD (DB 2회→1회)
  결과: p95 5,439ms → 5,835ms (+7% 악화)

  데이터 분석:
  ┌──────────────────────────────────────┐
  │  읽기 API → 전체 DB 호출의 78%      │ ← 진짜 병목
  │  투표 API → 전체 DB 호출의 22%      │
  └──────────────────────────────────────┘
  투표 1회 줄여봐야 전체에서 9% → 미미한 영향

Round 3: 올바른 가설 ✓
  가설: "읽기 캐시로 DB 호출 78% 제거"
  변경: Caffeine L1 캐시 (TTL 10s, Controller 레벨)
  핵심: 캐시 히트 시 DB 커넥션 미획득 (Service @Transactional 우회)
```
