# 동시성 제어 - Race Condition 해결

## 1. 문제 발견 과정

### 왜 동시성을 고려해야 했는가?

PetStar는 투표 기반 랭킹 시스템입니다. **투표 결과가 순위를 결정**합니다.

```
PetStar 서비스 특성:

1. 투표 결과 = 랭킹 = 콘테스트 우승자 결정
   → 1표라도 누락되면 안 됨

2. 챌린지 종료 직전 투표 집중
   → 동시 투표 발생 가능성 높음

3. 인기 Entry에 투표 집중 (Hot Spot)
   → 같은 Entry에 동시 투표
```

**단순 기능 테스트에서는 발견되지 않는 문제**입니다. 부하 테스트를 통해 동시성 문제를 검증했습니다.

### 발견된 두 가지 Race Condition

부하 테스트에서 **투표 50번 → 결과 48개** 같은 현상이 발생했습니다. 코드를 분석한 결과 두 가지 Race Condition을 발견했습니다.

#### 1-1. Check-then-Act Race Condition (중복 투표)

**"확인 후 행동" 패턴의 함정**

```java
// 의도: 중복 투표 방지
if (voteRepository.existsByEntryIdAndMemberId(entryId, memberId)) {
    throw new DuplicateVoteException();
}
// ← 이 시점에 다른 스레드가 끼어들 수 있음!
voteRepository.save(vote);
```

**왜 문제인가?**

"확인"과 "행동" 사이에 **원자성이 보장되지 않습니다**.

```
시간  Thread A                    Thread B
────────────────────────────────────────────────────
t1    existsBy → false
t2                               existsBy → false
t3    save(vote) ✓
t4                               save(vote) ✓ ← 중복!
────────────────────────────────────────────────────

결과: 같은 사용자가 같은 Entry에 2번 투표됨
```

#### 1-2. Lost Update (투표 수 누락)

**Read-Modify-Write 패턴의 함정**

```java
// 의도: 투표 수 증가
entry.increaseVoteCount();  // 내부: voteCount = voteCount + 1
```

이 코드는 실제로 **3단계**로 실행됩니다:

```
1. READ:   메모리에서 voteCount 값 읽기
2. MODIFY: 읽은 값에 1 더하기
3. WRITE:  새 값을 메모리에 쓰기
```

**왜 문제인가?**

```
시간  Thread A                    Thread B
────────────────────────────────────────────────────
t1    READ voteCount → 10
t2                               READ voteCount → 10
t3    MODIFY 10 + 1 = 11
t4    WRITE voteCount = 11
t5                               MODIFY 10 + 1 = 11
t6                               WRITE voteCount = 11
────────────────────────────────────────────────────

기대값: 12
실제값: 11 (Thread A의 업데이트가 사라짐!)
```

**PetStar에서의 영향**: 투표 1표 = 랭킹 순위 변동 가능 → **절대 허용 불가**

## 2. 해결책 탐색: 왜 간단한 방법은 안 되는가?

### 첫 번째 시도: synchronized

Java 개발자가 처음 떠올리는 방법입니다.

```java
public synchronized void vote(Long entryId, Integer memberId) {
    // 동기화된 투표 로직
}
```

**왜 안 되는가?**

`synchronized`는 **JVM 레벨의 락**입니다. 같은 JVM 내의 스레드끼리만 상호 배제됩니다.

```
단일 서버라면 동작:

┌────────────────────────────────┐
│  JVM                           │
│  ┌─────────┐    ┌─────────┐   │
│  │Thread A │    │Thread B │   │
│  │  대기   │ ←→ │  실행   │   │
│  └─────────┘    └─────────┘   │
│       synchronized로 조율      │
└────────────────────────────────┘
```

```
스케일 아웃 환경에서는 실패:

┌─────────────┐     ┌─────────────┐
│   JVM A     │     │   JVM B     │
│ synchronized│     │ synchronized│
│   (독립적)  │     │   (독립적)  │
└──────┬──────┘     └──────┬──────┘
       │                   │
       │  서로 모름!       │
       │                   │
       └───────┬───────────┘
               ▼
        ┌─────────────┐
        │   MySQL     │  ← 실제 데이터는 여기
        └─────────────┘
```

**결론**: 영속 데이터의 동시성은 **데이터가 있는 곳(DB)**에서 제어해야 합니다.

### 두 번째 의문: 트랜잭션으로 해결되지 않나?

"@Transactional 붙이면 되는 거 아닌가요?"

```java
@Transactional
public void vote(Long entryId, Integer memberId) {
    Entry entry = entryRepository.findById(entryId);
    entry.increaseVoteCount();  // 여전히 Lost Update 발생!
}
```

**MySQL 기본 격리 수준: REPEATABLE READ**

REPEATABLE READ가 보장하는 것:
- **같은 트랜잭션 내**에서 같은 데이터를 읽으면 **같은 값**을 얻음
- Non-Repeatable Read 방지

REPEATABLE READ가 보장하지 **않는** 것:
- **다른 트랜잭션**이 같은 데이터를 수정하는 것 방지
- **Lost Update 방지** ← 우리 문제!

```
Transaction A                    Transaction B
────────────────────────────────────────────────────
BEGIN
SELECT voteCount → 10
                                BEGIN
                                SELECT voteCount → 10
UPDATE voteCount = 11
COMMIT
                                UPDATE voteCount = 11  ← A의 업데이트 덮어씀!
                                COMMIT
────────────────────────────────────────────────────
```

**결론**: 트랜잭션 격리 수준만으로는 부족합니다. **명시적인 락** 또는 **원자적 쿼리**가 필요합니다.

## 3. 해결 방법 비교: 세 가지 전략의 트레이드오프

DB 레벨에서 동시성을 제어하는 세 가지 방법을 검토했습니다.

### 3-1. Pessimistic Lock (비관적 락)

**철학**: "충돌이 발생할 것이다" → 미리 락을 건다

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT e FROM Entry e WHERE e.id = :entryId")
Optional<Entry> findByIdWithPessimisticLock(@Param("entryId") Long entryId);
```

**동작 원리**:
```sql
SELECT * FROM entry WHERE entry_id = ? FOR UPDATE;
-- 이 행에 대한 배타적 락 획득
-- 다른 트랜잭션은 락 해제까지 대기
```

```
시간  Transaction A                Transaction B
────────────────────────────────────────────────────
t1    SELECT ... FOR UPDATE
      (락 획득 ✓)
t2                                SELECT ... FOR UPDATE
                                  (락 대기...)
t3    UPDATE voteCount = 11
t4    COMMIT (락 해제)
t5                                (락 획득 ✓)
t6                                SELECT voteCount → 11
t7                                UPDATE voteCount = 12
t8                                COMMIT
────────────────────────────────────────────────────
결과: 정확히 12 (Lost Update 없음!)
```

**장점**: 확실한 동시성 보장, 로직이 단순
**단점**: 락 대기 시간 발생, 처리량 감소

### 3-2. Optimistic Lock (낙관적 락)

**철학**: "충돌은 드물 것이다" → 충돌 시 재시도

```java
@Entity
public class Entry {
    @Version
    private Long version;  // 버전 관리
}
```

**동작 원리**:
```sql
-- 읽을 때 버전 기억
SELECT entry_id, vote_count, version FROM entry WHERE entry_id = 1;
-- → vote_count=10, version=5

-- 업데이트 시 버전 확인
UPDATE entry
SET vote_count = 11, version = 6
WHERE entry_id = 1 AND version = 5;  -- 버전 일치해야 성공

-- 다른 트랜잭션이 먼저 업데이트했다면 version=6이 되어 있음
-- → WHERE 조건 불일치 → 0 rows affected → OptimisticLockException
```

**장점**: 락 대기 없음, 읽기 성능 좋음
**단점**: 충돌 시 재시도 필요, 충돌이 많으면 성능 저하

### 3-3. Atomic Update + UK 제약조건

**철학**: DB의 원자성을 직접 활용

```java
// 중복 투표: UK 제약조건으로 DB에서 방지
// Vote 테이블에 UNIQUE(entry_id, member_id)
try {
    voteRepository.saveAndFlush(vote);
} catch (DataIntegrityViolationException e) {
    throw new DuplicateVoteException();
}

// Lost Update: 원자적 UPDATE 쿼리
@Query("UPDATE Entry e SET e.voteCount = e.voteCount + 1 WHERE e.id = :id")
int incrementVoteCount(@Param("id") Long id);
```

**동작 원리**:
```sql
-- 원자적 증가: DB가 한 번에 처리
UPDATE entry SET vote_count = vote_count + 1 WHERE entry_id = ?;

-- 중복 방지: UK 위반 시 에러
INSERT INTO vote (entry_id, member_id) VALUES (?, ?);
-- → Duplicate key error (UK 위반)
```

**장점**: 가장 빠름, DB 레벨 보장
**단점**: 복합 연산(INSERT + UPDATE)에서 데드락 가능

### 전략 비교 요약

| 기준 | Pessimistic | Optimistic | Atomic |
|------|-------------|------------|--------|
| 충돌 처리 | 사전 방지 (락) | 사후 감지 (재시도) | DB 위임 |
| 처리량 | 낮음 | 높음 (충돌 적을 때) | 높음 |
| 구현 복잡도 | 낮음 | 중간 | 중간 |
| Hot Spot에서 | 순차 처리 | 재시도 폭주 | 데드락 위험 |

## 4. 부하 테스트: 이론 vs 현실

### 테스트 설계

**"어떤 전략이 PetStar에 적합한가?"를 판단하기 위해 실제와 유사한 시나리오를 테스트했습니다.**

```
테스트 시나리오: Hot Spot 상황 시뮬레이션

- 도구: k6
- 동시 요청: 50명이 동시에 같은 Entry에 투표
- 기대 결과: voteCount +50, Vote 레코드 50개
- 핵심 검증: 데이터 정합성 + 응답 시간
```

**왜 "같은 Entry에 동시 투표"인가?**

PetStar에서 가장 치열한 상황은 **인기 있는 반려동물에 투표가 집중**되는 Hot Spot입니다. 이 상황에서 각 전략이 어떻게 동작하는지 검증해야 합니다.

### 1차 테스트 결과 (재시도 로직 없음)

| 전략 | 성공률 | p95 응답시간 | 데이터 정합성 |
|------|--------|-------------|--------------|
| **Pessimistic** | **100%** | 1.73s | 50/50 ✓ |
| Optimistic | 16% (8/50) | 885ms | 8/50 ✗ |
| Atomic | 12% (6/50) | 633ms | 6/50 ✗ |

**충격적인 결과**: Optimistic과 Atomic은 대부분 실패했습니다.

### 왜 Optimistic Lock이 실패했는가?

```
50명 동시 요청 시나리오:

Thread 1: SELECT version=1 → UPDATE WHERE version=1 → 성공!
Thread 2: SELECT version=1 → UPDATE WHERE version=1 → 실패! (이미 version=2)
Thread 3: SELECT version=1 → UPDATE WHERE version=1 → 실패! (이미 version=2)
...
Thread 50: SELECT version=1 → UPDATE WHERE version=1 → 실패!

결과: 50명 중 1명만 성공
(재시도하면 version=2로 다시 시도... 또 경쟁...)
```

**Hot Spot에서는 Optimistic Lock의 "충돌은 드물 것이다"라는 가정이 틀립니다.**

### 왜 Atomic Update에서 데드락이 발생했는가?

투표 트랜잭션은 **두 테이블을 수정**합니다:
1. `INSERT INTO vote` (투표 기록)
2. `UPDATE entry SET vote_count = vote_count + 1` (투표 수 증가)

```
데드락 발생 시나리오:

Transaction A                    Transaction B
────────────────────────────────────────────────────
INSERT vote (row lock on vote)
                                INSERT vote (row lock on vote)
UPDATE entry (entry lock 대기)
                                UPDATE entry (entry lock 대기)
────────────────────────────────────────────────────
→ A는 B의 락 대기, B도 A의 락 대기 → 데드락!
```

**문제의 본질**: 여러 테이블을 수정할 때, **락 획득 순서가 일정하지 않으면** 데드락 발생

### 2차 테스트: 재시도 로직 추가

Atomic 전략에 **지수 백오프 + 지터** 재시도를 추가했습니다.

```java
// 데드락 발생 시 재시도
@Retryable(value = DeadlockLoserDataAccessException.class, maxAttempts = 3)
public void voteAtomic(...) { ... }

// 지수 백오프: 재시도마다 대기 시간 2배
// 지터: 동시 재시도 방지를 위한 무작위 변동
private long calculateBackoff(int retryCount) {
    long baseBackoff = 100 * (long) Math.pow(2, retryCount - 1);  // 100, 200, 400ms
    double jitter = 1.0 + (Math.random() * 2 - 1) * 0.5;  // ±50% 변동
    return (long) (baseBackoff * jitter);
}
```

**왜 지터(Jitter)가 필요한가?**

```
Without Jitter:
  Thread A, B, C: 데드락! → 모두 100ms 후 재시도 → 또 데드락!

With Jitter:
  Thread A: 75ms 후 재시도
  Thread B: 130ms 후 재시도
  Thread C: 95ms 후 재시도
  → 분산된 재시도 → 충돌 확률 감소
```

**2차 테스트 결과**:

| 전략 | 성공률 | p95 응답시간 |
|------|--------|-------------|
| Pessimistic | 100% | 1.73s |
| Atomic + Retry | 94% | 2.98s |

**의외의 결과**: 재시도를 추가했더니 **Pessimistic보다 느려졌습니다**.

### 결과 분석: 재시도의 역설

```
Pessimistic Lock:
  요청 → 락 대기(1.5s) → 처리 → 응답
  → 예측 가능한 대기 시간

Atomic + Retry:
  요청 → 데드락 → 100ms 대기 → 재시도 → 또 데드락 → 200ms 대기 → ...
  → 재시도가 누적되어 응답 시간 급증

Hot Spot에서는:
  "재시도 지옥" vs "순차 처리"
  → 순차 처리가 더 빠름!
```

## 5. 최종 선택: PetStar에 맞는 전략

### 의사결정 과정

세 가지 전략을 테스트한 결과를 **PetStar 서비스 특성**에 비추어 분석했습니다.

**PetStar의 동시성 특성**:
1. 투표 = 랭킹 결정 → **100% 정합성 필수**
2. 인기 Entry에 집중 → **Hot Spot 빈번**
3. 투표 응답 시간 < 3초 → **사용자 경험 중요**

| 기준 | Pessimistic | Optimistic | Atomic + Retry |
|------|-------------|------------|----------------|
| 정합성 | **100%** | 재시도 필요 | 94% (일부 실패) |
| Hot Spot 성능 | 1.73s (순차) | 재시도 폭주 | 2.98s (재시도) |
| 구현 복잡도 | 낮음 | 중간 | 높음 |
| 예측 가능성 | **높음** | 낮음 | 낮음 |

### 선택: Pessimistic Lock

**핵심 판단 근거**:

1. **정합성 최우선**: 투표 1표 = 랭킹 순위 변동
   - 6%라도 실패하면 안 됨 (Atomic 94% 불가)
   - 재시도로 복구해도 데이터 신뢰성 저하

2. **Hot Spot에서의 효율성**: "재시도 지옥" vs "순차 처리"
   ```
   Atomic + Retry: 2.98s (재시도 누적)
   Pessimistic:    1.73s (락 대기 후 순차 처리)
   ```
   **역설적으로 "느린" 비관적 락이 더 빠름**

3. **예측 가능성**: 운영 관점
   - Pessimistic: 락 대기 시간 = 응답 시간 (예측 가능)
   - Retry 기반: 재시도 횟수에 따라 응답 시간 변동

### 언제 다른 전략이 적합한가?

**Optimistic Lock이 적합한 경우**:
- 충돌이 드문 상황 (Hot Spot 없음)
- 읽기 >> 쓰기 비율
- 예: 사용자 프로필 수정

**Atomic Update가 적합한 경우**:
- 단일 테이블 연산
- 단순 증가/감소 (조회수, 좋아요)
- 예: `UPDATE post SET view_count = view_count + 1`

**PetStar 투표의 특수성**:
- 두 테이블 동시 수정 (Vote INSERT + Entry UPDATE)
- Hot Spot 발생 (인기 Entry 집중)
- 100% 정합성 필수 (랭킹 결정)
→ **Pessimistic Lock이 최적**

## 6. 트러블슈팅

### 6-1. @Version 컬럼 NULL 문제

**현상**: Optimistic Lock 테스트 시 NullPointerException

```
java.lang.NullPointerException: Cannot invoke "java.lang.Long.longValue()"
```

**원인**: 기존 Entry 레코드에 `version` 컬럼이 NULL

**해결**:
```sql
UPDATE entry SET version = 0 WHERE version IS NULL;
```

**교훈**: `@Version` 필드 추가 시 기존 데이터 마이그레이션 필수

### 6-2. Self-Invocation으로 @Transactional 미적용

**현상**: `votePessimistic()` 호출 시 트랜잭션 없음 오류

```
Query requires transaction be in progress, but no transaction is known to be in progress
```

**원인**: Spring AOP의 프록시 기반 동작

```java
// vote()에서 직접 호출 → 프록시 우회 → @Transactional 미적용
public VoteResponseDto vote(Long entryId, Integer memberId, String strategy) {
    return switch (strategy) {
        case "pessimistic" -> votePessimistic(entryId, memberId);  // 내부 호출!
    };
}
```

```
┌───────────────────────────────────────────────────────────┐
│  Controller → Proxy → VoteService                         │
│                         │                                 │
│                         ├─ vote()                         │
│                         │    └─ votePessimistic() ← 직접  │
│                         │         @Transactional 미적용!  │
└───────────────────────────────────────────────────────────┘
```

**해결**: Self-Injection

```java
@Lazy
@Autowired
private VoteService self;

public VoteResponseDto vote(...) {
    return switch (strategy) {
        case "pessimistic" -> self.votePessimistic(...);  // 프록시 경유!
    };
}
```

## 7. 성과 및 교훈

### 정량적 성과

- **데이터 정합성**: 100% 보장 (50명 동시 투표 테스트 통과)
- **성공률**: 100% (Pessimistic Lock)
- **응답 시간**: p95 1.73s (Hot Spot 상황)

### 핵심 교훈

1. **synchronized는 분산 환경에서 무의미**
   - 영속 데이터 동시성은 DB 레벨에서 제어

2. **트랜잭션 격리 수준만으로 부족**
   - REPEATABLE READ도 Lost Update 방지 못 함

3. **Hot Spot에서는 비관적 락이 유리**
   - 재시도 지옥보다 순차 처리가 빠름

4. **Spring AOP 프록시 이해 필수**
   - 내부 호출 시 @Transactional 적용 안 됨
