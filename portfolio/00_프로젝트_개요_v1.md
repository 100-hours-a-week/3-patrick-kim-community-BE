# PetStar — 반려동물 사진 콘테스트 플랫폼

## 프로젝트 소개

**PetStar**는 반려동물 사진 콘테스트 플랫폼입니다. 사용자들이 반려동물 사진을 챌린지에 출품하고, 투표로 실시간 랭킹이 결정됩니다.

### 기술 스택

- **Backend**: Spring Boot 3.5, Java 21, JPA, QueryDSL
- **Database**: MySQL 8.0 (AWS RDS)
- **Cache**: Redis 7 (Sorted Set + Lettuce), Caffeine L1
- **Message Queue**: AWS SQS (비동기 투표)
- **Infra**: AWS EC2, Terraform, Docker, ECR
- **Monitoring**: Prometheus, Grafana, p6spy, k6

---

## PetStar가 만들어내는 기술적 도전

PetStar는 단순 CRUD가 아닙니다. **"투표 → 실시간 랭킹 → 우승자 결정"** 구조가 3가지 고유한 기술적 도전을 만들어냅니다.

```
투표 결과가 실시간으로 랭킹에 반영되어야 하고,
그 랭킹이 콘테스트 우승자를 결정합니다.

→ 1표도 누락/중복되면 안 됨
→ 인기 사진에 투표가 집중 (Hot Spot)
→ 읽기와 쓰기가 동시에 발생하는 혼합 부하
```

### 접근 방식: 측정 → 가설 → 실험 → 검증

```
1. 측정: "느리다"가 아니라 "어디서 얼마나 느린가"
2. 가설: "이것이 병목이다, 이렇게 하면 해결된다"
3. 실험: 가설대로 변경하고 동일 조건에서 재측정
4. 검증: 수치로 증명 — 가설이 틀리면 수정하고 재도전
```

## 3개 셀링포인트

### SP1. Hot Spot 동시성 — 정합성과 성능 사이의 트레이드오프

```
[비즈니스 문제]
챌린지 마감 직전, 인기 사진에 수백 명이 동시 투표
→ 1표 = 순위 변동 가능 → 누락/중복 불가

[기술적 여정]
① Race Condition 발견 (Lost Update + 중복 투표)
② 3가지 락 전략 비교 (Pessimistic / Optimistic / Atomic)
   → Hot Spot에서 Pessimistic Lock이 역설적으로 가장 빠름
③ BUT Lock 경합으로 p95 1.73초
④ "사용자가 보는 랭킹"과 "저장되는 데이터" 분리
   → Redis Sorted Set 즉시 반영 + SQS 비동기 DB 저장
   → Strong → Eventual Consistency 트레이드오프 결정

[임팩트]
투표 p95: 1.73s → 50ms (97% 개선)
정합성: 100% 유지 (Idempotency 이중 방어)
```

**면접 포인트**: "정합성을 포기한 게 아니라, 정합성의 위치를 바꿨다"

→ [상세 문서](./01_Hot_Spot_동시성_v1.md)

### SP2. 비동기 파이프라인 정합성 — 장애가 발생해도 데이터를 잃지 않는 설계

```
[비즈니스 문제]
비동기 전환으로 투표 p95 97% 개선
BUT Redis/SQS 장애 시 데이터 유실 경로 4가지 발견

[기술적 여정]
① 비동기 파이프라인에서 4가지 장애 시나리오 식별
   (SQS 전송 실패, Redis 장애, Redis 재시작, Scheduler 한계)
② 보상 트랜잭션: SQS 실패 시 Redis 롤백 (ZINCRBY -1 + SREM)
③ Circuit Breaker: Redis 장애 감지 → Pessimistic Lock 자동 전환
④ 4개 레이어 다층 방어 (SADD → DB UK → Scheduler → Circuit Breaker)

[임팩트]
서비스 중단: Redis 장애에도 0
데이터 유실: SQS 실패 시에도 0 (보상 트랜잭션)
SP1의 Pessimistic Lock이 Fallback 전략으로 재활용
```

**면접 포인트**: "비동기 시스템은 빠르지만, 장애 대응 없이는 불완전하다. 성능 최적화에서 버려진 전략이 장애 대응에서 마지막 방어선이 되었다"

→ [상세 문서](./02_비동기_파이프라인_정합성_v1.md)

### SP3. 혼합 부하 가설 검증 — 가설이 틀릴 때 진짜 엔지니어링이 시작된다

```
[비즈니스 문제]
읽기 전용 300 VUs에서 p95 130ms → 좋은 성능
BUT 실제 서비스(읽기 70% + 투표 30%)에서는 100 VUs에서 무너짐

[기술적 여정]
① Round 1: 혼합 부하 테스트 → HikariCP 커넥션 풀 포화 발견
② 가설: "투표가 DB를 많이 쓰니까 투표가 병목"
③ Round 2: 투표 DB exists → Redis SADD 전환 → 개선 없음 (가설 틀림!)
④ 데이터 분석: 읽기가 DB 호출의 78%를 차지
⑤ 가설 수정: "읽기를 캐싱해야 한다"
⑥ Round 3: Caffeine L1 캐시 (Controller 레벨) → 99.1% 개선

[임팩트]
전체 p95: 5,439ms → 48ms (99.1% 개선)
한계 VU: 100 → 800+ (8배 확장)
RPS: 120.9 → 220.5 (82.4% 증가)
```

**면접 포인트**: "가설이 틀릴 수 있다. 실패한 실험에서 핵심 데이터를 얻고, 가설을 수정해서 재도전했다"

→ [상세 문서](./02_측정기반_최적화_v1.md)

## 전체 성과 요약

| 영역 | Before | After | 개선율 |
|------|--------|-------|--------|
| 읽기 p95 (300 VUs) | 3.01s | **130ms** | **96%↓** |
| 혼합 p95 (800 VUs) | 5,439ms | **48ms** | **99.1%↓** |
| 처리량 (RPS) | 32.9 | **308** (읽기) / **220.5** (혼합) | **9.4배↑** |
| 동시 처리 한계 | 100명 | **800명+** | **8배↑** |
| 투표 정합성 | Race Condition 발생 | **100%** | - |
