# 혼합 부하 한계 탐색 — 가설이 틀릴 때 진짜 엔지니어링이 시작된다

## 1. 문제 발견: 읽기만 테스트하면 안 보이는 것

### Phase 9까지의 성과와 한계

Phase 3~9에서 읽기 API를 최적화하여 **p95 3.01s → 130ms (96% 개선)**, 300 VUs 안정 처리를 달성했습니다.

하지만 실제 서비스에서는 **읽기와 쓰기가 동시에 발생**합니다.

```
Phase 9 테스트: 읽기 전용 (300 VUs)
→ p95 130ms, RPS 308, 에러 0% ✅

실제 서비스: 읽기 70% + 투표 30%
→ ???
```

**의문**: "읽기+쓰기가 동시에 발생하면, 우리 시스템은 몇 명까지 버티는가?"

이 질문에 답하기 위해 혼합 부하 테스트를 설계했습니다.

## 3. Round 1: 병목 발견

### 측정 결과

| 지표 | 값 | 판정 |
|------|-----|------|
| **전체 p95** | **5,439ms** | FAIL (threshold 1,000ms) |
| 챌린지 목록 p95 | 5,426ms | FAIL |
| 랭킹 p95 | 4,708ms | FAIL |
| 엔트리 p95 | **7,002ms** | FAIL (최악) |
| 투표 p95 | 4,149ms | FAIL |
| 에러율 | 0.42% | PASS |
| HTTP RPS | 120.9 | - |

읽기 전용 300 VUs에서 p95 130ms였던 시스템이, 혼합 부하에서는 **100 VUs에서 이미 무너졌습니다**.

### 병목 발견: HikariCP 커넥션 풀 포화

Prometheus 메트릭에서 원인을 찾았습니다:

```
[HikariCP 메트릭]
Active Connections:  30/30 (100% 포화) — 100 VUs에서 이미!
Pending Threads:     170 (최대)
Acquire Time:        12,419ms (최대) — 커넥션 획득에만 12초!
```

**핵심 발견**: 쿼리 실행 시간은 6~19ms로 빠르지만, **커넥션을 획득하기 위한 대기 시간이 응답 시간의 99%를 차지**했습니다.

```
[요청 처리 흐름]
HTTP 요청 → HikariCP 커넥션 요청
                    ↓
         Active = 30/30 (모두 사용 중)
                    ↓
         Pending 큐에서 대기 (최대 12.4초)
                    ↓
         커넥션 획득 → 쿼리 실행 (6~19ms)
                    ↓
         응답 반환

→ 응답 시간 = 커넥션 대기(12초) + 쿼리 실행(수 ms)
→ 대기 시간이 문제의 본질
```

### 왜 읽기 전용에서는 괜찮았는가?

```
읽기 전용 (Phase 9):
  - Redis에서 랭킹 조회 → DB 커넥션 미사용
  - 챌린지/엔트리만 DB 사용
  - 커넥션 경합 적음

혼합 부하 (Round 1):
  - 읽기 70%: 매 반복 3회 DB 호출
  - 투표 30%: 매 반복 2회 DB 호출 (findById + exists)
  - 30개 커넥션으로 수백 VU 감당 불가
```

### 가설 수립

Round 1 데이터를 분석한 결과, 첫 번째 가설을 세웠습니다:

> **가설**: "투표 API에서 DB exists 호출을 Redis SADD로 대체하면(DB 2회 → 1회), 커넥션 사용이 줄어 성능이 개선될 것이다."

## 4. Round 2: 가설이 틀렸다

### 실험: DB exists → Redis SADD 전환

```java
// Before: DB 2회 + Redis 1 RTT
① DB findById          — Entry 존재 확인 (HikariCP)
② DB exists            — 중복 투표 체크 (HikariCP)
③ Redis Pipeline       — ZINCRBY + ZSCORE (1 RTT)
④ SQS Fire & Forget

// After: DB 1회 + Redis 1 RTT
① DB findById          — Entry 존재 확인 (HikariCP)
② Redis Pipeline       — SADD + ZINCRBY + ZSCORE (1 RTT, 중복체크 포함)
③ SQS Fire & Forget
→ DB 호출 2회 → 1회로 감소
```

Redis Pipeline으로 SADD(중복체크) + ZINCRBY(랭킹 증가) + ZSCORE(현재 점수)를 **1 RTT**로 처리합니다. SADD 결과가 0이면 중복 투표로 판단하고 ZINCRBY를 롤백합니다.

### 측정 결과: 개선 없음

| 지표 | Round 1 (Before) | Round 2 (After) | 변화 |
|------|------------------|-----------------|------|
| **전체 p95** | 5,439ms | 5,835ms | **+7% (악화)** |
| 챌린지 목록 p95 | 5,426ms | 5,889ms | +9% |
| 랭킹 p95 | 4,708ms | 4,869ms | +3% |
| 엔트리 p95 | 7,002ms | 7,595ms | +8% |
| 투표 p95 | 4,149ms | 4,257ms | +3% |
| HikariCP Active Max | 30/30 | 30/30 | 동일 (포화) |
| HikariCP Acquire Max | 12,419ms | **15,154ms** | **+22% 악화** |

**실패했습니다.** 투표의 DB 호출을 줄여도 전체 성능에 변화가 없었습니다.

### 실패 원인 분석: 읽기가 진짜 병목

데이터를 다시 분석했습니다:

```
[트래픽 비율별 DB 호출 수]

읽기 유저 70%: 챌린지 + 랭킹 + 엔트리 = 3회/반복
투표 유저 30%: findById + 랭킹 2회   = 3회/반복 (Before)
                                     = 2회/반복 (After, exists 제거)

[VU당 평균 DB 호출 수]
Before: 0.7 × 3 + 0.3 × 3 = 3.0회
After:  0.7 × 3 + 0.3 × 2 = 2.7회

→ DB 호출 10% 감소
→ 하지만 읽기가 전체 DB 호출의 78% (2.1/2.7)를 차지
→ 투표 1회 줄인 것은 전체에서 미미한 영향
```

**가설이 틀렸습니다.**

투표가 병목이라고 생각했지만, 데이터는 **읽기가 커넥션의 78%를 차지한다**고 말하고 있었습니다.

### 새로운 가설

> **수정된 가설**: "읽기 API에 캐시를 적용하면, DB 커넥션 사용을 78% 줄여 커넥션 포화가 해소될 것이다."

## 5. Round 3: 올바른 가설로 재도전

### Caffeine L1 캐시 설계

```java
@Configuration
@EnableCaching
public class CacheConfig {
    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager();
        cacheManager.setCaffeine(Caffeine.newBuilder()
                .expireAfterWrite(10, TimeUnit.SECONDS)
                .maximumSize(200)
                .recordStats());
        return cacheManager;
    }
}
```

### 핵심 설계 결정: Controller 레벨 캐싱

```java
// Controller에 @Cacheable 적용
@GetMapping
@Cacheable(value = "challengeList", key = "#status?.name() ?: 'ALL'")
public ApiResponse<List<ChallengeSummary>> getChallengeList(...)

@GetMapping("/{challengeId}/ranking")
@Cacheable(value = "ranking", key = "#challengeId + ':' + #limit")
public ApiResponse<List<RankingEntry>> getChallengeRanking(...)

@GetMapping("/{challengeId}/entries")
@Cacheable(value = "entryList", key = "#challengeId + ':' + #cursorId + ':' + #limit")
public ApiResponse<EntryResponseDto.ListDto> getEntryList(...)
```

**왜 Service가 아닌 Controller에 캐시를 적용했는가?**

```
Service 레벨 캐싱:
  요청 → Controller → Service(@Transactional 시작, 커넥션 획득) → @Cacheable 체크
  → 캐시 히트여도 이미 커넥션을 점유한 상태!

Controller 레벨 캐싱:
  요청 → Controller(@Cacheable 체크) → 캐시 히트 → 즉시 응답
  → DB 커넥션을 아예 획득하지 않음!
```

커넥션 포화가 병목인 상황에서, **캐시 히트 시 커넥션을 점유하지 않는 것**이 핵심이었습니다.

### TTL 설계 근거

| 캐시 | TTL | 근거 |
|------|-----|------|
| challengeList | 10초 | 운영자가 수동 생성, 변경 빈도 극히 낮음 |
| ranking | 10초 | 투표 결과는 Redis Sorted Set에서 실시간 반영 |
| entryList | 10초 | 참가 등록은 비투표 시간에 주로 발생 |

트레이드오프: 최대 10초간 stale 데이터 가능. 하지만 챌린지/엔트리 목록은 10초 지연이 UX에 영향 없고, 랭킹은 Redis에서 실시간 반영됩니다.

### 측정 결과: 99.1% 개선

| 지표 | Round 1 (Baseline) | Round 2 (투표 최적화) | **Round 3 (캐싱)** | **R1 대비** |
|------|--------------------|-----------------------|---------------------|-------------|
| **전체 p95** | 5,439ms | 5,835ms | **48ms** | **-99.1%** |
| 챌린지 목록 p95 | 5,426ms | 5,889ms | **66ms** | **-98.8%** |
| 랭킹 p95 | 4,708ms | 4,869ms | **38ms** | **-99.2%** |
| 엔트리 p95 | 7,002ms | 7,595ms | **49ms** | **-99.3%** |
| 투표 p95 | 4,149ms | 4,257ms | **77ms** | **-98.1%** |
| HTTP RPS | 120.9 | 119.2 | **220.5** | **+82.4%** |
| 에러율 | 0.42% | 0.73% | **0.43%** | 동일 |
| 투표 성공 | 5,976 | 5,995 | **10,836** | **+81.3%** |

**핵심 수치**:

```
전체 p95:    5,439ms → 48ms  (99.1% 개선, 113배 빨라짐)
처리량(RPS): 120.9   → 220.5 (82.4% 증가)
투표 처리:   5,976   → 10,836 (81.3% 증가)
한계 VU:     100     → 800+  (8배 이상 확장)
```

### 왜 투표 p95도 98% 개선되었는가?

투표 API 자체는 Round 3에서 **변경하지 않았습니다**. 하지만 투표 p95도 4,149ms → 77ms로 개선되었습니다.

```
Before:
  읽기 API가 커넥션 30개 중 78%를 점유
  → 투표가 커넥션 획득을 위해 수 초간 대기
  → 투표 응답 = 커넥션 대기(4초) + 실제 처리(10ms)

After:
  읽기 API가 캐시 히트로 커넥션 미사용
  → 투표가 즉시 커넥션 획득
  → 투표 응답 = 커넥션 대기(0ms) + 실제 처리(77ms)
```

**진짜 병목은 쿼리 속도가 아니라 커넥션 경합이었습니다.**

## 7. 핵심 교훈

### 1. 가설은 틀릴 수 있다 — 데이터가 답을 알고 있다

```
직감:  "투표가 DB를 많이 쓰니까 투표가 병목일 거야"
데이터: "읽기가 DB 호출의 78%를 차지합니다"

→ 직감이 아니라 데이터를 따라가야 한다
→ Round 2 실패가 있었기에 Round 3의 가설이 정확할 수 있었다
```

### 2. 실패한 실험도 가치가 있다

Round 2는 "실패"했지만, 그 실패에서 **읽기가 78%라는 핵심 데이터**를 얻었습니다. 이 데이터가 Round 3의 성공을 이끌었습니다.

```
Round 2에서 얻은 것:
- 투표 DB 1회 줄여도 전체에 미미 → 읽기가 지배적이라는 증거
- Redis SADD가 DB exists보다 중복 차단 정확 (377 → 93) → 정합성 개선
- Lettuce 풀링이 정상 동작 확인 → Redis 병목은 해소된 상태
```

### 3. 캐시 위치가 성능을 결정한다

같은 `@Cacheable`이라도 **어디에 적용하느냐**에 따라 효과가 완전히 다릅니다.

```
Controller 캐싱: 캐시 히트 → DB 커넥션 획득 안 함 → 커넥션 포화 해소
Service 캐싱:    캐시 히트 → 이미 커넥션 점유 중 → 커넥션 포화 지속

→ 병목이 "커넥션 풀"인 상황에서는 캐시 위치가 결정적
```

### 4. 읽기:쓰기 비율이 최적화 방향을 결정한다

```
읽기 70% × DB 3회 = 2.1회/VU  (전체의 78%)
투표 30% × DB 2회 = 0.6회/VU  (전체의 22%)

→ 78%를 차지하는 읽기를 최적화해야 전체 성능이 개선
→ 22%인 투표를 아무리 최적화해도 효과 미미
```
