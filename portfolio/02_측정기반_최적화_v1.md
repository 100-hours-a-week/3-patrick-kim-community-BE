### 병목 발견: HikariCP 커넥션 풀 포화

Prometheus 메트릭에서 원인을 찾았습니다:

```
HikariCP 메트릭 (Prometheus):
  Active Connections:  30/30 (100% 포화) — 100 VUs에서 이미!
  Pending Threads:     170 (최대)
  Acquire Time:        12,419ms (최대) — 커넥션 획득에만 12초!
```

쿼리 실행 시간은 6~19ms로 빠릅니다. **커넥션 대기 시간이 응답 시간의 99%를 차지**했습니다.

```
요청 처리 흐름:
HTTP 요청 → HikariCP 커넥션 요청
                    ↓
         Active = 30/30 (모두 사용 중)
                    ↓
         Pending 큐에서 대기 (최대 12.4초)
                    ↓
         커넥션 획득 → 쿼리 실행 (6~19ms)
                    ↓
         응답 반환

→ 응답 시간 ≈ 커넥션 대기 시간
```

### 가설 수립

직감적으로 가설을 세웠습니다:

> **가설**: "투표 API에서 DB exists 호출을 Redis SADD로 대체하면 (DB 2회 → 1회), 커넥션 사용이 줄어 성능이 개선될 것이다."

## 4. Round 2: 가설이 틀렸다

### 실험: 투표 DB exists → Redis SADD 전환

```
Before: DB 2회 + Redis 1 RTT
① DB findById       — Entry 존재 확인
② DB exists          — 중복 투표 체크
③ Redis Pipeline     — ZINCRBY + ZSCORE

After: DB 1회 + Redis 1 RTT
① DB findById       — Entry 존재 확인
② Redis Pipeline     — SADD + ZINCRBY + ZSCORE (중복체크 포함)
→ DB 호출 2회 → 1회
```

### 측정 결과: 개선 없음

| 지표 | Round 1 | Round 2 | 변화 |
|------|---------|---------|------|
| **전체 p95** | 5,439ms | 5,835ms | **+7% 악화** |
| HikariCP Active Max | 30/30 | 30/30 | 동일 (포화) |
| HikariCP Acquire Max | 12,419ms | **15,154ms** | **+22% 악화** |

**실패했습니다.** 투표의 DB 호출을 줄여도 전체 성능에 변화가 없었습니다.

### 실패 원인 분석: 데이터가 답을 알고 있었다

Round 2 실패 후, 데이터를 다시 분석했습니다:

```
[트래픽 비율별 DB 호출 수]

읽기 유저 70%: 챌린지 + 랭킹 + 엔트리 = 3회/반복
투표 유저 30%: findById + 랭킹 2회   = 2회/반복 (After)

[VU당 평균 DB 호출 수]
After:  0.7 × 3 + 0.3 × 2 = 2.7회

읽기: 2.1 / 2.7 = 78%
투표: 0.6 / 2.7 = 22%

→ 읽기가 전체 DB 호출의 78%를 차지
→ 투표 1회 줄인 것은 전체에서 미미한 영향
```

**가설이 틀렸습니다.** 투표가 병목이라고 생각했지만, 데이터는 **읽기가 커넥션의 78%를 차지한다**고 말하고 있었습니다.

### 실패에서 얻은 것

Round 2는 "실패"했지만, 가치 있는 데이터를 얻었습니다:

```
1. 읽기가 78%라는 핵심 데이터 → Round 3 가설의 근거
2. Redis SADD가 DB exists보다 중복 차단 정확 (377 → 93) → 정합성 개선
3. Lettuce 풀링 정상 동작 확인 → Redis 병목 해소 확인
```

### 가설 수정

> **수정된 가설**: "읽기 API에 캐시를 적용하면, DB 호출의 78%를 제거하여 커넥션 포화가 해소될 것이다."

## 5. Round 3: 올바른 가설로 재도전

### Caffeine L1 캐시 설계

```java
@Configuration
@EnableCaching
public class CacheConfig {
    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager();
        cacheManager.setCaffeine(Caffeine.newBuilder()
                .expireAfterWrite(10, TimeUnit.SECONDS)
                .maximumSize(200)
                .recordStats());
        return cacheManager;
    }
}
```

### 핵심 설계 결정: Controller 레벨 캐싱

```java
@GetMapping
@Cacheable(value = "challengeList", key = "#status?.name() ?: 'ALL'")
public ApiResponse<List<ChallengeSummary>> getChallengeList(...)

@GetMapping("/{challengeId}/ranking")
@Cacheable(value = "ranking", key = "#challengeId + ':' + #limit")
public ApiResponse<List<RankingEntry>> getChallengeRanking(...)

@GetMapping("/{challengeId}/entries")
@Cacheable(value = "entryList", key = "#challengeId + ':' + #cursorId + ':' + #limit")
public ApiResponse<EntryResponseDto.ListDto> getEntryList(...)
```

**왜 Service가 아닌 Controller에 캐시를 적용했는가?**

```
Service 레벨 캐싱:
  요청 → Controller → Service(@Transactional → 커넥션 획득) → @Cacheable 체크
  → 캐시 히트여도 이미 커넥션을 점유한 상태!

Controller 레벨 캐싱:
  요청 → Controller(@Cacheable 체크) → 캐시 히트 → 즉시 응답
  → DB 커넥션을 아예 획득하지 않음!
```

병목이 "커넥션 풀"인 상황에서, **캐시 히트 시 커넥션을 점유하지 않는 것**이 핵심입니다.

### 캐싱 패턴: Cache Aside

Caffeine L1 캐시는 **Cache Aside (Lazy Loading)** 패턴을 사용합니다.

```
Cache Aside 동작:
1. 요청 → 캐시 확인
2. 캐시 히트 → 즉시 응답 (DB 접근 없음)
3. 캐시 미스 → DB 조회 → 결과를 캐시에 저장 → 응답

@Cacheable이 이 패턴을 자동 구현:
  @Cacheable(value = "ranking", key = "#challengeId + ':' + #limit")
  → 키가 존재하면 메서드 실행 안 함 (캐시 반환)
  → 키가 없으면 메서드 실행 → 결과 캐시 저장
```

**왜 Cache Aside인가?**

```
Write-Through: 쓰기 시 캐시 + DB 동시 업데이트
  → 읽기 API에는 "쓰기"가 없음 → 해당 없음

Write-Behind: 쓰기 시 캐시만 업데이트, DB 비동기
  → 읽기 전용 캐시에는 해당 없음

Cache Aside: 읽기 시 캐시 미스 → DB 조회 → 캐시 저장
  → 읽기 API에 가장 적합한 패턴
  → 캐시 미스 시에만 DB 접근 → 커넥션 풀 보호
```

### 캐싱으로 발생할 수 있는 문제와 대응

#### Cache Stampede (캐시 쇄도)

```
문제:
  TTL 10초 만료 → 동시 100개 요청이 모두 캐시 미스
  → 100개 요청이 동일 쿼리를 DB에 실행 (Thundering Herd)

대응:
  1. Caffeine의 키별 동기화
     → 같은 키에 대한 동시 미스 시 1개 스레드만 DB 조회, 나머지 대기
     → @Cacheable의 sync 옵션과 유사한 내부 동작

  2. 짧은 TTL (10초) + 경량 쿼리
     → Stampede 주기가 짧지만 쿼리가 빠름 (Fetch Join + Index 완료)
     → 10초에 1회 DB 쿼리 vs 초당 수백 회 → 99%+ 캐시 히트율

  3. 근본 보호: Controller 레벨
     → Stampede가 발생해도 캐시 미스 1건만 커넥션 사용
     → 나머지 요청은 Caffeine 내부 대기 → 커넥션 풀 보호
```

#### Cache Invalidation (캐시 무효화)

```
문제:
  투표 발생 → Redis Sorted Set 즉시 반영
  BUT Caffeine 랭킹 캐시에는 이전 데이터가 최대 10초간 잔존

설계 결정 — TTL 기반 만료만 사용, @CacheEvict 미사용:

  @CacheEvict를 쓰면:
    투표마다 랭킹 캐시 무효화 → 잦은 캐시 미스
    → 투표 빈도 ↑ → 캐시 히트율 ↓ → 커넥션 포화 재발
    → "캐시를 적용한 의미가 사라지는" 역설

  TTL 10초만 사용하면:
    투표 빈도와 무관하게 캐시 히트율 유지 (99%+)
    → 최대 10초 stale 데이터
    → 실시간 랭킹은 Redis Sorted Set에서 직접 제공 가능
    → 비즈니스적으로 "목록 조회"의 10초 지연은 무해

  트레이드오프:
    신선도 ←→ 히트율
    PetStar: 히트율 우선 (커넥션 풀이 진짜 병목)
```

#### Cache Penetration (캐시 관통)

```
문제:
  존재하지 않는 challengeId로 조회 → 캐시 미스 → DB 조회 → null
  → 결과가 null이면 캐시 저장 안 됨 → 매번 DB 조회

PetStar 상황:
  → challengeId는 API 명세에 정의된 유효한 값만 사용
  → 인증된 사용자만 접근 → 랜덤 ID 공격 가능성 낮음
  → challengeList 캐시가 유효한 ID 목록을 제공
  → 현재 규모에서는 리스크 낮음
```

### TTL 설계 근거

| 캐시 | TTL | 근거 |
|------|-----|------|
| challengeList | 10초 | 운영자가 수동 생성, 변경 빈도 극히 낮음 |
| ranking | 10초 | 투표 결과는 Redis Sorted Set에서 실시간 반영 |
| entryList | 10초 | 참가 등록은 비투표 시간에 주로 발생 |

트레이드오프: 최대 10초간 stale 데이터 가능. 하지만 랭킹 점수는 Redis에서 실시간 반영되고, 챌린지/엔트리 목록의 10초 지연은 UX에 무해합니다.

### 측정 결과: 99.1% 개선

| 지표 | Round 1 | Round 2 (실패) | **Round 3** | **R1 대비** |
|------|---------|----------------|-------------|-------------|
| **전체 p95** | 5,439ms | 5,835ms | **48ms** | **-99.1%** |
| 챌린지 p95 | 5,426ms | 5,889ms | **66ms** | -98.8% |
| 랭킹 p95 | 4,708ms | 4,869ms | **38ms** | -99.2% |
| 엔트리 p95 | 7,002ms | 7,595ms | **49ms** | -99.3% |
| 투표 p95 | 4,149ms | 4,257ms | **77ms** | -98.1% |
| HTTP RPS | 120.9 | 119.2 | **220.5** | **+82.4%** |
| 한계 VU | 100 | 100 | **800+** | **8배 확장** |

### 왜 투표 p95도 98% 개선되었는가?

투표 API는 Round 3에서 **변경하지 않았습니다**. 하지만 투표 p95도 4,149ms → 77ms로 개선되었습니다.

```
Before:
  읽기 API가 커넥션 30개 중 78%를 점유
  → 투표가 커넥션 획득을 위해 수 초 대기
  → 투표 응답 = 커넥션 대기(4초) + 실제 처리(10ms)

After:
  읽기 API가 캐시 히트로 커넥션 미사용
  → 투표가 즉시 커넥션 획득
  → 투표 응답 = 커넥션 대기(0ms) + 실제 처리(77ms)
```

**진짜 병목은 쿼리 속도가 아니라 커넥션 경합이었습니다.**

## 7. 핵심 교훈

### 1. 가설은 틀릴 수 있다 — 데이터가 답을 알고 있다

```
직감:  "투표가 DB를 많이 쓰니까 투표가 병목일 거야"
데이터: "읽기가 DB 호출의 78%를 차지합니다"

→ Round 2 실패가 있었기에 Round 3의 가설이 정확할 수 있었다
```

### 2. 실패한 실험도 가치가 있다

Round 2는 "실패"했지만, **읽기가 78%라는 핵심 데이터**를 얻었습니다. 이 데이터가 Round 3의 성공을 이끌었습니다.

### 3. 캐시 위치가 성능을 결정한다

```
Controller 캐싱: 캐시 히트 → DB 커넥션 획득 안 함 → 커넥션 포화 해소
Service 캐싱:    캐시 히트 → 이미 커넥션 점유 중 → 커넥션 포화 지속

→ 병목이 "커넥션 풀"인 상황에서는 캐시 위치가 결정적
```

### 4. 읽기:쓰기 비율이 최적화 방향을 결정한다

```
읽기 70% × DB 3회 = 2.1회/VU  (전체의 78%)
투표 30% × DB 2회 = 0.6회/VU  (전체의 22%)

→ 78%를 차지하는 읽기를 최적화해야 전체 성능 개선
→ 22%인 투표를 아무리 최적화해도 효과 미미 (암달의 법칙)
```
