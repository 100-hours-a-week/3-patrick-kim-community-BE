# Phase 10: MQ 선택 분석

> **목표**: PetStar 투표 시스템에 최적화된 메시지 큐 선택
>
> **문제 정의**: Pessimistic Lock으로 100% 정합성은 확보했지만, 50명 동시 투표 시 p95 = 1.73초
> 챌린지 마감 직전 Hot Spot에서 사용자 경험 저하 → 비동기 처리로 응답 시간 단축 필요

---

## 1. 요구사항 분석

### 1.1 PetStar 투표 시스템 특성

```
트래픽 패턴:
  평상시: 분당 100~500 투표
  마감 직전: 분당 10,000+ 투표 (20~100배 스파이크)

데이터 특성:
  - 투표 메시지 크기: ~100 bytes (memberId, entryId, timestamp)
  - 순서 보장: 필요 없음 (각 투표는 독립적)
  - 중복 처리: 필요 (동일 투표 2번 처리되면 안 됨)
  - 유실 허용: 불가 (투표는 반드시 DB에 반영되어야 함)

처리 지연 허용:
  - 랭킹 표시: Redis 즉시 반영 (사용자에게 보이는 값)
  - DB 반영: 5초 이내 (Eventual Consistency 허용)
```

### 1.2 비기능 요구사항

| 요구사항 | 목표 | 우선순위 |
|:---------|:-----|:---------|
| **응답 시간** | 1.73s → 50ms | 🔴 필수 |
| **처리량** | 10,000 msg/min 이상 | 🔴 필수 |
| **메시지 유실** | 0% | 🔴 필수 |
| **순서 보장** | 불필요 | 🟢 낮음 |
| **운영 복잡도** | 낮음 선호 (소규모 팀) | 🟡 중요 |
| **비용** | AWS 프리티어 활용 | 🟡 중요 |
| **확장성** | 수평 확장 가능 | 🟢 선호 |

## 2. 후보 MQ 비교

### 2.1 Kafka

```
아키텍처:
  Producer → Broker (Partition) → Consumer Group

특징:
  - 분산 로그 기반 메시지 스트리밍
  - 높은 처리량 (초당 수백만 메시지)
  - 메시지 영속성 (디스크 저장, 재처리 가능)
  - Partition 기반 순서 보장
```

**장점:**
- 초당 수백만 메시지 처리 가능
- 메시지 재처리 가능 (offset 기반)
- 다양한 Consumer Group 지원
- 스케일아웃 용이

**단점:**
- 높은 운영 복잡도 (Zookeeper/KRaft, Broker 관리)
- 초기 학습 곡선 높음
- 최소 3대 브로커 권장 (HA 구성)
- AWS MSK 비용 높음 (~$200/월 최소)

**PetStar 적합도:**
```
처리량: ✅ 오버스펙 (분당 10만도 여유)
복잡도: ❌ 소규모 팀에 부담
비용:   ❌ MSK 최소 $200/월
결론:   ❌ 오버엔지니어링
```

### 2.2 RabbitMQ

```
아키텍처:
  Producer → Exchange → Queue → Consumer

특징:
  - AMQP 프로토콜 기반
  - Exchange 라우팅 (Direct, Topic, Fanout)
  - 메시지 ACK 기반 신뢰성
  - 클러스터링 지원
```

**장점:**
- 성숙한 생태계, 풍부한 문서
- 유연한 라우팅 (Exchange 패턴)
- Docker로 쉬운 로컬 개발
- 적절한 처리량 (초당 수만 메시지)

**단점:**
- 클러스터 운영 시 복잡도 증가
- AWS에서 직접 관리 필요 (EC2 + Docker)
- 메모리 기반 (대량 백로그 시 성능 저하)

**PetStar 적합도:**
```
처리량: ✅ 충분 (초당 1만 이상)
복잡도: 🟡 Docker 단일 노드는 간단
비용:   ✅ EC2에 같이 배포 가능
결론:   🟡 적합하나 관리 필요
```

### 2.3 Amazon SQS

```
아키텍처:
  Producer → SQS Queue → Consumer (Polling)

특징:
  - 완전 관리형 서비스 (Serverless)
  - Standard Queue: At-least-once, 순서 보장 X
  - FIFO Queue: Exactly-once, 순서 보장
  - 자동 스케일링
```

**장점:**
- 제로 운영 (AWS 완전 관리)
- 무제한 확장
- 사용량 기반 과금 (요청당 $0.0000004)
- Dead Letter Queue 내장
- 프리티어: 월 100만 요청 무료

**단점:**
- Polling 기반 (Push 없음)
- Standard Queue는 순서 보장 X
- AWS 종속성

**PetStar 적합도:**
```
처리량: ✅ 무제한 (AWS 관리)
복잡도: ✅ 제로 (완전 관리형)
비용:   ✅ 프리티어 + 저렴 ($1/월 수준)
결론:   ✅ 최적
```

## 4. 결정: Amazon SQS Standard Queue

### 4.1 선택 근거

```
1. 운영 부담 최소화
   - 1인 개발자/소규모 팀에 적합
   - 인프라 관리 시간을 개발에 투자
   - 장애 대응도 AWS 책임

2. 비용 효율성
   - 프리티어: 월 100만 요청 무료
   - 초과 시에도 $0.40/백만 요청
   - 분당 1만 투표 = 월 4.3억 요청 = ~$172/월
   - 실제 예상: $1-5/월 수준

3. PetStar 요구사항 충족
   - 순서 보장 불필요 → Standard Queue 적합
   - 중복 처리 → Consumer에서 Idempotency 처리
   - 유실 없음 → DLQ + 재시도로 보장

4. 확장성
   - 트래픽 스파이크 시 자동 확장
   - Consumer만 늘리면 처리량 증가
   - 챌린지 마감 직전도 문제없음
```

### 4.2 FIFO vs Standard Queue

```
FIFO Queue 고려 이유:
  - Exactly-once 처리 보장
  - 순서 보장

Standard Queue 선택 이유:
  - 투표는 순서 무관 (각각 독립적)
  - Exactly-once는 Consumer에서 구현 가능 (memberId + entryId로 중복 체크)
  - Standard가 처리량 높음 (FIFO는 3000 msg/초 제한)
  - 비용 저렴 (FIFO는 2배)

결론: Standard Queue + Consumer Idempotency
```

### 4.3 아키텍처 설계

```
투표 요청 흐름:

  Client
    │
    ▼
  API Server
    │
    ├──▶ Redis ZINCRBY (즉시 랭킹 반영)
    │
    ├──▶ SQS SendMessage (비동기 처리 요청)
    │
    └──▶ Response (50ms 이내)


비동기 처리 흐름:

  SQS Queue
    │
    ▼
  Consumer (별도 스레드 또는 Lambda)
    │
    ├──▶ 중복 체크 (DB에 이미 있는가?)
    │      └── 있으면 → 무시 (Idempotency)
    │      └── 없으면 → DB INSERT
    │
    └──▶ ACK (메시지 삭제)


실패 처리:

  SQS Queue ──실패──▶ DLQ (Dead Letter Queue)
                        │
                        ▼
                    모니터링/알림
                        │
                        ▼
                    수동 조치 또는 재처리
```

### 4.4 메시지 스키마

```json
{
  "voteId": "uuid",           // Idempotency key
  "memberId": 12345,
  "entryId": 67890,
  "challengeId": 11,
  "timestamp": "2026-02-20T10:30:00Z"
}
```

### 4.5 Consumer 구현 전략

```java
// 1. SQS Listener (Spring Cloud AWS)
@SqsListener("petstar-votes")
public void handleVote(VoteMessage message) {
    // 중복 체크 (Idempotency)
    if (voteRepository.existsByMemberIdAndEntryId(
            message.getMemberId(), message.getEntryId())) {
        return; // 이미 처리됨, 무시
    }

    // DB 저장 (Pessimistic Lock 필요 없음 - 단일 Consumer)
    Vote vote = new Vote(message.getMemberId(), message.getEntryId());
    voteRepository.save(vote);

    // Entry vote_count 증가 (Atomic Update)
    entryRepository.incrementVoteCount(message.getEntryId());
}

// 2. Idempotency 보장
// - SQS Standard는 at-least-once (중복 가능)
// - DB unique constraint: (member_id, entry_id)
// - 중복 INSERT 시 무시 (IGNORE 또는 ON CONFLICT)
```

```yaml

  localstack:
    image: localstack/localstack
    ports:
      - "4566:4566"
    environment:
      - SERVICES=sqs
      - DEFAULT_REGION=ap-northeast-2
```

### 5.2 AWS 인프라 (Terraform)

```hcl
# SQS Queue
resource "aws_sqs_queue" "votes" {
  name = "petstar-votes"

  visibility_timeout_seconds = 30
  message_retention_seconds  = 86400  # 1 day

  redrive_policy = jsonencode({
    deadLetterTargetArn = aws_sqs_queue.votes_dlq.arn
    maxReceiveCount     = 3
  })
}

# Dead Letter Queue
resource "aws_sqs_queue" "votes_dlq" {
  name = "petstar-votes-dlq"
}
```

### 5.3 Spring 의존성

```gradle
// build.gradle
implementation 'io.awspring.cloud:spring-cloud-aws-starter-sqs:3.0.0'
```
