# PetStar 동시성 제어 분석

## 개요
- **목표**: 투표 기능의 Race Condition 해결 및 데이터 정합성 보장
- **테스트 환경**: k6, 50 VUs 동시 투표
- **서버 스펙**: EC2 t3.small, RDS db.t3.micro (MySQL 8.0)

---

## 문제 분석

### 발견된 Race Condition

#### 1. Check-then-Act Race Condition (중복 투표)
```java
// Before: Race Condition 있음
if (voteRepository.existsByEntryIdAndMemberId(entryId, memberId)) {
    throw new DuplicateVoteException();
}
// ← 여기서 다른 스레드가 끼어들 수 있음!
voteRepository.save(vote);
```

```
Thread A: existsBy... → false
Thread B: existsBy... → false
Thread A: save(vote)  ← 저장 성공
Thread B: save(vote)  ← 중복 저장!
```

#### 2. Lost Update (투표 수 누락)
```java
entry.increaseVoteCount();  // voteCount++ (비원자적)
```

```
Thread A: read voteCount → 10
Thread B: read voteCount → 10
Thread A: write voteCount = 11
Thread B: write voteCount = 11  ← 10+2=12가 아닌 11!
```

## 해결 방법 비교

### 고려한 접근 방식

| 방법 | 장점 | 단점 |
|------|------|------|
| **synchronized** | 간단 | 단일 JVM만 유효, 스케일 아웃 불가 |
| **Pessimistic Lock** | 확실한 동시성 보장 | 락 대기 시간, 성능 저하 |
| **Optimistic Lock** | 락 없이 동작 | 충돌 시 재시도 필요 |
| **Atomic Update** | 빠름, DB 레벨 보장 | 복합 연산 시 데드락 가능 |
| **Redis 분산 락** | 분산 환경 지원 | 인프라 복잡성 증가 |

### 왜 애플리케이션 레벨(synchronized)은 안 되는가?

```
┌─────────────┐     ┌─────────────┐
│  Server A   │     │  Server B   │
│ synchronized│     │ synchronized│
│  (독립적)   │     │  (독립적)   │
└──────┬──────┘     └──────┬──────┘
       │                   │
       └───────┬───────────┘
               ▼
        ┌─────────────┐
        │   MySQL     │  ← 여기서 제어해야 함
        └─────────────┘
```

- `synchronized`는 **단일 JVM 내에서만** 동작
- 스케일 아웃 시 각 서버의 synchronized가 서로를 모름
- **결론**: 웹 애플리케이션의 영속 데이터 동시성은 DB 레벨에서 제어

### 트랜잭션 격리 수준과의 관계

MySQL 기본 격리 수준: **REPEATABLE READ**

```
Tx A: SELECT voteCount → 10
Tx B: SELECT voteCount → 10
Tx A: UPDATE voteCount = 11, COMMIT
Tx B: UPDATE voteCount = 11, COMMIT  ← Lost Update!
```

- REPEATABLE READ는 "읽은 값이 변하지 않음"을 보장
- **쓰기 충돌(Lost Update)은 방지하지 않음**
- 별도의 락 또는 원자적 쿼리 필요

## 구현한 세 가지 전략

### 전략 1: Pessimistic Lock

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT e FROM Entry e WHERE e.id = :entryId")
Optional<Entry> findByIdWithPessimisticLock(@Param("entryId") Long entryId);
```

**동작**: `SELECT ... FOR UPDATE` → 행 락 획득 후 처리

### 전략 2: Optimistic Lock

```java
@Entity
public class Entry {
    @Version
    private Long version;
}
```

**동작**: 업데이트 시 버전 체크, 충돌 시 예외 → 재시도

### 전략 3: Atomic Update + UK 예외 처리

```java
// 중복 투표: UK 제약조건으로 DB에서 방지
try {
    voteRepository.saveAndFlush(vote);
} catch (DataIntegrityViolationException e) {
    throw new DuplicateVoteException();
}

// Lost Update: 원자적 쿼리로 방지
@Query("UPDATE Entry e SET e.voteCount = e.voteCount + 1 WHERE e.id = :entryId")
int incrementVoteCount(@Param("entryId") Long entryId);
```

- 50명이 동시에 같은 Entry에 투표

### 결과 비교 (1차: 재시도 없음)

| 전략 | 성공률 | p95 응답시간 | 특징 |
|------|--------|-------------|------|
| **Pessimistic** | 100% (50/50) | 1.73s | 느리지만 확실 |
| **Optimistic** | 16% (8/50) | 885ms | 버전 충돌 다발 |
| **Atomic** | 12% (6/50) | 633ms | 데드락 발생 |

### 결과 비교 (2차: 데드락 재시도 추가)

| 전략 | 성공률 | p95 응답시간 | 특징 |
|------|--------|-------------|------|
| **Pessimistic** | 100% (50/50) | 1.73s | 락 대기 |
| **Optimistic** | 16% (8/50) | 885ms | 충돌 다발 |
| **Atomic (no retry)** | 12% (6/50) | 633ms | 데드락 |
| **Atomic + Retry** | **94% (47/50)** | 2.98s | ✅ 재시도로 복구 |


실패한 트랜잭션은 롤백되어 **데이터 정합성 유지됨**

## 데드락 분석

### 왜 Atomic 전략에서 데드락이 발생했는가?

```
Transaction A: INSERT vote(entry_id=99991) → vote 테이블 락
Transaction B: INSERT vote(entry_id=99991) → vote 테이블 락
Transaction A: UPDATE entry SET vote_count+1 → entry 테이블 락 대기
Transaction B: UPDATE entry SET vote_count+1 → entry 테이블 락 대기
```

**두 테이블(Vote, Entry)을 동시에 조작**하면서:
- 각 트랜잭션이 서로 다른 순서로 락 요청
- 순환 대기 → 데드락

### MySQL 데드락 감지

```
Deadlock found when trying to get lock; try restarting transaction
```

MySQL InnoDB는 데드락 감지 후 하나의 트랜잭션을 **희생양(victim)**으로 선택해 롤백

---

## 해결 방안: 데드락 재시도 로직

### 구현 방식 (TransactionTemplate 사용)

```java
public VoteResponseDto.VoteResult voteAtomicWithRetry(Long entryId, Integer memberId) {
    int retryCount = 0;

    while (retryCount < MAX_RETRY_COUNT) {
        try {
            // TransactionTemplate으로 새 트랜잭션에서 실행
            return transactionTemplate.execute(status -> {
                return doVoteAtomic(entryId, memberId);
            });
        } catch (DeadlockLoserDataAccessException | CannotAcquireLockException e) {
            retryCount++;
            log.warn("[Atomic+Retry] Deadlock detected, retry {}/{}", retryCount, MAX_RETRY_COUNT);

            if (retryCount >= MAX_RETRY_COUNT) {
                throw new GeneralException(ErrorStatus._INTERNAL_SERVER_ERROR);
            }

            // 지수 백오프 + 지터
            sleep(calculateBackoff(retryCount));
        }
    }
}
```

### 재시도 전략
- **최대 재시도**: 5회
- **백오프**: 지수 백오프 + 지터 (동시 재시도 방지)
  - retry 1: 100ms ± 50ms
  - retry 2: 200ms ± 100ms
  - retry 3: 400ms ± 200ms
  - retry 4: 800ms ± 400ms
  - retry 5: 1600ms ± 800ms
- **재시도 대상**: `DeadlockLoserDataAccessException`, `CannotAcquireLockException`

### 지터(Jitter)가 필요한 이유

```
Without Jitter:
  Thread A: retry after 100ms
  Thread B: retry after 100ms
  Thread C: retry after 100ms
  → 동시에 재시도 → 다시 충돌!

With Jitter:
  Thread A: retry after 75ms
  Thread B: retry after 130ms
  Thread C: retry after 95ms
  → 분산된 재시도 → 충돌 감소
```

## 결론 및 선택

### PetStar 투표의 특성
1. **Hot Spot 가능**: 인기 Entry에 투표 집중
2. **실시간 정합성 필요**: 랭킹 결정에 사용
3. **신뢰성 중요**: 대회/투표의 특성상 정확해야 함

### 최종 선택: Pessimistic Lock

| 기준 | Pessimistic | Optimistic | Atomic + Retry |
|------|-------------|------------|----------------|
| 성공률 | ✅ **100%** | ❌ 16% | ⚠️ 94% |
| p95 응답 | ✅ **1.73s** | 885ms | ❌ 2.98s |
| 복잡도 | ✅ 낮음 | 중간 | 중간 |
| 예측 가능성 | ✅ 순차 처리 | ❌ | ❌ 재시도 횟수 불확실 |

### 선택 이유

1. **Pessimistic Lock 선택**:
   - **100% 성공률** - Hot Spot에서도 확실
   - **더 빠름** - 재시도 백오프(2.98s)보다 락 대기(1.73s)가 빠름
   - **단순함** - 재시도 로직 불필요
   - **예측 가능** - 순차적으로 확실하게 처리

2. **Optimistic Lock 제외**:
   - 16% 성공률로 Hot Spot에서 사용 불가
   - 충돌이 많으면 재시도해도 계속 충돌

3. **Atomic + Retry 제외**:
   - 94% 성공률로 Pessimistic보다 낮음
   - 재시도 백오프로 인해 오히려 더 느림
   - 복잡한 재시도 로직 필요

### 핵심 교훈

**"재시도가 항상 답은 아니다"**

- Atomic + Retry: 데드락 → 대기 → 재시도 → 또 데드락 → 더 긴 대기...
- Pessimistic Lock: 락 대기 → 순차 처리 → 확실하게 완료

Hot Spot 상황에서는 **재시도보다 순차 처리가 더 효율적**

### 향후 개선 방향

더 높은 동시성이 필요하다면:
1. **Redis 분산 락** (Redisson) - DB 데드락 자체를 방지
2. **메시지 큐** (Kafka/RabbitMQ) - 투표 요청을 큐에 넣고 순차 처리
3. **CQRS** - 쓰기(투표)와 읽기(집계)를 분리

## 트러블슈팅

### 1. @Version 컬럼 NULL 문제

**현상**: Optimistic Lock 테스트 시 `NullPointerException` 발생

```
java.lang.NullPointerException: Cannot invoke "java.lang.Long.longValue()"
```

**원인**: 기존 Entry 레코드에 `version` 컬럼이 NULL

**해결**:
```sql
UPDATE entry SET version = 0 WHERE version IS NULL;
```

**교훈**: `@Version` 필드 추가 시 기존 데이터 마이그레이션 필수

### 2. Self-Invocation으로 @Transactional 미적용

**현상**: `votePessimistic()` 호출 시 트랜잭션 없음 오류

```
Query requires transaction be in progress, but no transaction is known to be in progress
```

**원인**: Spring AOP의 프록시 기반 동작 방식

```java
// vote() 메서드에서 직접 호출 → 프록시 우회 → @Transactional 미적용
public VoteResponseDto.VoteResult vote(Long entryId, Integer memberId, String strategy) {
    return switch (strategy) {
        case "pessimistic" -> votePessimistic(entryId, memberId);  // ❌ 직접 호출
        // ...
    };
}

@Transactional  // 이 어노테이션이 적용되지 않음!
public VoteResponseDto.VoteResult votePessimistic(Long entryId, Integer memberId) {
    // SELECT FOR UPDATE 실행 → 트랜잭션 없어서 실패
}
```

**Spring 프록시 동작 원리**:
```
┌─────────────────────────────────────────────────────────────┐
│  Controller                                                  │
│     │                                                        │
│     ▼                                                        │
│  ┌──────────────────────┐                                   │
│  │   Proxy (CGLIB)      │  ← 외부 호출: 프록시 경유         │
│  │   - @Transactional   │    → 트랜잭션 시작 ✅             │
│  │   - AOP 적용         │                                   │
│  └──────────┬───────────┘                                   │
│             │                                                │
│             ▼                                                │
│  ┌──────────────────────┐                                   │
│  │   VoteService        │                                   │
│  │   vote() ──────────► votePessimistic()                   │
│  │          내부 호출: 프록시 우회 → 트랜잭션 없음 ❌       │
│  └──────────────────────┘                                   │
└─────────────────────────────────────────────────────────────┘
```

**해결: Self-Injection**:
```java
@Service
@RequiredArgsConstructor
public class VoteService {

    // Self-injection으로 프록시 참조
    @Lazy
    @Autowired
    private VoteService self;

    public VoteResponseDto.VoteResult vote(Long entryId, Integer memberId, String strategy) {
        return switch (strategy) {
            // self를 통해 호출 → 프록시 경유 → @Transactional 적용 ✅
            case "pessimistic" -> self.votePessimistic(entryId, memberId);
            case "optimistic" -> self.voteOptimistic(entryId, memberId);
            // ...
        };
    }
}
```

**대안들**:
1. **별도 클래스 분리**: `VoteExecutor` 등으로 실제 로직 분리
2. **ApplicationContext 사용**: `context.getBean(VoteService.class)`
3. **AspectJ 모드**: 프록시 대신 바이트코드 위빙 (복잡)

**교훈**:
- Spring AOP는 프록시 기반이므로 내부 호출 시 적용되지 않음
- `@Transactional`, `@Cacheable`, `@Async` 등 모든 AOP 어노테이션에 해당
- Self-injection 또는 클래스 분리로 해결

### 3. 데드락 발생 (Atomic 전략)

**현상**: 50명 동시 투표 시 약 88% 실패

```
Deadlock found when trying to get lock; try restarting transaction
```

**원인**: 두 테이블 동시 조작으로 순환 대기 발생

```
Tx A: INSERT vote → vote 테이블 락 획득
Tx B: INSERT vote → vote 테이블 락 대기
Tx A: UPDATE entry → entry 테이블 락 대기 (B가 먼저 entry 락 시도)
Tx B: UPDATE entry → entry 테이블 락 대기
→ 순환 대기 → 데드락!
```

**해결: 재시도 + 지수 백오프 + 지터**:
```java
public VoteResponseDto.VoteResult voteAtomicWithRetry(Long entryId, Integer memberId) {
    int retryCount = 0;

    while (retryCount < MAX_RETRY_COUNT) {
        try {
            return transactionTemplate.execute(status -> doVoteAtomic(entryId, memberId));
        } catch (DeadlockLoserDataAccessException | CannotAcquireLockException e) {
            retryCount++;
            sleep(calculateBackoff(retryCount));  // 지수 백오프 + 지터
        }
    }
    throw new GeneralException(ErrorStatus._INTERNAL_SERVER_ERROR);
}

// 지수 백오프 + 지터: 동시 재시도 방지
private long calculateBackoff(int retryCount) {
    long baseBackoff = 100 * (long) Math.pow(2, retryCount - 1);
    double jitter = 1.0 + (Math.random() * 2 - 1) * 0.5;  // ±50%
    return (long) (baseBackoff * jitter);
}
```

**지터(Jitter)가 필요한 이유**:
```
Without Jitter:
  Thread A, B, C: 모두 100ms 후 재시도 → 다시 충돌!

With Jitter:
  Thread A: 75ms 후 재시도
  Thread B: 130ms 후 재시도
  Thread C: 95ms 후 재시도
  → 분산된 재시도 → 충돌 감소
```

**교훈**:
- 여러 테이블을 동시 수정하면 데드락 가능성 증가
- 재시도 시 지터 없으면 동시 재시도로 반복 충돌
- 하지만 Hot Spot에서는 재시도보다 Pessimistic Lock이 더 효율적
