# Phase 10: 비동기 쓰기 최적화 (SQS)

> **핵심 성과**: 동시 투표 응답시간 50% 개선 (1.47s → 740ms p95)
> **기술 스택**: AWS SQS, Redis, Spring Cloud AWS 3.x

---

## 1. 문제 발견: 왜 이 작업을 시작했는가?

### 1.1 Phase 6에서 발견한 한계

Phase 6에서 Pessimistic Lock으로 동시성 문제를 100% 해결했지만, **성능 병목**이 발생했습니다.

```
50명 동시 투표 테스트 결과 (Pessimistic Lock):

| 지표 | 값 |
|------|------|
| 성공률 | 100% |
| p95 응답시간 | 1.73초 |
| 평균 응답시간 | 865ms |

→ 정합성은 확보했지만, 응답 시간이 너무 느림
```

### 1.2 왜 느린가? - Lock 경합 분석

Pessimistic Lock의 작동 방식을 시각화:

```
SELECT FOR UPDATE 동작:

시간  Thread A     Thread B     Thread C     ...Thread 50
──────────────────────────────────────────────────────────
t1    Lock 획득
t2    DB 작업 중    대기         대기         대기
t3    Lock 해제
t4               Lock 획득
t5               DB 작업 중    대기         대기
t6               Lock 해제
...
──────────────────────────────────────────────────────────

50명이 순차적으로 처리 → 마지막 사람은 49명을 기다림!
```

### 1.3 Hot Spot 문제의 본질

PetStar에서 이 문제가 심각한 이유:

```
챌린지 마감 직전 시나리오:

"챌린지 종료까지 10분!"
       ↓
인기 Entry에 투표 집중
       ↓
같은 Row에 수백 명 동시 접근
       ↓
Lock 대기 → 응답 지연 → 사용자 이탈
```

**결론**: Lock을 유지하면서 빠른 응답을 얻는 것은 **구조적으로 불가능**합니다.

## 2. 해결 전략: 비동기 분리

### 2.1 핵심 통찰

```
질문: 꼭 투표 API에서 DB 저장까지 해야 하나?

관찰:
1. 사용자가 보는 랭킹 → Redis에서 조회
2. DB는 영속 저장소 역할
3. 랭킹만 즉시 반영되면 사용자 경험 OK

→ DB 저장을 비동기로 분리하면?
```

### 2.2 아키텍처 변경

```
Before: 동기 처리 (Pessimistic Lock)
─────────────────────────────────────
Client → API → DB(Lock 대기) → Response
                  ↑
             병목 지점

응답시간: 1.73초 (p95)


After: 비동기 처리 (Redis + SQS)
─────────────────────────────────────
Client → API → Redis(즉시) → Response
                  ↓
                SQS → Consumer → DB

응답시간: 740ms (p95) - 50% 개선
```

### 2.3 Eventual Consistency 수용

```
트레이드오프 분석:

포기한 것:
- Strong Consistency (즉시 일관성)
- DB 반영까지 지연 가능

얻은 것:
- 빠른 응답 (Lock 대기 없음)
- 트래픽 스파이크 흡수
- 처리량 향상

허용 가능한 이유:
- 사용자가 보는 랭킹은 Redis (즉시 반영)
- DB는 백그라운드 저장
- 챌린지 결과 정산은 마감 후 (충분한 여유)
```

## 3. MQ 선택: 왜 SQS인가?

### 3.1 후보 비교

| 기준 | Kafka | RabbitMQ | SQS |
|:-----|:------|:---------|:----|
| **처리량** | 수백만/초 | 수만/초 | 무제한 |
| **운영 복잡도** | 높음 (Zookeeper) | 중간 | **없음** |
| **AWS 비용** | $200+/월 (MSK) | EC2 필요 | **$1/월** |
| **순서 보장** | 파티션 내 보장 | 선택적 | 미보장 |

### 3.2 선택 근거

**1. 운영 부담 제로**
```
Kafka: 브로커 3대, Zookeeper, 모니터링, 장애 대응
SQS: AWS가 전부 관리, 인프라 걱정 없음
```

**2. 비용 효율성**
```
Kafka (MSK): 최소 $200/월
SQS: 월 100만 요청 무료, 초과 시 $0.40/백만

PetStar 예상 (분당 1000 투표):
→ 월 4,320만 요청 → ~$17 (무료 포함 시 $0)
```

**3. PetStar 요구사항 적합성**
```
✅ 순서 보장 필요? → 불필요 (투표는 독립적)
✅ 처리량? → 분당 1만도 여유
✅ 메시지 유실? → DLQ로 100% 방지
✅ 중복 처리? → Consumer에서 Idempotency
```

**결론**: Kafka는 오버엔지니어링. 100 bytes 투표 메시지에 분산 로그 시스템 불필요.

## 4. 트러블슈팅 기록

### 4.1 문제 1: IAM 권한 설정

**증상**
```
Failed to start bean 'sqsListenerEndpointContainer'
Unable to load credentials from any provider
```

**원인 분석**
```
1. EC2 Instance Profile 미연결
2. Docker 컨테이너에서 EC2 메타데이터 접근 불가 (IMDSv2 hop limit)
3. 기존 S3 IAM User에 SQS 권한 없음
```

**해결 과정**
```bash
# 1. EC2 Instance Profile 연결
aws ec2 associate-iam-instance-profile \
  --instance-id i-052e365914c75fda5 \
  --iam-instance-profile Name=petstar-app-profile

# 2. IMDSv2 hop limit 증가 (Docker용)
aws ec2 modify-instance-metadata-options \
  --instance-id i-052e365914c75fda5 \
  --http-put-response-hop-limit 2

# 3. 결국 S3 User에 SQS 권한 추가 (application.yml의 기존 자격 증명 활용)
aws iam put-user-policy --user-name S3 --policy-name SQS-Access \
  --policy-document '{"Version":"2012-10-17","Statement":[...]}'
```

**배운 점**
- Docker 환경에서 EC2 메타데이터 접근 시 hop limit 주의
- 기존 인프라와의 통합 고려 필요

**증상**
```bash
curl -X POST http://localhost:8080/auth -d '{"email":"...","password":"..."}'
```

**원인 분석**

```java
// WebConfig.java

// AuthController.java
@RequestMapping("/auth")  // 실제 경로: /api/auth
```

**해결**
```java

.requestMatchers("/auth", "/auth/**").permitAll()
```

**교훈**
- WebMvcConfigurer의 PathPrefix와 Security 설정 불일치 주의
- 실제 URL 매핑 확인 후 Security 설정

### 4.6 문제 6: SQS Consumer 데드락 ✅ 해결

**증상**
```
Deadlock found when trying to get lock; try restarting transaction
```

**원인 분석**
```
SQS Consumer 병렬 처리 시:

Thread 1: UPDATE entry SET vote_count = vote_count + 1 WHERE entry_id = 100
Thread 2: UPDATE entry SET vote_count = vote_count + 1 WHERE entry_id = 100
Thread 3: UPDATE entry SET vote_count = vote_count + 1 WHERE entry_id = 100
...

→ 같은 Row에 동시 UPDATE → Deadlock
```

**해결 방법**

1. **SqsConfig.java - Consumer 순차 처리 설정**
```java
@Bean
public SqsMessageListenerContainerFactory<Object> defaultSqsListenerContainerFactory(
        SqsAsyncClient sqsAsyncClient) {
    return SqsMessageListenerContainerFactory.builder()
            .sqsAsyncClient(sqsAsyncClient)
            .configure(options -> options
                    .maxConcurrentMessages(1)      // 동시 처리 1개 (데드락 방지)
                    .maxMessagesPerPoll(1)         // 폴링당 1개
            )
            .build();
}
```

2. **VoteConsumer.java - 데드락 재시도 로직**
```java
} catch (CannotAcquireLockException e) {
    // 데드락 발생 시 예외를 던져서 SQS 재시도 유도
    log.warn("[VoteConsumer] Deadlock detected, will retry: entryId={}, memberId={}",
            message.getEntryId(), message.getMemberId());
    throw e;  // SQS visibility timeout 후 재시도
}
```

**결과**
```
| 저장소 | 수정 전 | 수정 후 |
|:-------|:--------|:--------|
| Redis | 50표 ✅ | 50표 ✅ |
| DB | 30표 (40% 손실) | 50표 (100%) ✅ |
```

**핵심 통찰**
- Consumer 처리량 < API 처리량이지만, 사용자 경험에 영향 없음
- Redis에서 즉시 랭킹 반영되므로 DB 지연은 허용 가능
- 순차 처리로 데드락 완전 제거

| **성공률** | 100% | 100% | 유지 |

```
Async (SQS) 50명 동시 투표 - 데드락 수정 후:
```

## 6. 핵심 교훈

### 6.1 동기 vs 비동기 선택 기준

```
동기 처리가 적합한 경우:
- 즉시 일관성 필수 (결제, 재고 차감)
- 트랜잭션 범위가 명확
- 부하가 예측 가능

비동기 처리가 적합한 경우:
- 지연 허용 가능 (알림, 로깅, 분석)
- 트래픽 스파이크 흡수 필요
- 실패해도 재시도로 복구 가능
```

### 6.2 Eventual Consistency 설계 원칙

```
1. "사용자가 보는 값"과 "저장되는 값" 분리
   - 랭킹: Redis (즉시)
   - 영속: DB (비동기)

2. Idempotency 필수
   - 같은 요청 2번 와도 결과 동일
   - UK + 애플리케이션 로직 이중 방어

3. 실패 처리 전략
   - DLQ로 실패 메시지 보관
   - 모니터링 + 알람
   - 수동 재처리 절차
```

### 6.3 기술 선택의 원칙

```
"문제에 맞는 도구를 선택하라"

Kafka: 대용량 스트리밍, 이벤트 소싱, 로그 집계
RabbitMQ: 복잡한 라우팅, 우선순위 큐
SQS: 단순 메시지 전달, 운영 부담 최소화

PetStar 투표:
- 단순 메시지 (100 bytes)
- 순서 무관
- 실패 시 재시도

→ SQS가 정답 (Kafka는 오버엔지니어링)
```
