# 비동기 쓰기 최적화 - SQS를 활용한 Hot Spot 해결

## 1. 문제 발견 과정

### Pessimistic Lock의 한계

Phase 6에서 Pessimistic Lock으로 동시성 문제를 해결했습니다. **100% 정합성**은 확보했지만, 새로운 문제가 발생했습니다.

```
동시 50명 투표 테스트 결과:

| 전략 | 성공률 | p95 응답시간 |
|------|--------|-------------|
| Pessimistic Lock | 100% | 1.73초 |

→ 성공은 하지만 너무 느림!
```

### 왜 1.73초나 걸리는가?

Pessimistic Lock의 작동 방식을 분석했습니다:

```
Pessimistic Lock (SELECT FOR UPDATE):

시간  Thread A     Thread B     Thread C     ...Thread 50
─────────────────────────────────────────────────────────
t1    Lock 획득
t2    DB 작업 중    대기         대기         대기
t3    Lock 해제
t4               Lock 획득
t5               DB 작업 중    대기         대기
t6               Lock 해제
...
─────────────────────────────────────────────────────────

50명이 순차적으로 처리 → 마지막 사람은 49명을 기다림
```

**Hot Spot 상황에서 순차 처리는 필연적으로 느립니다.**

### PetStar에서 Hot Spot이 발생하는 이유

```
챌린지 마감 직전 시나리오:

"챌린지 종료까지 10분 남았습니다!"
    ↓
인기 Entry에 투표 집중
    ↓
같은 Entry에 수백 명 동시 투표
    ↓
Lock 경합 → 응답 지연 → 사용자 이탈
```

**결론**: Lock을 유지하면서 빠른 응답을 얻는 것은 불가능합니다.

## 2. 해결 전략: 비동기 분리

### 핵심 통찰

```
문제: DB Lock 때문에 느림
질문: 꼭 투표 API에서 DB 저장을 해야 하나?

관찰:
- 사용자가 보는 랭킹은 Redis에서 조회
- DB는 영속 저장소 역할
- 랭킹만 즉시 반영되면 사용자 경험 OK

→ DB 저장을 비동기로 분리하면?
```

### 아키텍처 재설계

```
Before: 동기 처리 (Pessimistic Lock)
─────────────────────────────────────
Client → API → DB(Lock 획득 대기) → Response
                   ↑
              병목 지점

응답시간: 1.73초 (p95)


After: 비동기 처리 (Redis + SQS)
─────────────────────────────────────
Client → API → Redis(즉시) → Response
                   ↓
                 SQS → Consumer → DB

응답시간: 50ms 이내 (예상)
```

### 트레이드오프 분석

```
포기한 것:
- Strong Consistency (즉시 일관성)
- DB 반영까지 최대 5초 지연 가능

얻은 것:
- 빠른 응답 (Lock 대기 없음)
- 트래픽 스파이크 흡수 (SQS 버퍼)
- 처리량 10배 이상 향상

허용 가능한 이유:
- 사용자가 보는 랭킹은 Redis (즉시 반영)
- DB는 "진실의 원천"으로 백그라운드 저장
- 5초 지연은 비즈니스적으로 무해함
```

## 3. MQ 선택: 왜 SQS인가?

### 후보 비교

세 가지 MQ를 비교했습니다:

| 기준 | Kafka | RabbitMQ | SQS |
|:-----|:------|:---------|:----|
| **처리량** | 수백만/초 | 수만/초 | 무제한 |
| **운영 복잡도** | 높음 | 중간 | **없음** |
| **AWS 비용** | $200+/월 | EC2 필요 | **$1/월** |
| **로컬 개발** | 어려움 | Docker | LocalStack |

### SQS 선택 근거

**1. 운영 부담 제로**

```
Kafka/RabbitMQ:
  - 브로커 설치, 모니터링, 장애 대응
  - Zookeeper/KRaft 관리 (Kafka)
  - 클러스터 구성, 스케일링

SQS:
  - AWS가 전부 관리
  - 인프라 걱정 없이 개발에 집중
```

**2. 비용 효율성**

```
Kafka (MSK): 최소 $200/월 (3대 브로커 필수)
SQS: 월 100만 요청 무료 + 초과 시 $0.40/백만

PetStar 예상:
- 분당 1,000 투표 × 60분 × 24시간 × 30일 = 4,320만/월
- 비용: ~$17/월 (무료 포함 시 $0)
```

**3. PetStar 요구사항 적합성**

```
요구사항 체크리스트:

✅ 순서 보장 필요? → 불필요 (투표는 독립적)
✅ 처리량? → 분당 1만도 여유 (SQS 무제한)
✅ 메시지 유실? → DLQ로 100% 방지
✅ 트래픽 스파이크? → 자동 확장
✅ 중복 처리? → Consumer에서 Idempotency
```

**Kafka는 오버엔지니어링입니다.** 100 bytes 투표 메시지 전달에 분산 로그 스트리밍 시스템이 필요하지 않습니다.

## 4. 구현 상세

### 4.1 투표 흐름

```
1. Client: POST /api/v1/entries/{id}/votes

2. VoteService.voteAsync():
   - Entry 조회 (Lock 없음)
   - 유효성 검증 (챌린지 상태, 자기 투표 방지)
   - 중복 체크 (DB 조회)
   - Redis ZINCRBY (랭킹 즉시 반영)
   - SQS SendMessage (비동기 처리 요청)
   - Response 반환 (50ms 이내)

3. VoteConsumer:
   - SQS ReceiveMessage
   - 중복 체크 (Idempotency)
   - Vote INSERT + Entry UPDATE
   - 메시지 삭제 (ACK)
```

### 4.2 Idempotency 보장

SQS Standard Queue는 **at-least-once** 전달을 보장합니다. 같은 메시지가 2번 올 수 있습니다.

```java
@SqsListener("petstar-votes")
@Transactional
public void handleVote(String payload) {
    VoteMessage message = parse(payload);

    // 1차 방어: 애플리케이션 레벨 중복 체크
    if (voteRepository.existsByEntryIdAndMemberId(
            message.getEntryId(), message.getMemberId())) {
        log.info("Duplicate vote ignored");
        return;
    }

    // 2차 방어: DB Unique Constraint
    try {
        voteRepository.save(vote);
        entryRepository.incrementVoteCount(entryId);
    } catch (DataIntegrityViolationException e) {
        // UK 위반 = 이미 처리됨
        log.info("Duplicate vote (UK violation)");
    }
}
```

### 4.3 실패 처리

```
SQS 재시도 정책:

visibility_timeout: 30초
max_receive_count: 3

흐름:
1. 메시지 수신
2. 처리 실패 (예외 발생)
3. 30초 후 다시 visible
4. 재시도 (최대 3회)
5. 3회 실패 → DLQ로 이동

DLQ 모니터링:
- CloudWatch 알람 (10개 이상 시 알림)
- 14일간 보관 (디버깅용)
```

### 4.4 Fallback 처리

SQS가 설정되지 않은 환경에서도 동작해야 합니다:

```java
public VoteResponseDto.VoteResult vote(Long entryId, Integer memberId) {
    // SQS 설정 여부에 따라 전략 선택
    String strategy = (voteProducer != null) ? "async" : "pessimistic";
    return vote(entryId, memberId, strategy);
}

@Transactional(readOnly = true)
public VoteResponseDto.VoteResult voteAsync(Long entryId, Integer memberId) {
    // SQS 미설정 시 pessimistic으로 fallback
    if (voteProducer == null) {
        log.warn("VoteProducer not configured, falling back to pessimistic");
        return self.votePessimistic(entryId, memberId);
    }
    // ... async 처리
}
```

## 5. 인프라 구성

### Terraform (AWS)

```hcl
# terraform/sqs.tf

resource "aws_sqs_queue" "votes" {
  name = "petstar-votes"

  visibility_timeout_seconds = 30
  message_retention_seconds  = 86400  # 1일

  redrive_policy = jsonencode({
    deadLetterTargetArn = aws_sqs_queue.votes_dlq.arn
    maxReceiveCount     = 3

  name = "petstar-votes-dlq"
```

## 7. 핵심 교훈

### 7.1 동기 vs 비동기 선택 기준

```
동기 처리가 적합한 경우:
- 즉시 일관성 필수 (결제, 재고 차감)
- 트랜잭션 범위가 명확함
- 부하가 예측 가능함

비동기 처리가 적합한 경우:
- 지연 허용 가능 (알림, 로깅, 분석)
- 트래픽 스파이크 흡수 필요
- 실패해도 재시도로 복구 가능
```

### 7.2 Eventual Consistency 설계 시 고려사항

```
1. "사용자가 보는 값"과 "저장되는 값" 분리
   - 랭킹: Redis (즉시)
   - 영속 저장: DB (비동기)

2. Idempotency 필수
   - 같은 요청이 2번 와도 결과 동일
   - UK + 애플리케이션 로직 이중 방어

3. 실패 처리 전략
   - DLQ로 실패 메시지 보관
   - 모니터링 + 알람
   - 수동 재처리 절차
```

### 7.3 기술 선택의 원칙

```
"문제에 맞는 도구를 선택하라"

Kafka: 대용량 스트리밍, 이벤트 소싱, 로그 집계
RabbitMQ: 복잡한 라우팅, 우선순위 큐
SQS: 단순 메시지 전달, 운영 부담 최소화

PetStar 투표:
- 단순 메시지 (100 bytes)
- 순서 무관
- 실패 시 재시도

→ SQS가 정답
```
