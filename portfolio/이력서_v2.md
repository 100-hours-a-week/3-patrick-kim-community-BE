# PetStar — 반려동물 사진 콘테스트 플랫폼

**기간**: 2026.01 ~ 2026.03 (3개월)
**역할**: 백엔드 개발 (1인)
**기술 스택**: Spring Boot 3.5 / Java 21 / MySQL 8.0 / Redis 7 / AWS SQS / JPA / QueryDSL / Terraform / Docker

---

## 프로젝트 개요

반려동물 사진 콘테스트 플랫폼. 사용자가 반려동물 사진을 챌린지에 출품하고, 투표로 실시간 랭킹이 결정되어 우승자를 가리는 서비스.

---

## 핵심 구현 기능

### 인증/인가
- JWT 기반 Stateless 인증 (Access Token 15분 + Refresh Token 7일)
- Spring Security Filter Chain + Custom JWT Filter
- `@LoginUser` 커스텀 어노테이션으로 인증 사용자 주입

### 커뮤니티
- 게시글 CRUD + 커서 기반 페이지네이션 (QueryDSL)
- 댓글 CRUD + Fetch Join (N+1 방지)
- 게시글 좋아요 (Unique Key 중복 방지)

### 반려동물 관리
- 반려동물 프로필 CRUD (Soft Delete 적용)
- AWS S3 이미지 업로드/삭제 (Image 엔티티 상태 관리: USED/UNUSED/DELETED)

### 챌린지 시스템
- 챌린지 목록 조회 (상태별 필터: UPCOMING/ACTIVE/ENDED)
- 챌린지 엔트리 등록/삭제 + 커서 기반 페이지네이션
- 응원 메시지 CRUD

### 투표 시스템
- 5가지 동시성 제어 전략 비교 구현 (Pessimistic / Optimistic / Atomic / Atomic+Retry / Async)
- Redis Sorted Set 실시간 랭킹 (ZINCRBY O(log N) + ZREVRANGE)
- AWS SQS 비동기 투표 파이프라인 (Fire & Forget + Consumer)
- 투표 취소 기능

### 인프라 & 모니터링
- Terraform으로 AWS 인프라 전체 IaC (VPC, EC2, RDS, SQS, ECR, IAM)
- Docker 컨테이너 빌드 + ECR 푸시 + EC2 배포 파이프라인
- Prometheus + Grafana 메트릭 모니터링 (HikariCP, JVM, HTTP, Redis)
- p6spy 쿼리 로깅
- k6 부하 테스트 (혼합 부하 시나리오, Hot Spot 시뮬레이션)

## 성능 최적화

### 읽기 API 최적화 (p95: 4.76s → 130ms, **97% 개선**)

- **커넥션 풀 튜닝**: Grafana HikariCP 메트릭 분석 → Pool 10→30, 획득 시간 1.79s → 0.19s (89%↓)
- **Pageable 적용**: Java Stream.limit() → DB LIMIT 전환, 100K row → 10 row 로드 (81%↓)
- **N+1 해결**: p6spy로 21개 쿼리 발견 → QueryDSL Fetch Join → 1개 쿼리 (73%↓)
- **복합 인덱스**: EXPLAIN filesort 발견 → `(challenge_id, vote_count DESC)` 인덱스 추가 (38%↓)
- **최종 결과**: RPS 32.9 → 308 (9.4배↑), 동시 처리 100명 → 300명 (3배↑)

### 투표 Hot Spot 최적화 (p95: 1.73s → 50ms, **97% 개선**)

- **동시성 제어**: 3가지 Lock 전략 비교 테스트 → Hot Spot에서 Pessimistic Lock이 유일한 100% 정합성 보장 확인
- **비동기 분리**: 투표 응답 경로에서 DB Lock 제거, Redis Sorted Set 즉시 반영 + SQS 비동기 DB 저장
- **Redis 캐싱 전략**: Write-Through(투표 즉시 반영) + Cache Aside(첫 조회 시 Warming) 하이브리드
- **Redis Pipeline**: SADD(중복체크) + ZINCRBY(랭킹) + ZSCORE(점수) = 1 RTT로 통합
- **Lettuce Connection Pooling**: commons-pool2 기반 max-active 20, 고동시성 병목 해소
- **SQS 큐 설계**: at-least-once 전달 + DLQ(3회 실패 → 14일 보관) + Consumer 순차 처리(데드락 방지) + Backpressure 흡수
- **Idempotency 이중 방어**: Redis SADD (API 레벨) + DB Unique Key (Consumer 레벨)

### 혼합 부하 최적화 (p95: 5,439ms → 48ms, **99.1% 개선**)

- **문제 발견**: 읽기 70% + 투표 30% 혼합 부하 → 100 VUs에서 HikariCP 포화 (30/30)
- **가설 검증 실패**: 투표 DB→Redis 전환 → 개선 없음 → 데이터 분석으로 읽기가 78% 차지 확인
- **Cache Aside 패턴**: Caffeine L1 캐시 (TTL 10s, max 200), Controller 레벨 @Cacheable
- **캐시 위치 설계**: Controller(@Cacheable) vs Service(@Transactional) — 캐시 히트 시 DB 커넥션 미획득이 핵심
- **Cache Stampede 대응**: Caffeine 키별 동기화 + 짧은 TTL + 경량 쿼리 (Fetch Join + Index)
- **Cache Invalidation**: TTL-only 전략 (@CacheEvict 미사용 — 히트율 > 신선도 트레이드오프)
- **최종 결과**: 한계 VU 100→800+ (8배↑), RPS 120.9→220.5 (82.4%↑)

## 장애 대응 설계

### 보상 트랜잭션

- SQS 전송 실패 시 Redis 롤백 (ZINCRBY -1 + SREM), 데이터 유실 0

### Circuit Breaker (Resilience4j)

- Redis 장애 감지 → Pessimistic Lock 자동 Fallback (서비스 중단 0)
- 설정: failureRate 50%, slidingWindow 10, waitDuration 30s, slowCall 2s/80%
- Actuator 엔드포인트 (`/actuator/circuitbreakers`)로 상태 실시간 모니터링

### 정합성 보장 다층 방어

- L1: Redis SADD (API 중복 차단)
- L2: DB Unique Key (Consumer 최종 방어)
- L3: VoteConsistencyScheduler (5분 주기 Redis↔DB 정합성 검증)
- L4: Circuit Breaker (Redis 장애 시 동기 Fallback)

## 기술적 의사결정

| 의사결정 | 선택 | 근거 |
|---------|------|------|
| 동시성 제어 | Pessimistic Lock | Hot Spot에서 유일한 100% 정합성 (Optimistic 16%, Atomic 12% 실패) |
| 실시간 랭킹 | Redis Sorted Set | O(log N) 모든 연산, Skip List 기반 100만 건에서도 ~20 탐색 |
| 랭킹 캐싱 전략 | Write-Through + Cache Aside | 투표 시 즉시 반영(Write-Through) + 첫 조회 시 DB에서 Warming(Cache Aside) |
| 읽기 캐싱 전략 | Cache Aside (Caffeine) | 읽기 전용 API에 최적, 캐시 미스 시에만 DB 접근 |
| 캐시 무효화 | TTL 10s only (@CacheEvict 미사용) | 투표 빈도와 무관한 히트율 유지, 커넥션 풀이 진짜 병목 |
| 메시지 큐 | AWS SQS Standard (at-least-once) | 운영 부담 0, $1/월, FIFO 300 TPS 제한 회피, Idempotency로 중복 처리 |
| Consumer 동시성 | maxConcurrentMessages=1 | 같은 Entry 동시 UPDATE 데드락 방지, 응답은 Redis에서 이미 완료 |
| 장애 Fallback | Circuit Breaker → Pessimistic Lock | SP1에서 구현한 동기 경로를 Fallback 자산으로 재활용 |
| 인프라 관리 | Terraform (IaC) | VPC/EC2/RDS/SQS/ECR 전체 코드 관리, 재현 가능한 인프라 |
