# SP2. 비동기 파이프라인 정합성 — 장애가 발생해도 데이터를 잃지 않는 설계

> **핵심 질문**: "비동기 시스템에서 장애가 발생하면, 데이터 정합성을 어떻게 보장하는가?"

---

## 1. 비동기 전환의 명암

SP1에서 동기 Pessimistic Lock(p95 1.73s)을 Redis + SQS 비동기(p95 50ms)로 전환했습니다. 성능은 **97% 개선**되었지만, 새로운 문제가 생겼습니다.

```
동기 시스템:
  Client → API → DB(Lock + Write) → Response
  → 실패하면 트랜잭션 롤백 → 데이터 항상 일관

비동기 시스템:
  Client → API → Redis(반영) → SQS(전송) → Response
                                    ↓
                               Consumer → DB(저장)
  → Redis 반영 후 SQS 실패하면? DB 반영 안 되는데 Redis에는 남아있음
  → Redis 자체가 죽으면? 투표 API 전체 장애
```

**"비동기는 빠르지만, 장애 시 데이터가 사라질 수 있다."**

---

## 2. 장애 시나리오 식별: 4가지 데이터 유실 경로

비동기 파이프라인의 각 구간에서 장애가 발생할 수 있는 시나리오를 체계적으로 식별했습니다.

### 장애 1: SQS 전송 실패 → 투표 유실

```
Redis: SADD ✓ ZINCRBY ✓ (투표 기록 + 랭킹 반영 완료)
SQS:   전송 실패 ✗
DB:    메시지가 안 옴 → Vote 영원히 미저장

→ 사용자는 투표 성공으로 보이지만 DB에 없음
→ ConsistencyScheduler가 5분 후 DB 기준으로 Redis 덮어씀
→ 투표가 사라짐!
```

**가장 위험한 시나리오.** 사용자 경험과 데이터가 불일치하고, 시간이 지나면 Scheduler가 Redis를 DB 기준으로 "교정"하면서 투표가 유실됩니다.

### 장애 2: Redis 전체 장애 → 투표 API 전면 중단

```
현재 코드:
  voteAsync() → rankingRedisService.recordAndIncrementPipelined(...)
  → RedisConnectionFailureException → 500 에러

→ Redis 장애 = 투표 불가능
→ SP1에서 만든 Pessimistic Lock Fallback이 있지만, 자동 전환 로직 없음
```

### 장애 3: Redis 재시작 → 중복 투표 방어 붕괴

```
Redis 재시작 시:
  ranking:challenge:{id} → Cache Warming으로 복구 ✓
  votes:{entryId} SET    → 복구 안 됨 ✗

→ SADD 중복 체크 데이터 소실
→ 이미 투표한 사용자가 다시 투표 가능
→ Consumer에서 DB UK로 최종 방어되지만, 그 전까지 Redis 랭킹 왜곡
```

### 장애 4: ConsistencyScheduler의 구조적 한계

```
Redis 100표, DB 99표
→ Scheduler: "SQS 지연이겠지" → 30초 대기
→ BUT 실제로는 SQS 전송 실패 (장애 1)
→ 30초 후에도 불일치 → DB 기준 덮어씀 → 투표 유실

→ "SQS 처리 지연"과 "SQS 전송 실패"를 구분하지 못함
```

## 3. 보상 트랜잭션: SQS 전송 실패 시 Redis 롤백

### 문제 분석

장애 1의 핵심은 **Redis 반영과 SQS 전송 사이의 원자성이 없다**는 점입니다.

```
기존 코드 (Fire & Forget):
  Redis Pipeline (SADD + ZINCRBY + ZSCORE)  → 성공
  voteProducer.sendVote(message)             → 실패 시 아무것도 안 함
  → Redis에 반영된 투표가 DB에 영원히 도달하지 않음
```

### 해결: 보상 트랜잭션 패턴

분산 시스템에서는 2PC(Two-Phase Commit)가 비현실적이므로, **보상 트랜잭션(Compensating Transaction)** 패턴을 적용합니다. SQS 전송이 실패하면 Redis에 반영한 것을 역으로 취소합니다.

```java
// 3. SQS 메시지 전송 + 실패 시 보상 트랜잭션
VoteMessage message = VoteMessage.create(memberId, entryId, challengeId);
try {
    voteProducer.sendVote(message);
} catch (Exception e) {
    // SQS 전송 실패 → Redis 보상 (롤백)
    rankingRedisService.decrementVote(challengeId, entryId);   // ZINCRBY -1
    rankingRedisService.removeVoteRecord(entryId, memberId);   // SREM
    log.error("[VoteAsync] SQS send failed, Redis compensated: entryId={}, memberId={}",
            entryId, memberId, e);
    throw new GeneralException(ErrorStatus._INTERNAL_SERVER_ERROR);
}
```

### 보상 순서가 중요한 이유

```
보상 순서: decrementVote → removeVoteRecord

왜 이 순서인가?
1. decrementVote (ZINCRBY -1): 랭킹 점수 복구 → 사용자가 보는 순위 정상화
2. removeVoteRecord (SREM): 투표 기록 삭제 → 재투표 가능하도록 복구

만약 순서가 반대라면:
  SREM 성공 → (여기서 장애) → ZINCRBY -1 실행 안 됨
  → 투표 기록은 지웠지만 랭킹 점수는 남아있음
  → 유령 투표: 투표자는 없는데 점수만 올라감

→ 랭킹 정상화가 더 중요하므로 먼저 실행
→ 최악의 경우(SREM 실패)에도 재투표만 막히지, 랭킹은 정확
```

### 장애 4도 함께 해소

보상 트랜잭션으로 장애 1을 해결하면, ConsistencyScheduler의 구조적 한계(장애 4)도 대부분 해소됩니다.

```
Before: SQS 전송 실패 → Redis ≠ DB → Scheduler가 DB 기준 덮어씀 → 유실
After:  SQS 전송 실패 → Redis 즉시 롤백 → Redis = DB → Scheduler 개입 불필요
```

## 4. Circuit Breaker: Redis 장애 시 자동 Fallback

### 문제 분석

장애 2의 핵심은 **Redis 의존성이 단일 장애 지점(Single Point of Failure)**이라는 것입니다.

```
Redis 정상: Client → voteAsync() → Redis + SQS → 50ms ✓
Redis 장애: Client → voteAsync() → RedisConnectionFailureException → 500 ✗

→ Redis가 죽으면 투표 자체가 불가능
→ BUT SP1에서 만든 votePessimistic()은 DB만으로 동작
→ 느리지만(1.73s) 서비스는 유지할 수 있음
```

### Resilience4j Circuit Breaker 도입

Circuit Breaker 패턴으로 Redis 장애를 자동 감지하고, Pessimistic Lock으로 자동 전환합니다.

```
                ┌──────────────────────────────────────┐
                │         Circuit Breaker                │
                │                                        │
  투표 요청 ──→ │  CLOSED ──────→ Redis + SQS (비동기)    │
                │    │                                   │
                │    │ failure rate ≥ 50%                 │
                │    ▼                                   │
                │  OPEN ────────→ Pessimistic Lock       │
                │    │            (동기, 느리지만 안전)     │
                │    │ 30초 대기                           │
                │    ▼                                   │
                │  HALF_OPEN ──→ Redis 테스트             │
                │    │            (3개 요청만)             │
                │    │ 성공 → CLOSED / 실패 → OPEN        │
                └──────────────────────────────────────┘
```

### 구현

```java
@CircuitBreaker(name = "redisVote", fallbackMethod = "voteFallback")
public VoteResponseDto.VoteResult voteAsync(Long entryId, Integer memberId) {
    // Redis + SQS 비동기 투표 (정상 경로)
    List<Object> results = rankingRedisService.recordAndIncrementPipelined(
            entryId, memberId, challengeId);
    // ...
}

/**
 * Circuit Breaker OPEN 시 자동 호출
 * - SP1의 Pessimistic Lock이 Fallback 전략으로 다시 활용됨
 */
public VoteResponseDto.VoteResult voteFallback(Long entryId, Integer memberId, Exception e) {
    log.warn("[CircuitBreaker] Redis unavailable, falling back to pessimistic lock: entryId={}, cause={}",
            entryId, e.getClass().getSimpleName());
    return self.votePessimistic(entryId, memberId);
}
```

### Circuit Breaker 설정 근거

```yaml
resilience4j:
  circuitbreaker:
    instances:
      redisVote:
        failure-rate-threshold: 50        # 최근 10건 중 5건 실패 → OPEN
        sliding-window-size: 10           # 최근 10건으로 판단 (빠른 감지)
        wait-duration-in-open-state: 30s  # Redis 일시적 불안정 복구 대기
        permitted-number-of-calls-in-half-open-state: 3  # 3건으로 복구 확인
        slow-call-duration-threshold: 2s  # Redis 응답 2초 초과 = slow call
        slow-call-rate-threshold: 80      # slow call 80% 이상 = 장애
```

| 설정 | 값 | 근거 |
|------|-----|------|
| slidingWindowSize | 10 | Redis 장애는 갑작스럽고 명확 → 적은 샘플로 빠르게 감지 |
| failureRateThreshold | 50% | 10건 중 5건 실패 = 충분히 불안정 |
| waitDurationInOpenState | 30초 | Redis 일시적 네트워크 끊김 복구 시간 고려 |
| slowCallDurationThreshold | 2초 | Redis 정상 응답 < 10ms, 2초는 명백한 이상 |
| slowCallRateThreshold | 80% | 느린 응답이 80% 이상이면 Redis 과부하 판단 |

## 5. SP1의 Pessimistic Lock이 Fallback으로 재탄생

SP1에서 Pessimistic Lock → 비동기로 전환할 때, Pessimistic Lock은 "버려진 전략"이 아니었습니다.

```
SP1의 여정:
  Pessimistic Lock (정합성 100%, p95 1.73s)
  → "너무 느리다" → Redis + SQS 비동기로 전환
  → Pessimistic Lock은 코드에 남아있지만 안 쓰임

SP2의 재발견:
  Redis 장애 시 → Circuit Breaker OPEN → voteFallback()
  → "느리지만 안전한" Pessimistic Lock이 Fallback 전략으로 부활

  느린 것이 단점이었지만, 장애 상황에서는:
  → DB만으로 동작 (Redis 의존성 없음)
  → 100% 정합성 보장 (Lock 기반)
  → 서비스 중단 0 (p95 1.73s지만 동작함)
```

**"성능 최적화에서 버려진 전략이, 장애 대응에서 마지막 방어선이 되었다."**

## 6. 정합성 보장의 다층 방어 설계

4가지 장애 시나리오에 대해, 4개 레이어가 단계적으로 방어합니다.

```
┌─────────────────────────────────────────────────┐
│  L1: Redis SADD (API 레벨 중복 차단)              │
│  → SADD 결과 0 = 이미 투표 → 즉시 거부            │
│  → O(1), 네트워크 1 RTT                          │
├─────────────────────────────────────────────────┤
│  L2: DB Unique Key (Consumer 레벨 최종 방어)      │
│  → SQS at-least-once에서 중복 메시지 도착 시       │
│  → DataIntegrityViolationException → 안전 무시    │
├─────────────────────────────────────────────────┤
│  L3: VoteConsistencyScheduler (5분 주기 검증)     │
│  → Redis ↔ DB 투표 수 비교                       │
│  → 불일치 시 DB 기준으로 Redis 동기화              │
│  → SQS 지연과 전송 실패 구분 (30초 대기 후 재검증)  │
├─────────────────────────────────────────────────┤
│  L4: Circuit Breaker (Redis 장애 시 Fallback)    │
│  → Redis 장애 감지 → Pessimistic Lock 자동 전환   │
│  → 서비스 중단 없이 투표 처리 계속                  │
│  → Redis 복구 후 비동기 복귀                      │
└─────────────────────────────────────────────────┘
```

### 장애 시나리오 × 방어 레이어 매핑

| 장애 시나리오 | 방어 레이어 | 결과 |
|-------------|-----------|------|
| SQS 전송 실패 | 보상 트랜잭션 (Redis 롤백) | 데이터 유실 방지 |
| Redis 전체 장애 | L4 Circuit Breaker → Pessimistic Lock | 서비스 중단 0 |
| Redis 재시작 (SET 소실) | L2 DB UK (최종 방어) | 중복 투표 DB에서 차단 |
| SQS 중복 전달 | L1 exists 체크 + L2 DB UK | Idempotency 보장 |
| Redis ↔ DB 불일치 | L3 ConsistencyScheduler | 5분 내 자동 보정 |

## 7. Eventual Consistency 모델과 정합성 타임라인

### 3-Layer 정합성 모델

비동기 시스템에서 **"데이터가 언제 어디에 반영되는가"**를 명확히 정의해야 합니다.

```
투표 발생 시점 (t=0)
  │
  ├─ L1: Redis Sorted Set ──→ 즉시 반영 (5ms)
  │   → 사용자가 보는 랭킹에 즉시 반영
  │   → ZINCRBY O(log N), 원자적 증가
  │
  ├─ L2: SQS → Consumer → DB ──→ 수 초 후 반영 (500ms~5s)
  │   → 영속 저장소에 최종 기록
  │   → Vote INSERT + Entry voteCount UPDATE
  │
  └─ L3: ConsistencyScheduler ──→ 5분 주기 검증
      → Redis ↔ DB 불일치 시 DB 기준으로 보정
      → "SQS 지연"과 "SQS 실패"를 구분 (30초 대기 후 재검증)
```

### 정합성 윈도우 분석

```
시나리오별 정합성 상태:

[정상 흐름] (99.9% 케이스)
  t=0ms:   Redis 반영 ✓ (사용자가 보는 값)
  t=500ms: DB 반영 ✓ (영속 데이터)
  → 정합성 윈도우: 500ms (사용자 체감 없음)

[SQS 지연] (0.09% 케이스)
  t=0ms:   Redis 반영 ✓
  t=5s:    DB 반영 ✓ (Consumer 처리 지연)
  → 정합성 윈도우: 5초 (비즈니스적으로 무해)

[SQS 전송 실패] (0.01% 케이스)
  t=0ms:   Redis 반영 ✓
  t=10ms:  SQS 실패 감지 → Redis 보상 트랜잭션 (롤백)
  → 정합성 윈도우: 10ms (사용자 에러 응답)
  → 데이터 유실: 0 (보상 트랜잭션으로 방지)
```

### Source of Truth 설계

```
질문: "Redis와 DB 중 누가 진짜인가?"

답: 용도에 따라 다르다.

[실시간 랭킹] → Redis가 Source of Truth
  → 투표 즉시 반영 (사용자 경험)
  → Sorted Set ZREVRANGE로 O(log N) 조회

[영속 데이터] → DB가 Source of Truth
  → 최종 투표 기록, 챌린지 결과 결정
  → ConsistencyScheduler가 Redis를 DB 기준으로 보정

[불일치 시] → DB 기준 (Redis는 "캐시"의 성격)
  → Scheduler가 Redis > DB 감지 → 30초 대기 → 재검증 → DB로 동기화
  → 보상 트랜잭션이 있으므로 불일치 자체가 거의 발생하지 않음
```

## 8. SQS Consumer의 Idempotency 설계

SQS Standard Queue는 **at-least-once** 전달 보장입니다. 같은 메시지가 2번 올 수 있습니다.

```java
@SqsListener("${app.sqs.vote-queue-name:petstar-votes}")
@Transactional
public void handleVote(String payload) {
    VoteMessage message = objectMapper.readValue(payload, VoteMessage.class);

    // 1차 방어: 애플리케이션 레벨 중복 체크
    if (voteRepository.existsByEntryIdAndMemberId(entryId, memberId)) {
        return;  // 이미 처리됨 → 안전 무시
    }

    // 2차 방어: DB Unique Constraint
    try {
        voteRepository.save(vote);
        entryRepository.incrementVoteCount(entryId);  // Atomic Update
    } catch (DataIntegrityViolationException e) {
        // UK 위반 = 이미 처리됨 → 안전 무시
    }
}
```

**왜 2중 방어인가?**

```
1차 방어만 있으면:
  Thread A: existsBy → false → save 시작
  Thread B: existsBy → false → save 시작 (Race Condition!)
  → 둘 다 통과 → 중복 저장

2차 방어(UK)만 있으면:
  매번 save 시도 → UK 예외 → 예외 처리 오버헤드
  → 트랜잭션 롤백 + 예외 로깅 비용

둘 다 있으면:
  대부분 1차에서 걸러짐 (빠름, DB 쿼리 1회)
  극히 드문 Race Condition만 2차에서 방어 (안전)
```

## 8. 전체 여정 요약

```
[1막] 비동기 전환 성공
  Hot Spot Lock 병목 → Redis + SQS 비동기로 전환
  → 투표 p95: 1.73s → 50ms (97% 개선) ✓
  → BUT "장애가 발생하면 데이터는 어떻게 되는가?"

[2막] 장애 시나리오 식별
  "비동기는 빠르지만, 장애 시 데이터가 사라질 수 있다"
  → SQS 전송 실패: Redis에만 투표 남음 → 5분 후 유실
  → Redis 장애: 투표 API 전면 중단
  → Redis 재시작: 중복 투표 방어 붕괴
  → ConsistencyScheduler: 지연과 실패 구분 불가

[3막] 보상 트랜잭션 + Circuit Breaker
  → SQS 실패 시 Redis 롤백 (ZINCRBY -1 + SREM)
  → Redis 장애 시 Pessimistic Lock 자동 Fallback
  → 4개 레이어 다층 방어 (SADD → UK → Scheduler → Circuit Breaker)
  → SP1의 Pessimistic Lock이 "마지막 방어선"으로 재탄생
```

### 핵심 교훈

1. **비동기 시스템의 숨겨진 비용** — 성능은 97% 개선되지만, 장애 대응 설계가 없으면 데이터 유실이라는 더 큰 비용을 지불한다
2. **보상 트랜잭션의 순서가 정합성을 결정** — 랭킹 복구(ZINCRBY -1)를 투표 기록 삭제(SREM)보다 먼저 실행해야, 보상 도중 장애 시에도 랭킹 왜곡을 방지
3. **Graceful Degradation** — Circuit Breaker로 "빠른 경로(Redis, 50ms)"가 실패하면 "느린 경로(DB Lock, 1.73s)"로 자동 전환, 서비스 중단 0
4. **이전 단계의 코드가 Fallback 자산** — Pessimistic Lock은 비동기 전환 시 "느리다"고 교체됐지만, 장애 대응에서 "안전하다"는 가치로 재활용됨
