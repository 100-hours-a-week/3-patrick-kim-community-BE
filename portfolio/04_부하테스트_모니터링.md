# 부하테스트 & 모니터링 기반 병목 분석

## 1. 왜 부하테스트가 필요한가?

### 기능 테스트의 한계

```
기능 테스트:
  "로그인이 되나요?" → 됩니다 ✓
  "투표가 되나요?" → 됩니다 ✓
  "랭킹이 보이나요?" → 보입니다 ✓

부하 테스트:
  "100명이 동시에 투표하면?" → ???
  "10만 개 데이터에서 랭킹 조회하면?" → ???
  "1시간 동안 지속적으로 요청하면?" → ???
```

**PetStar 서비스 특성**:
- 챌린지 종료 직전 **트래픽 집중**
- 인기 Entry에 **동시 투표**
- 실시간 **랭킹 경쟁**

→ **동시 접속 상황에서의 성능**이 서비스 품질을 결정

### 2-2. 테스트 데이터 규모

**"실제 서비스와 유사한 데이터량"**으로 테스트해야 의미 있는 결과

| 테이블 | 레코드 수 | 설계 근거 |
|--------|----------|----------|
| Member | 10,000 | 초기 서비스 사용자 규모 |
| Pet | 15,000 | 사용자당 평균 1.5마리 |
| Challenge | 50 | 진행 중/예정/종료 챌린지 |
| **Entry** | **100,000** | 챌린지당 평균 2,000개 출품 |
| **Vote** | **1,000,000** | Entry당 평균 10표 |

**왜 10만 Entry, 100만 Vote인가?**

```
데이터량이 적으면 문제가 숨겨짐:

1,000건 Entry:
  SELECT ... ORDER BY vote_count DESC LIMIT 10;
  → 1,000건 정렬 → 빠름 (문제 안 보임)

100,000건 Entry:
  SELECT ... ORDER BY vote_count DESC LIMIT 10;
  → 100,000건 정렬 → 느림 (문제 발견!)
  → 인덱스 필요성 명확해짐
```

### 2-3. 데이터 시딩

MySQL Stored Procedure로 대량 데이터 생성:

```sql
-- Entry 100,000건 생성 (각 Challenge에 2,000개씩)
DELIMITER //
CREATE PROCEDURE generate_entries()
BEGIN
    DECLARE i INT DEFAULT 1;
    DECLARE challenge_id INT;

    WHILE i <= 100000 DO
        SET challenge_id = (i % 50) + 1;  -- 50개 챌린지에 분산

        INSERT INTO entry (challenge_id, pet_id, image_id, vote_count, created_at)
        VALUES (
            challenge_id,
            (i % 15000) + 1,  -- 15,000 pets에 분산
            i,
            FLOOR(RAND() * 200),  -- 0~199 랜덤 투표수
            NOW()
        );

        SET i = i + 1;
    END WHILE;
END //
DELIMITER ;
```

## 3. 부하테스트 시나리오

### 3-1. k6 선택 이유

| 도구 | 장점 | 단점 |
|------|------|------|
| JMeter | GUI, 다양한 프로토콜 | 무거움, 스크립트 복잡 |
| Locust | Python 기반, 유연 | Python 환경 필요 |
| **k6** | **경량, JavaScript, CLI 친화적** | HTTP/gRPC 위주 |
| Gatling | Scala DSL, 상세 리포트 | Scala 학습 곡선 |

**k6 선택 이유**:
- JavaScript로 시나리오 작성 (익숙함)
- CLI 기반 (CI/CD 통합 용이)
- 경량 (별도 서버에서 실행 용이)
- Prometheus/Grafana 연동 가능

### 3-2. 테스트 시나리오 설계

**핵심 질문**: "어떤 사용자 행동을 시뮬레이션해야 하는가?"

PetStar에서 가장 빈번하고 부하가 높은 시나리오를 분석했습니다:

```
실제 사용자 플로우 분석:

1. 앱 진입 → 챌린지 목록 조회 (홈 화면)
   - 가장 빈번한 API 호출
   - 상대적으로 단순한 쿼리

2. 관심 챌린지 클릭 → 참여작 목록 조회
   - 페이징 필요 (10개씩)
   - Entry + Pet + Image JOIN

3. 랭킹 탭 클릭 → 랭킹 조회 ⬅️ 가장 부하 높음!
   - ORDER BY vote_count DESC
   - Entry + Pet + Image JOIN
   - 정렬 + 조인 = 복합 쿼리

4. 투표 (인증 필요)
   - 동시성 테스트는 별도 시나리오
```

**k6 스크립트 구현**:

```javascript
// k6/01_basic-load-test.js

// ACTIVE 상태 챌린지만 테스트 (시딩 데이터 기준)
var CHALLENGE_IDS = [1, 2, 3, ..., 30];  // 30개 챌린지

export default function () {
    // 랜덤 챌린지 선택 (트래픽 분산)
    var challengeId = getRandomElement(CHALLENGE_IDS);

    // 1. 챌린지 목록 조회
    var res = http.get(BASE_URL + '/api/v1/challenges');
    challengeListTrend.add(res.timings.duration);  // 응답시간 측정
    check(res, { 'status 200': (r) => r.status === 200 });

    sleep(getRandomInt(1, 2));  // 사용자 탐색 시간

    // 2. 참여작 목록 조회
    res = http.get(BASE_URL + '/api/v1/challenges/' + challengeId + '/entries?limit=10');
    entriesTrend.add(res.timings.duration);

    sleep(getRandomInt(1, 2));

    // 3. 랭킹 조회 (메인 타겟)
    res = http.get(BASE_URL + '/api/v1/challenges/' + challengeId + '/ranking?limit=10');
    rankingTrend.add(res.timings.duration);

    sleep(getRandomInt(1, 2));
}
```

**스크립트 설계 의도**:

| 요소 | 설계 | 근거 |
|------|------|------|
| `sleep(1~2초)` | 랜덤 대기 | 실제 사용자 탐색 시간 반영 |
| `랜덤 challengeId` | 30개 챌린지 분산 | 단일 챌린지 집중 방지 |
| `limit=10` | 기본 페이지 크기 | 실제 서비스 설정 |
| `Custom Trend` | API별 응답시간 측정 | 병목 API 식별 |

**테스트 대상 API 선정 이유**:

| API | 예상 부하 | 선정 이유 |
|-----|----------|----------|
| challenges | 낮음 | 단순 목록 조회, 비교 기준 |
| entries | 중간 | N+1 문제 발생 가능 |
| **ranking** | **높음** | ORDER BY + JOIN, 핵심 최적화 대상 |

**투표 API를 제외한 이유**:
- 인증 필요 (JWT 토큰)
- 동시성 테스트는 별도 시나리오로 분리 (Phase 6)
- 읽기 부하 테스트와 쓰기 부하 테스트 분리

### 3-3. 부하 단계 설계

**점진적 부하 증가**로 한계점 파악:

```javascript
export const options = {
    scenarios: {
        // Stage 1: Warm up (JIT 컴파일, Connection Pool 초기화)
        warmup: {
            executor: 'constant-vus',
            vus: 10,
            duration: '30s',
        },
        // Stage 2: Normal load (평상시 트래픽)
        normal_load: {
            executor: 'constant-vus',
            vus: 100,
            duration: '2m',
            startTime: '30s',
        },
        // Stage 3: Peak load (피크 시간)
        peak_load: {
            executor: 'constant-vus',
            vus: 200,
            duration: '2m',
            startTime: '2m30s',
        },
        // Stage 4: Stress test (한계 테스트)
        stress: {
            executor: 'constant-vus',
            vus: 300,
            duration: '1m',
            startTime: '4m30s',
        },
    },
    thresholds: {
        http_req_duration: ['p(95)<500'],  // p95 < 500ms
        errors: ['rate<0.05'],              // 에러율 < 5%
    },
};
```

**VUs(Virtual Users) 설계 근거**:
```
동시 접속 100명 ≈ DAU 1,000~2,000명
동시 접속 200명 ≈ DAU 2,000~4,000명
동시 접속 300명 ≈ DAU 3,000~6,000명

→ 초기 서비스 ~ 성장 단계 커버
```

### 4-2. 핵심 모니터링 지표

**1. HikariCP (Connection Pool)**

| 지표 | 의미 | 정상 범위 |
|------|------|----------|
| `hikaricp_connections_active` | 사용 중인 연결 수 | < max pool size |
| `hikaricp_connections_pending` | 대기 중인 요청 수 | 0에 가까워야 함 |
| `hikaricp_connections_acquire_seconds` | 연결 획득 시간 | < 100ms |

**왜 Connection Pool이 중요한가?**
```
50 VUs 동시 요청, Pool Size = 10인 경우:

요청 1~10: 즉시 Connection 획득
요청 11~50: Connection 대기... (Pool 부족)

→ 응답 시간 = 실제 쿼리 시간 + Connection 대기 시간
→ Pool 부족하면 대기 시간이 응답 시간 대부분 차지
```

**2. JVM 메트릭**

| 지표 | 의미 | 주의 사항 |
|------|------|----------|
| `jvm_memory_used_bytes` | 힙 메모리 사용량 | 90% 이상 지속 시 OOM 위험 |
| `jvm_gc_pause_seconds` | GC 멈춤 시간 | 100ms 이상이면 응답 지연 |

**3. HTTP 요청 메트릭**

| 지표 | 의미 | 목표 |
|------|------|------|
| `http_server_requests_seconds_p95` | p95 응답 시간 | < 500ms |
| `http_server_requests_seconds_count` | 요청 수 | RPS 계산에 활용 |

## 5. 병목 발견 과정

### 5-1. 첫 번째 병목: Connection Pool

**발견 경로**: Grafana → HikariCP 메트릭

```
부하 테스트 시작 (50 VUs)
    ↓
Grafana에서 이상 징후 발견:
  - hikaricp_connections_pending: 40 (높음!)
  - hikaricp_connections_acquire_seconds: 1.79s (매우 높음!)
    ↓
원인 분석:
  - Pool Size: 10 (Spring Boot 기본값)
  - 동시 요청: 50
  - 40개 요청이 Connection 대기
    ↓
해결:
  - maximum-pool-size: 30
    ↓
검증:
  - Connection 획득 시간: 1.79s → 0.19s (89% 개선)
```

### 5-2. 두 번째 병목: 대량 데이터 로드

**발견 경로**: p6spy 로그 분석

```
API 응답 느림 (ranking p95 = 4.76s)
    ↓
p6spy 로그 확인:
  SELECT * FROM entry WHERE challenge_id = 1
  ORDER BY vote_count DESC;
  -- 실행 시간: 3.2초
  -- 반환 행: 100,000건 (!)
    ↓
원인 분석:
  - LIMIT 없이 전체 조회
  - Java에서 stream().limit(10) 적용
  - 100K row가 이미 메모리에 로드된 후
    ↓
해결:
  - Pageable 파라미터로 DB LIMIT 적용
    ↓
검증:
  - p95: 4.76s → 884ms (81% 개선)
```

### 5-3. 세 번째 병목: N+1 쿼리

**발견 경로**: p6spy 쿼리 카운트

```
API 호출 1회당 쿼리 수 확인
    ↓
p6spy 로그:
  Query 1: SELECT * FROM entry ... (1회)
  Query 2: SELECT * FROM pet WHERE pet_id = 101; (10회)
  Query 3: SELECT * FROM image WHERE image_id = 201; (10회)
  -- 총 21개 쿼리!
    ↓
원인 분석:
  - JPA Lazy Loading
  - Mapper에서 entry.getPet().getName() 접근
  - 각 접근마다 SELECT 쿼리 발생
    ↓
해결:
  - Fetch Join으로 한 번에 로드
    ↓
검증:
  - 쿼리 수: 21개 → 1개 (95% 감소)
  - p95: 884ms → 234ms (73% 개선)
```

### 5-4. 네 번째 병목: filesort

**발견 경로**: MySQL EXPLAIN

```
인덱스 추가 검토
    ↓
EXPLAIN 실행:
  | type | key         | Extra          |
  |------|-------------|----------------|
  | ref  | uk_entry... | Using filesort |
  -- Using filesort = 인덱스 정렬 불가
    ↓
원인 분석:
  - 기존 인덱스: (challenge_id, pet_id)
  - ORDER BY: vote_count DESC
  - 인덱스에 vote_count 없음 → 별도 정렬
    ↓
해결:
  - 복합 인덱스: (challenge_id, vote_count DESC)
    ↓
검증:
  - EXPLAIN Extra: NULL (filesort 제거)
  - p95: 234ms → 146ms (38% 개선)
```

## 7. 핵심 교훈

### 1. 모니터링 없이 최적화 없다

```
"느리다" → 어디서? 얼마나? 왜?

Grafana: 어디서 병목인가? (Connection Pool? GC?)
```

### 2. 실제 데이터량으로 테스트해야 문제가 보인다

```
개발 환경 (100건): 모든 게 빠름
운영 환경 (10만건): 갑자기 느림

→ 테스트 데이터는 운영 규모와 유사하게
```

### 3. 최적화 순서가 효과에 영향

```
잘못된 순서:
  인덱스 먼저 → 100K 로드 문제 여전 → 효과 미미

올바른 순서:
  1. 데이터량 줄이기 (Pageable)
  2. 쿼리 수 줄이기 (Fetch Join)
  3. 쿼리 속도 높이기 (Index)
  → 각 단계별 효과 측정 가능
```

### 4. 부하 테스트는 점진적으로

```
갑자기 1000 VUs → 서버 다운 → 원인 파악 어려움

10 → 50 → 100 → 200 → 300 VUs
→ 어느 시점에서 병목 발생하는지 파악 가능
→ 서비스 한계점(capacity) 측정
```
