# SP3. 읽기 성능 최적화 — 병목 순서대로 해결

> **핵심 질문**: "복합 병목을 어떤 순서로 해결할 것인가?"

---

## 1. 비즈니스 문제

100 VUs에서 랭킹 API p95가 **3.83초**입니다. 사용자 체감 기준 3초를 초과하면 이탈률이 급증합니다.

```
첫 부하 테스트 결과 (100 VUs):

| API          | p95     | 판정   |
|--------------|---------|--------|
| 챌린지 목록  | 2.94s   | FAIL   |
| 엔트리 목록  | 2.09s   | FAIL   |
| 랭킹 조회    | 3.83s   | FAIL   |
| 전체         | 3.01s   | FAIL   |
| RPS          | 32.9    |        |

목표: p95 < 200ms, 동시 200명 안정 처리
```

### 이상 현상 3가지

1. **랭킹 API가 유독 느림**: 엔트리 2.09s인데 랭킹 3.83s
2. **VUs 증가해도 RPS 정체**: 50→100→200명으로 늘려도 RPS ~33으로 고정
3. **응답 시간 비선형 증가**: VUs 2배 → 응답 시간 2배 이상

RPS가 정체된다는 건 **어딘가에 병목**이 있다는 뜻입니다.

---

## 2. 병목 분석: "어디가 문제인가?"

"느리다"는 현상만으로는 해결할 수 없습니다. **정확히 어디서 시간이 소요되는지** 파악해야 합니다.

```
요청 처리 과정에서 시간 소요 가능한 곳:

Client → Network → WAS → Connection Pool → DB → 쿼리 실행 → 결과 전송
                              ↑                     ↑
                         여기서 대기?           여기서 느림?
```

### 분석 도구

| 도구 | 역할 | 발견한 것 |
|------|------|----------|
| **Grafana** (Prometheus) | HikariCP 메트릭 | Connection Pool 포화 |
| **p6spy** | 실행 SQL 로깅 | 100K row 로드, N+1 쿼리 |
| **EXPLAIN** | 쿼리 실행 계획 | filesort (인덱스 미활용) |

### 1번째 발견: Connection Pool 병목

```
Grafana HikariCP 메트릭 (50 VUs):
  Pool Size: 10 (Spring Boot 기본값)
  Active: 10 (풀 포화)
  Pending: 40 (대기 중인 요청)
  Acquire Time: 평균 1.79초

→ 50개 요청 중 40개가 DB 쿼리 실행 전에 이미 대기!
→ 응답 시간의 대부분이 커넥션 대기 시간
```

### 2번째 발견: 100K row 전체 로드

```java
// p6spy 로그로 발견한 문제 코드
List<Entry> entries = repository.findByChallenge(challengeId);  // 100K 로드!
return entries.stream()
    .limit(10)  // ← Java에서 10개 자르기 (이미 100K 로드 완료)
    .map(...)
    .toList();
```

**Java의 Stream.limit()과 DB의 LIMIT은 다릅니다:**

| 위치 | 동작 | 네트워크 비용 | 메모리 비용 |
|------|------|-------------|------------|
| Java Stream.limit() | 이미 로드된 데이터에서 자름 | 전체 데이터 전송 | 전체 객체 생성 |
| DB LIMIT | 쿼리 결과 자체를 제한 | 필요한 만큼만 | 필요한 만큼만 |

### 3번째 발견: N+1 쿼리

```sql
-- p6spy 로그: 랭킹 API 1회 호출당 21개 쿼리
SELECT * FROM entry WHERE challenge_id = 1 ORDER BY vote_count DESC;  -- 1회
SELECT * FROM pet WHERE pet_id = 101;  -- 10회 (Lazy Loading)
SELECT * FROM image WHERE image_id = 201;  -- 10회 (Lazy Loading)
-- 총 1 + 10 + 10 = 21개 쿼리
```

Entry 목록을 가져온 후, 각 Entry의 Pet과 Image에 접근할 때마다 **추가 쿼리가 발생**합니다. JPA Lazy Loading이 목록 조회에서 N+1 문제를 만듭니다.

### 4번째 발견: filesort

```sql
EXPLAIN SELECT * FROM entry
WHERE challenge_id = 1 ORDER BY vote_count DESC LIMIT 10;

| type | key         | rows | Extra          |
|------|-------------|------|----------------|
| ref  | uk_entry... | 3334 | Using filesort |  ← 문제!
```

기존 인덱스 `(challenge_id, pet_id)`는 `ORDER BY vote_count`에 활용 불가. **임시 테이블에서 별도 정렬** 수행.

### 병목 원인 정리

| 순서 | 원인 | 발견 도구 | 증거 |
|------|------|----------|------|
| 1 | Connection Pool 부족 | Grafana | 획득 시간 1.79s, Pending 40 |
| 2 | LIMIT 미적용 | p6spy | 100K row 로드 |
| 3 | N+1 쿼리 | p6spy | 21개 쿼리 |
| 4 | filesort | EXPLAIN | Using filesort |

## 3. 최적화 순서: 왜 순서가 중요한가?

여러 병목을 발견했는데, **어떤 순서로 해결해야 할까?**

```
잘못된 접근:
  "인덱스 먼저 추가하자"
  → 100K row 로드 문제가 여전히 존재
  → 인덱스 효과 측정 불가

올바른 접근:
  1. 데이터량 먼저 줄이기 (Pageable)
  2. 쿼리 수 줄이기 (Fetch Join)
  3. 쿼리 속도 높이기 (Index)
  → 각 단계별 효과 측정 가능
```

**원칙**: 상위 레이어의 문제를 먼저 해결해야, 하위 레이어 최적화의 효과를 정확히 측정할 수 있습니다.

---

## 4. 해결 과정

### 4-0. Connection Pool 튜닝

**문제**: 동시 요청 50개 vs Pool 10개 → 40개가 대기

```yaml
# application-dev.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 30    # 10 → 30
      minimum-idle: 10
```

Pool 크기를 무작정 키우면 안 됩니다. RDS db.t3.micro의 `max_connections ≈ 60~80`이고, HikariCP 권장 공식 `(core_count × 2) + spindle_count`를 고려하여 30으로 설정했습니다.

**결과**: Connection 획득 시간 1.79s → 0.19s (**89% 개선**)

이제 실제 쿼리 성능을 측정할 수 있습니다.

### 4-1. Pageable 적용 — "DB가 잘하는 건 DB에게"

```java
// Before: Java에서 limit
List<Entry> entries = repository.findAllOrderByVoteCountDesc(challengeId);
return entries.stream().limit(10).toList();  // 100K 로드 후 10개 자르기

// After: DB에서 LIMIT
List<Entry> entries = repository.findAllOrderByVoteCountDesc(
    challengeId,
    PageRequest.of(0, 10)  // → SQL: ... LIMIT 10
);
```

**결과**: p95 4.76s → 884ms (**81% 개선**)

### 4-2. Fetch Join — N+1 해결

```java
// Before: 21개 쿼리
List<Entry> entries = repository.findTop10(challengeId);
for (Entry entry : entries) {
    entry.getPet().getName();   // → SELECT from pet (10회)
    entry.getImage().getUrl();  // → SELECT from image (10회)
}

// After: 1개 쿼리
@Query("SELECT e FROM Entry e " +
       "JOIN FETCH e.pet " +
       "JOIN FETCH e.image " +
       "WHERE e.challenge.id = :id " +
       "ORDER BY e.voteCount DESC")
List<Entry> findTopEntriesWithFetchJoin(Long id, Pageable pageable);
```

Fetch Join을 선택한 이유: 랭킹 조회 시 Pet, Image가 **항상** 필요하므로, 명시적으로 한 번에 로드하는 것이 최적입니다.

**결과**: 쿼리 21개 → 1개 (**95% 감소**), p95 884ms → 234ms (**73% 개선**)

### 4-3. 인덱스 최적화 — filesort 제거

```sql
-- 복합 인덱스 설계 원칙:
-- 1. WHERE 조건 컬럼 (등치 조건)
-- 2. ORDER BY 컬럼

-- 기존: (challenge_id, pet_id) → vote_count 정렬 불가
-- 신규: (challenge_id, vote_count DESC) → WHERE + ORDER BY 모두 커버

CREATE INDEX idx_entry_challenge_vote
ON entry(challenge_id, vote_count DESC);
```

```
EXPLAIN Before: type=ref, Extra=Using filesort  ← 정렬 필요
EXPLAIN After:  type=ref, Extra=NULL             ← 인덱스로 정렬 완료
```

**결과**: p95 234ms → 146ms (**38% 개선**)

### 단계별 개선 효과

| 단계 | p95 | 개선율 | 누적 |
|------|-----|--------|------|
| Before | 4.76s | - | - |
| +Connection Pool | (대기 89%↓) | - | - |
| +Pageable | 884ms | 81% | 81% |

| +Index | 146ms | 38% | **97%** |

## 6. 핵심 교훈

### 1. 측정 없이 최적화 없다

```
"느리다" → 어디서? 얼마나? 왜?

Grafana: 어디서 병목인가? (Connection Pool? GC?)
p6spy:   어떤 쿼리가 느린가? 몇 개 실행되는가?
EXPLAIN: 왜 쿼리가 느린가? (filesort? full scan?)

→ 측정 기반 최적화가 "감" 기반보다 정확하고 빠르다
```

### 2. DB가 잘하는 건 DB에게

```java
// ❌ Java에서 처리
entries.stream().limit(10)...
entries.stream().sorted()...

// ✅ DB에서 처리
SELECT ... LIMIT 10
SELECT ... ORDER BY ... (with index)
```

정렬, 필터링, 페이징은 DB가 인덱스를 활용하여 처리할 때 가장 효율적입니다.

### 3. 한 번에 하나씩, 효과 측정

```
여러 최적화를 동시에 적용하면:
→ "어떤 변경이 얼마나 효과가 있었는가?" 알 수 없음

순차 적용 + 각 단계 측정:
→ Pageable 81%, Fetch Join 73%, Index 38%
→ 각각의 기여도를 정확히 파악
→ 포트폴리오에서 "왜 이 순서인가?" 설명 가능
```

### 4. 최적화 순서의 원칙

```
상위 레이어부터 해결:
  1. 데이터량 (100K → 10) — 가장 큰 문제 먼저
  2. 쿼리 수 (21 → 1) — 네트워크 왕복 감소
  3. 쿼리 속도 (filesort 제거) — 쿼리 하나의 성능

하위 문제를 먼저 풀면 상위 문제에 가려져 효과가 보이지 않음
```
