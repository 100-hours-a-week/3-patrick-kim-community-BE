# PetStar — 반려동물 사진 콘테스트 플랫폼 포트폴리오

> **한 줄 요약**: 투표 Hot Spot 동시성 → 비동기 전환 → 장애 대응 → 혼합 부하 가설 검증까지, "측정 → 가설 → 실험 → 검증"으로 풀어낸 성능 엔지니어링 포트폴리오

---

## 프로젝트 소개

**PetStar**는 반려동물 사진 콘테스트 플랫폼입니다. 사용자들이 반려동물 사진을 챌린지에 출품하고, 투표로 실시간 랭킹이 결정되어 우승자가 가려집니다.

**"투표 → 실시간 랭킹 → 우승자 결정"** 이 구조가 PetStar만의 기술적 도전을 만들어냅니다:
- 1표도 누락/중복되면 안 됨 (투표가 곧 순위)
- 인기 사진에 수백 명이 동시 투표 (Hot Spot)
- 읽기와 쓰기가 동시에 발생하는 혼합 부하

### 기술 스택

| 영역 | 기술 |
|------|------|
| **Backend** | Spring Boot 3.5.6, Java 21, JPA, QueryDSL |
| **Database** | MySQL 8.0 (AWS RDS), HikariCP |
| **Cache** | Redis 7 (Sorted Set, Lettuce Pool), Caffeine L1 |
| **Message Queue** | AWS SQS (Standard Queue + DLQ) |
| **Resilience** | Resilience4j Circuit Breaker |
| **Auth** | JWT (Access + Refresh Token), Spring Security |
| **Storage** | AWS S3 (이미지 업로드) |
| **Infra** | AWS EC2, Terraform (IaC), Docker, ECR |
| **Monitoring** | Prometheus, Grafana, p6spy, Actuator |
| **Load Test** | k6 (혼합 부하 시나리오) |

## 성능 최적화 전체 여정

PetStar의 최적화는 10개 Phase + 4개 Round에 걸쳐 진행되었습니다. 단계별로 **하나씩 변경하고, 동일 조건에서 재측정하여 효과를 수치로 증명**했습니다.

### Phase 0~10: 기반 구축 → 읽기 최적화 → 쓰기 최적화

```
Phase 0-2: 기반 구축
  Entity/Service 구조 → Terraform 인프라 → Prometheus/Grafana 모니터링

Phase 3: Baseline 측정
  k6 부하 테스트 + 데이터 시딩 (Member 10K, Entry 100K)
  → p95: 4.76s, RPS: 32.9 (100 VUs) — "느리다"의 정량화

Phase 4: 인덱스 최적화
  EXPLAIN 분석 → (challenge_id, vote_count DESC) 복합 인덱스
  → filesort 제거 → p95 38% 개선

Phase 5: N+1 해결
  p6spy로 21개 쿼리 발견 → QueryDSL Fetch Join → 1개 쿼리
  → p95 73% 추가 개선

Phase 6: 동시성 제어
  Race Condition 발견 → 3가지 Lock 전략 비교 → Pessimistic Lock 선택
  → 정합성 100%, BUT p95 1.73s (Hot Spot)

Phase 7: Connection Pool 튜닝
  Grafana HikariCP 메트릭 → Pool 10→30, 획득 시간 89% 개선

Phase 8: Redis 실시간 랭킹
  Sorted Set (ZINCRBY O(log N)) + Cache Warming + Source of Truth 설계

Phase 9: 최종 검증
  읽기 전용 부하: p95 4.76s → 130ms (96% 개선), RPS 32.9 → 308 (9.4배)

Phase 10: 비동기 투표 (Redis + SQS)
  투표 p95: 1.73s → 50ms (97% 개선)
  Redis Pipeline (SADD + ZINCRBY + ZSCORE = 1 RTT)
  Lettuce Connection Pooling (max-active 20)
```

### Round 1~4: 혼합 부하 한계 탐색 → 장애 대응

```
Round 1: 혼합 부하 Baseline
  읽기 70% + 투표 30% → 100 VUs에서 HikariCP 포화 30/30
  p95: 5,439ms, Pending 170 스레드, Acquire 12.4초

Round 2: 가설 검증 실패
  "투표가 병목" 가설 → DB exists → Redis SADD → p95 +7% 악화
  → 데이터 분석: 읽기가 DB 호출의 78%

Round 3: Caffeine L1 캐시
  Controller 레벨 @Cacheable → 캐시 히트 시 DB 커넥션 미획득
  p95: 5,439ms → 48ms (99.1% 개선), 800+ VUs 안정

Round 4: 비동기 파이프라인 장애 대응
  4가지 장애 시나리오 식별 → 보상 트랜잭션 + Circuit Breaker
  Redis 장애 시 Pessimistic Lock 자동 Fallback
```

### 전체 성과 요약

| 영역 | Before | After | 개선율 |
|------|--------|-------|--------|
| 읽기 p95 (300 VUs) | 4.76s | **130ms** | **97%↓** |
| 혼합 p95 (800 VUs) | 5,439ms | **48ms** | **99.1%↓** |
| 투표 p95 (Hot Spot 50명) | 1.73s | **50ms** | **97%↓** |
| 처리량 (RPS) | 32.9 | **308** (읽기) / **220.5** (혼합) | **9.4배↑** |
| 동시 처리 한계 | 100명 | **800명+** | **8배↑** |
| 투표 정합성 | Race Condition 발생 | **100%** (다층 방어) | - |
| Redis 장애 시 | 투표 불가 (500 에러) | **자동 Fallback** (p95 1.73s) | 서비스 중단 0 |

## 3개 셀링포인트

### SP1. Hot Spot 동시성 — 정합성과 성능 사이의 트레이드오프

> **핵심 질문**: "정합성을 유지하면서 어떻게 빠르게 만들 것인가?"

```
[비즈니스 문제]
챌린지 마감 직전, 인기 사진에 수백 명이 동시 투표
→ 1표 = 순위 변동 가능 → 누락/중복 절대 불가

[기술적 여정]
① Race Condition 발견
   - Check-then-Act: 중복 투표 통과
   - Lost Update: 투표 수 누락 (기대 12, 실제 11)

② 3가지 Lock 전략 실험 (50명 동시, 같은 Entry)
   | 전략         | 성공률 | p95   | 정합성     |
   | Pessimistic  | 100%  | 1.73s | 50/50 ✓   |
   | Optimistic   | 16%   | 885ms | 8/50 ✗    |
   | Atomic       | 12%   | 633ms | 6/50 ✗    |
   → Hot Spot에서 Pessimistic이 역설적으로 가장 빠르고 안전

③ Pessimistic Lock의 한계 (p95 1.73s)
   → 50명 순차 처리, 마지막 사람은 49명 대기

④ "사용자가 보는 값"과 "저장되는 값" 분리
   → Redis Sorted Set으로 즉시 랭킹 반영 (O(log N))
   → SQS Fire & Forget으로 비동기 DB 저장
   → Redis Pipeline (SADD + ZINCRBY + ZSCORE = 1 RTT)
   → Strong → Eventual Consistency 트레이드오프 결정

[트레이드오프]
포기: DB 즉시 반영 (수 초 지연)
획득: 투표 p95 1.73s → 50ms, Lock 경합 완전 제거
허용 이유: 사용자가 보는 랭킹(Redis)은 즉시 반영, DB는 백그라운드

[임팩트]
투표 p95: 1.73s → 50ms (97% 개선)
정합성: 100% 유지 (Redis SADD + DB UK 이중 방어)
```

**기술 딥다이브**:
- Redis 캐싱 전략: Write-Through(랭킹) + Cache Aside(초기화) 하이브리드
- Cache Stampede 대응: Caffeine 키별 동기화 + 짧은 TTL + 경량 쿼리
- Hot Key 대응: L1(Caffeine) → L2(Redis) 2단계 캐시로 Redis 읽기 부하 분산
- Cache Invalidation: TTL-only 전략 (히트율 vs 신선도 트레이드오프)
- SQS 설계: at-least-once + DLQ + Consumer 순차 처리(데드락 방지) + Backpressure 흡수

**면접 한 줄**: *"정합성을 포기한 게 아니라, 정합성의 위치를 바꿨다"*

→ [상세 문서](./01_Hot_Spot_동시성_v1.md)

### SP2. 비동기 파이프라인 정합성 — 장애가 발생해도 데이터를 잃지 않는 설계

> **핵심 질문**: "비동기 시스템에서 장애가 발생하면, 데이터 정합성을 어떻게 보장하는가?"

```
[비즈니스 문제]
SP1에서 비동기 전환으로 투표 p95 97% 개선
BUT 분석 결과 데이터 유실 가능한 장애 경로 4가지 발견

[4가지 장애 시나리오]
① SQS 전송 실패: Redis 반영 O, DB 반영 X → 5분 후 투표 유실
② Redis 전체 장애: 투표 API 500 에러 → 서비스 전면 중단
③ Redis 재시작: 중복 투표 체크 SET 소실 → 중복 방어 붕괴
④ Scheduler 한계: SQS "지연"과 "실패"를 구분 불가

[해결]
① 보상 트랜잭션 (Compensating Transaction)
   SQS 전송 실패 → Redis ZINCRBY -1 + SREM (역순으로 롤백)
   → 보상 순서가 정합성을 결정 (랭킹 복구 → 투표 기록 삭제)

② Circuit Breaker (Resilience4j)
   CLOSED → Redis + SQS 비동기 (50ms)
   OPEN   → Pessimistic Lock 자동 전환 (1.73s, 느리지만 안전)
   HALF_OPEN → Redis 복구 확인 후 비동기 복귀
   → 설정 근거: slidingWindow 10, failureRate 50%, slowCall 2s/80%

③ 4개 레이어 다층 방어
   L1: Redis SADD (API 중복 차단, O(1))
   L2: DB Unique Key (Consumer 최종 방어)
   L3: ConsistencyScheduler (5분 주기 Redis↔DB 정합성 검증)
   L4: Circuit Breaker (Redis 장애 시 동기 Fallback)

[SP1과의 연결]
SP1에서 "느리다"고 교체한 Pessimistic Lock이
SP2에서 "안전하다"는 가치로 Circuit Breaker Fallback이 됨
→ 이전 단계의 코드가 Fallback 자산으로 재탄생

[임팩트]
서비스 중단: Redis 장애에도 0 (Graceful Degradation)
데이터 유실: SQS 실패 시에도 0 (보상 트랜잭션)
Actuator 모니터링: /actuator/circuitbreakers로 상태 실시간 확인
```

**기술 딥다이브**:
- Eventual Consistency 모델: 3-Layer 정합성 (Redis 즉시 / DB 수초 / Scheduler 5분)
- Source of Truth 설계: 실시간 랭킹은 Redis, 영속 데이터는 DB
- 정합성 윈도우 분석: 정상 500ms, SQS 지연 5s, SQS 실패 10ms(보상 즉시 롤백)

**면접 한 줄**: *"비동기 시스템은 빠르지만, 장애 대응 없이는 불완전하다. 성능 최적화에서 버려진 전략이 장애 대응에서 마지막 방어선이 되었다"*

→ [상세 문서](./02_비동기_파이프라인_정합성_v1.md)

### SP3. 혼합 부하 가설 검증 — 가설이 틀릴 때 진짜 엔지니어링이 시작된다

> **핵심 질문**: "가설이 틀렸을 때 어떻게 대응할 것인가?"

```
[비즈니스 문제]
읽기 전용 300 VUs: p95 130ms → 좋은 성능
실제 서비스 (읽기 70% + 투표 30%): 100 VUs에서 이미 무너짐
→ 읽기만 테스트하면 보이지 않는 병목이 존재

[Round 1: 병목 발견]
Prometheus → HikariCP Active 30/30 (100 VUs에서 이미 포화)
→ Pending 170 스레드, Acquire Time 12.4초
→ "쿼리는 빠른데(6~19ms) 커넥션 대기가 99%"
→ 직감적 가설: "투표가 DB를 많이 쓰니까 투표가 병목"

[Round 2: 가설이 틀렸다]
투표 DB exists → Redis SADD 전환 (DB 2회 → 1회)
→ p95: 5,439ms → 5,835ms (+7% 악화!)
→ 실패 후 데이터 분석:
  읽기 70% × DB 3회 = 2.1회/VU (전체의 78%)
  투표 30% × DB 2회 = 0.6회/VU (전체의 22%)
→ 교훈: "22%를 아무리 최적화해도 78%가 남아있다" (암달의 법칙)

[Round 3: 올바른 가설로 재도전]
수정 가설: "읽기에 캐시를 적용하면 DB 호출 78%를 제거할 수 있다"
→ Caffeine L1 캐시 (10s TTL, 200 max)
→ Controller 레벨 @Cacheable (캐시 히트 시 DB 커넥션 미획득)
→ p95: 5,439ms → 48ms (99.1% 개선), 800+ VUs 안정

[왜 Controller 레벨인가?]
  Service(@Transactional): 캐시 히트여도 이미 커넥션 점유
  Controller(@Cacheable): 캐시 히트 → DB 커넥션 자체를 획득 안 함
  → 병목이 "커넥션 풀"인 상황에서 캐시 위치가 결정적

[왜 투표 p95도 같이 개선되었는가?]
  Round 3에서 투표 코드는 변경하지 않음
  BUT 투표 p95: 4,149ms → 77ms (98% 개선)
  → 읽기가 더 이상 커넥션을 점유하지 않으니, 투표가 즉시 획득
  → "진짜 병목은 쿼리 속도가 아니라 커넥션 경합이었다"

[임팩트]
전체 p95: 5,439ms → 48ms (99.1% 개선)
한계 VU: 100 → 800+ (8배 확장)
RPS: 120.9 → 220.5 (82.4% 증가)
```

**기술 딥다이브**:
- Cache Aside 패턴 적용과 이유 (읽기 전용 API에 최적)
- Cache Stampede: Caffeine 키별 동기화로 Thundering Herd 방지
- Cache Invalidation: @CacheEvict 미사용 결정 (히트율 > 신선도, 커넥션 풀이 진짜 병목)
- Controller vs Service 캐싱: 커넥션 획득 시점이 핵심

**면접 한 줄**: *"가설이 틀릴 수 있다. 실패한 실험에서 '읽기 78%'라는 핵심 데이터를 얻었고, 그 데이터가 Round 3의 성공을 이끌었다"*

→ [상세 문서](./02_측정기반_최적화_v1.md)

## SP들의 연결 구조

```
SP1 → SP2 → SP3은 독립이 아니라 하나의 스토리로 연결됩니다.

[SP1] Race Condition 발견 → Pessimistic Lock → 비동기 전환
                                    │                 │
                                    │            ┌────┘
                                    ▼            ▼
[SP2] 비동기 장애 시나리오 식별 → 보상 트랜잭션 + Circuit Breaker
                                    │
                  Pessimistic Lock ──┘ (Fallback으로 재탄생)

[SP3] 혼합 부하에서 진짜 병목 탐색
      → Round 2 실패 → 데이터가 가설을 교정
      → Round 3에서 99.1% 개선

전체 스토리:
"동시성 문제 → 비동기로 해결 → 비동기의 장애 대응 →
 실제 혼합 부하에서 가설 검증 → 측정 기반 문제 해결"
```
