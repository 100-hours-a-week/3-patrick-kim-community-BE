# 성능 최적화 종합

## 1. 문제 발견 과정

### 왜 성능 테스트를 시작했는가?

PetStar는 반려동물 사진 콘테스트 플랫폼입니다. **챌린지 종료 시점에 트래픽이 집중**됩니다.

```
일반적인 콘테스트 서비스 트래픽 패턴:

             ▲ 트래픽
             │
    종료직전 │        ████████
             │      ██
    평상시   │ ████
             └─────────────────────▶ 시간
                   챌린지 기간
```

**의문**: "기능은 완성됐는데, 실제로 트래픽을 감당할 수 있을까?"

단순히 API가 동작하는 것과, **동시에 많은 사용자가 접속했을 때도 동작하는 것**은 다른 문제입니다. 이를 검증하기 위해 부하테스트를 시작했습니다.

### 첫 번째 부하테스트 결과

k6로 동시 접속자 수를 점진적으로 늘리며 테스트했습니다.

| VUs | ranking p95 | entries p95 | 전체 p95 | RPS |
|-----|-------------|-------------|----------|-----|
| 50 | 1.79s | 469ms | 1.28s | 27.5 |
| 100 | **3.83s** | **2.09s** | **3.01s** | 32.9 |
| 200 | **10.24s** | **7.13s** | **9.58s** | 32.7 |

**발견한 이상 현상 3가지**:

1. **랭킹 API가 유독 느림**: entries API는 469ms인데 ranking은 1.79s
2. **VUs 증가해도 RPS 정체**: 50→100→200명으로 늘려도 RPS는 27~33으로 비슷
3. **응답 시간 급증**: VUs 2배 증가 시 응답 시간은 2배 이상 증가

**가설**: 어딘가에 병목이 있다. 그게 어디인지 찾아야 한다.

### 목표 설정

PetStar 서비스 특성을 고려한 목표:

| 목표 | 근거 |
|------|------|
| p95 < 200ms | 사용자 체감 기준 (3초 초과 시 이탈률 급증) |
| 동시 200명 | 소규모 서비스 기준 (DAU 2,000~4,000 대응) |
| 정합성 100% | 투표 결과가 랭킹을 결정하므로 데이터 손실 불가 |

## 2. 병목 분석: 어디가 문제인가?

### 분석 전략

"느리다"는 현상만으로는 해결할 수 없습니다. **정확히 어디서 시간이 소요되는지** 파악해야 합니다.

```
요청 처리 과정에서 시간이 소요될 수 있는 곳:

Client → Network → WAS → Connection Pool → DB → 쿼리 실행 → 결과 전송
                          ↑                     ↑
                       여기서 대기?           여기서 느림?
```

**분석 도구 선정**:
- Grafana + Prometheus: Connection Pool 상태 모니터링
- p6spy: 실제 실행되는 SQL 쿼리 로깅
- MySQL EXPLAIN: 쿼리 실행 계획 분석

### 첫 번째 발견: Connection Pool 병목

Grafana에서 HikariCP 메트릭을 확인했습니다.

```
┌─────────────────────────────────────────────────────────────┐
│  HikariCP 메트릭 (50 VUs 테스트 중)                         │
│                                                             │
│  Pool Size: 10 (Spring Boot 기본값)                         │
│  Active Connections: 10 (풀 포화)                           │
│  Pending Requests: 40 (대기 중인 요청)                      │
│  Connection 획득 시간: 평균 1.79초                          │
│                                                             │
│  ┌───────────────────────────────────────────────────┐     │
│  │   동시 50개 요청                                   │     │
│  │         ↓                                         │     │
│  │   Connection Pool 10개 ← 40개가 대기!             │     │
│  │         ↓                                         │     │
│  │   대기 시간만 1.79초 → 응답 시간의 대부분          │     │
│  └───────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────┘
```

**분석**: Connection Pool이 10개인데 동시 요청이 50개면, 40개는 **순수하게 대기만** 합니다. DB 쿼리 실행 전에 이미 병목이 발생하고 있었습니다.

### 두 번째 발견: N+1 쿼리 문제

p6spy로 랭킹 API 호출 시 발생하는 쿼리를 로깅했습니다.

```sql
-- 랭킹 조회 1회당 발생하는 쿼리

-- 1번: Entry 목록 조회
SELECT * FROM entry WHERE challenge_id = 1 ORDER BY vote_count DESC;
-- → 100,000건 전체 로드! (LIMIT 없음)

-- 10번 반복: 각 Entry의 Pet 조회 (Lazy Loading)
SELECT * FROM pet WHERE pet_id = 101;
SELECT * FROM pet WHERE pet_id = 102;
... (10번)

-- 10번 반복: 각 Entry의 Image 조회 (Lazy Loading)
SELECT * FROM image WHERE image_id = 201;
SELECT * FROM image WHERE image_id = 202;
... (10번)

-- 총 21개 쿼리, 100,000건 데이터 메모리 로드
```

**왜 이렇게 되었는가?**

```java
// 문제가 있는 코드
List<Entry> entries = repository.findByChallenge(challengeId);  // 100K 로드
return entries.stream()
    .limit(10)  // ← Java에서 10개 자르기 (이미 100K 로드 완료)
    .map(entry -> {
        entry.getPet().getName();   // ← Lazy Loading → 추가 쿼리
        entry.getImage().getUrl();  // ← Lazy Loading → 추가 쿼리
    })
    .toList();
```

**핵심 문제**:
1. `LIMIT`을 DB가 아닌 Java에서 처리 → 100K row 전체 로드
2. JPA Lazy Loading으로 연관 엔티티 접근 시 추가 쿼리 발생

### 세 번째 발견: 인덱스 미활용

MySQL EXPLAIN으로 쿼리 실행 계획을 분석했습니다.

```sql
EXPLAIN SELECT * FROM entry
WHERE challenge_id = 1
ORDER BY vote_count DESC
LIMIT 10;

-- 결과
+------+-------------+-------+------+------------------------+------+---------+
| type | key         | rows  | Extra                         |
+------+-------------+-------+------+------------------------+------+---------+
| ref  | uk_entry... | 3334  | Using filesort                | ← 문제!
+------+-------------+-------+------+------------------------+------+---------+
```

**"Using filesort"의 의미**:
- 인덱스로 정렬하지 못하고 **임시 테이블에서 정렬** 수행
- 기존 인덱스 `(challenge_id, pet_id)`는 `ORDER BY vote_count`에 활용 불가
- 데이터가 많을수록 정렬 비용 증가

### 병목 원인 정리

| 순서 | 원인 | 발견 도구 | 증거 |
|------|------|----------|------|
| 1 | Connection Pool 부족 | Grafana | 획득 시간 1.79s |
| 2 | DB LIMIT 미적용 | p6spy | 100K row 로드 |
| 3 | N+1 쿼리 | p6spy | 21개 쿼리 |
| 4 | filesort | EXPLAIN | Using filesort |

**핵심 통찰**: 여러 문제가 복합적으로 작용하고 있었습니다. 하나씩 해결하며 **각 해결책의 효과를 측정**해야 합니다.

## 3. 해결 과정: 순서가 중요하다

### 최적화 순서 결정

여러 병목을 발견했는데, **어떤 순서로 해결해야 할까?**

```
잘못된 접근:
  "인덱스 먼저 추가하자" → 효과 측정 불가 (100K 로드 문제가 여전히 존재)

올바른 접근:
  1. 데이터량 먼저 줄이기 (Pageable)
  2. 쿼리 수 줄이기 (Fetch Join)
  3. 쿼리 속도 높이기 (Index)
  → 각 단계별 효과 측정 가능
```

**원칙**: 상위 레이어의 문제를 먼저 해결해야, 하위 레이어 최적화의 효과를 정확히 측정할 수 있습니다.

### 3-1. Connection Pool 튜닝

**문제**: 동시 요청 50개 vs Pool 10개 → 40개가 대기

**Pool 크기를 어떻게 결정할 것인가?**

무작정 크게 하면 안 됩니다. DB 서버의 max_connections에도 제한이 있고, 너무 많은 연결은 오히려 성능을 저하시킵니다.

```
Pool 크기 결정 시 고려 사항:

1. HikariCP 권장 공식:
   connections = (core_count * 2) + effective_spindle_count

2. AWS RDS db.t3.micro 기준:
   max_connections ≈ 60~80

3. PetStar 환경:
   - t3.small (2 vCPU)
   - 단일 애플리케이션 서버
   → 적정 크기: 20~30
```

```yaml
# application-dev.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 30    # 10 → 30
      minimum-idle: 10
      connection-timeout: 30000
```

**결과**: Connection 획득 시간 1.79s → 0.19s (**89% 개선**)

**의미**: DB 쿼리 실행 전 대기 시간이 거의 사라졌습니다. 이제 실제 쿼리 성능을 측정할 수 있습니다.

### 3-2. DB LIMIT 적용 (Pageable)

**문제**: 왜 100,000건을 다 로드하고 있었는가?

코드를 분석해보니 **잘못된 위치에서 limit을 적용**하고 있었습니다.

```java
// Before: 의도는 10개만 보여주기
List<Entry> entries = repository.findAllOrderByVoteCountDesc(challengeId);
return entries.stream()
    .limit(10)  // ← Java Stream에서 10개 자르기
    .toList();

// 실제 동작:
// 1. DB에서 100,000건 전체 SELECT
// 2. 네트워크로 100,000건 전송
// 3. JVM 메모리에 100,000개 객체 생성
// 4. 그 중 10개만 사용, 나머지 99,990개는 GC 대상
```

**Java의 Stream.limit()과 DB의 LIMIT은 다릅니다**:

| 위치 | 동작 | 네트워크 비용 | 메모리 비용 |
|------|------|-------------|------------|
| Java Stream.limit() | 이미 로드된 데이터에서 자름 | 전체 데이터 전송 | 전체 객체 생성 |
| DB LIMIT | 쿼리 결과 자체를 제한 | 필요한 만큼만 | 필요한 만큼만 |

**해결**: Spring Data JPA의 Pageable 사용

```java
// After: DB에서 LIMIT 적용
List<Entry> entries = repository.findAllOrderByVoteCountDesc(
    challengeId,
    PageRequest.of(0, 10)  // → SQL: ... LIMIT 10
);

// 실행되는 SQL
SELECT * FROM entry
WHERE challenge_id = ?
ORDER BY vote_count DESC
LIMIT 10;  -- DB에서 10건만 반환
```

**결과**: p95 4.76s → 884ms (**81% 개선**)

**교훈**: "DB가 잘하는 건 DB에게 맡겨라". 정렬, 필터링, 페이징은 DB에서 처리해야 합니다.

### 3-3. N+1 해결 (Fetch Join)

**문제**: 왜 21개의 쿼리가 발생하는가?

JPA의 **Lazy Loading** 메커니즘을 이해해야 합니다.

```java
@Entity
public class Entry {
    @ManyToOne(fetch = FetchType.LAZY)  // ← Lazy Loading
    private Pet pet;

    @OneToOne(fetch = FetchType.LAZY)   // ← Lazy Loading
    private Image image;
}
```

Lazy Loading은 "연관 엔티티를 실제로 사용할 때까지 로딩을 지연"합니다. 이는 대부분 좋은 전략이지만, **목록 조회 시 문제**가 됩니다.

```java
// 랭킹 목록을 DTO로 변환하는 과정
List<Entry> entries = repository.findTop10();  // 쿼리 1회

for (Entry entry : entries) {
    dto.setPetName(entry.getPet().getName());   // Pet 로딩 → 쿼리 N회
    dto.setImageUrl(entry.getImage().getUrl()); // Image 로딩 → 쿼리 N회
}

// 총 1 + N + N = 21개 쿼리 (N=10인 경우)
```

**왜 "N+1 문제"라고 부르는가?**
- 1: 메인 쿼리 (Entry 목록)
- N: 각 Entry마다 연관 엔티티 조회 (Pet, Image)

**해결 방법 비교**:

| 방법 | 장점 | 단점 |
|------|------|------|
| EAGER Loading | 자동 로드 | 항상 로드 (불필요한 경우에도) |
| @EntityGraph | 선언적 | 복잡한 쿼리에서 제한 |
| **Fetch Join** | 명시적, 유연함 | JPQL 작성 필요 |
| Batch Size | 설정만으로 개선 | 최적이 아닐 수 있음 |

**PetStar에서 Fetch Join을 선택한 이유**:
- 랭킹 조회 시 **항상** Pet, Image가 필요 (선택적이 아님)
- 어떤 데이터를 로드하는지 **명확하게 제어** 가능
- **한 번의 JOIN으로** 모든 데이터 획득

```java
// After: Fetch Join으로 한 번에 로드
@Query("SELECT e FROM Entry e " +
       "JOIN FETCH e.pet " +
       "JOIN FETCH e.image " +
       "WHERE e.challenge.id = :id " +
       "ORDER BY e.voteCount DESC")
List<Entry> findTopEntriesWithFetchJoin(Long id, Pageable pageable);
```

**실행 쿼리 비교**:
```sql
-- Before: 21개 쿼리
SELECT * FROM entry WHERE challenge_id = ? ORDER BY vote_count DESC LIMIT 10;
SELECT * FROM pet WHERE pet_id = ?;  -- x10
SELECT * FROM image WHERE image_id = ?;  -- x10

-- After: 1개 쿼리
SELECT e.*, p.*, i.*
FROM entry e
JOIN pet p ON e.pet_id = p.pet_id
JOIN image i ON e.image_id = i.image_id
WHERE e.challenge_id = ?
ORDER BY e.vote_count DESC
LIMIT 10;
```

**결과**: 쿼리 21개 → 1개 (**95% 감소**), p95 884ms → 234ms (**73% 개선**)

**네트워크 비용 감소**: DB 왕복이 21회 → 1회로 줄어 네트워크 지연도 대폭 감소

### 3-4. 인덱스 최적화

**문제**: 왜 "Using filesort"가 발생하는가?

EXPLAIN 결과를 다시 분석해봤습니다.

```sql
EXPLAIN SELECT * FROM entry
WHERE challenge_id = 1
ORDER BY vote_count DESC
LIMIT 10;

-- 결과
| type | key         | rows | Extra          |
|------|-------------|------|----------------|
| ref  | uk_entry... | 3334 | Using filesort |
```

**"Using filesort"의 의미**:
- 인덱스만으로 정렬을 해결하지 못함
- **임시 테이블을 만들어서 정렬** 수행
- 데이터가 많을수록 비용 증가

**기존 인덱스 분석**:

```sql
-- 기존 인덱스 구조
UNIQUE INDEX uk_entry_challenge_pet (challenge_id, pet_id)

-- 이 인덱스가 ORDER BY vote_count에 도움이 안 되는 이유:
-- 인덱스는 (challenge_id, pet_id) 순으로 정렬되어 있음
-- vote_count로 정렬하려면 별도 작업 필요
```

**인덱스 설계 원칙**:

```
복합 인덱스 설계 시 컬럼 순서:
1. WHERE 조건에 사용되는 컬럼 (등치 조건)
2. ORDER BY에 사용되는 컬럼
3. SELECT에 사용되는 컬럼 (Covering Index)

우리 쿼리:
WHERE challenge_id = ?     → challenge_id 먼저
ORDER BY vote_count DESC   → vote_count 다음
```

**해결**: 쿼리 패턴에 최적화된 복합 인덱스 생성

```sql
CREATE INDEX idx_entry_challenge_vote
ON entry(challenge_id, vote_count DESC);

-- 이 인덱스로 가능한 것:
-- 1. challenge_id로 필터링 (인덱스 탐색)
-- 2. vote_count DESC로 이미 정렬됨 (filesort 불필요)
-- 3. LIMIT 10 → 상위 10개만 읽고 종료
```

**EXPLAIN 결과 비교**:
```
Before: type=ref, Extra=Using filesort  ← 정렬 필요
After:  type=ref, Extra=NULL            ← 인덱스로 정렬 완료
```

**결과**: p95 234ms → 146ms (**38% 개선**)

**추가 개선 가능성 (Covering Index)**:

```sql
-- 만약 SELECT 컬럼도 포함한다면?
CREATE INDEX idx_entry_covering
ON entry(challenge_id, vote_count DESC, pet_id, image_id);

-- 장점: 테이블 접근 없이 인덱스만으로 조회 가능
-- 단점: 인덱스 크기 증가, INSERT/UPDATE 성능 저하
-- → PetStar에서는 Fetch Join을 사용하므로 불필요
```

## 4. 최적화 순서의 중요성

### 왜 순서가 중요한가?

```
┌─────────────────────────────────────────────────────────────┐
│  최적화 순서에 따른 효과 차이                               │
│                                                             │
│  ❌ 잘못된 순서: 인덱스 먼저                                │
│     → 100K row 로드 문제 해결 안 됨                        │
│     → 인덱스 효과 측정 불가                                │
│                                                             │
│  ✅ 올바른 순서:                                           │
│     1. Pageable (데이터량 감소)                            │
│     2. Fetch Join (쿼리 수 감소)                           │
│     3. Index (쿼리 속도 향상)                              │
│                                                             │
│  각 단계의 효과를 정확히 측정 가능!                        │
└─────────────────────────────────────────────────────────────┘
```

### 실제 측정 결과

| 단계 | p95 | 개선율 | 누적 |
|------|-----|--------|------|
| Before | 4.76s | - | - |
| +Pageable | 884ms | 81% | 81% |
| +Fetch Join | 234ms | 73% | 95% |
| +Index | 146ms | 38% | **97%** |

## 5. 핵심 교훈

### 1. 측정 없이 최적화 없다

- Grafana로 메트릭 수집
- p6spy로 쿼리 로깅
- EXPLAIN으로 실행 계획 분석
- k6로 부하테스트

### 2. DB가 잘하는 건 DB에게

```java
// ❌ Java에서 처리
entries.stream().limit(10)...
entries.stream().sorted()...

// ✅ DB에서 처리
SELECT ... LIMIT 10
SELECT ... ORDER BY ... (with index)
```

### 3. 한 번에 하나씩, 효과 측정

- 여러 최적화를 동시에 적용하면 각각의 효과를 알 수 없음
