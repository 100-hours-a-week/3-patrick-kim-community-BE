# Phase 10-2: 응답시간 100ms 달성을 위한 아키텍처 개선

> **목표**: p95 응답시간 516ms → 100ms 이하
> **핵심 전략**: 동기 → 비동기 전환, DB 의존성 제거

---

## 1. 문제 발견: 왜 아직 500ms인가?

### 1.1 Phase 10 결과 분석

Phase 10에서 SQS 비동기 처리를 도입하여 응답시간을 65% 개선했지만, 목표인 100ms에는 미달:

```
현재 상태:
- p95: 516ms (목표: 100ms)
- 평균: 361ms

→ 아직 5배 느림. 왜?
```

### 1.2 병목 지점 분석

voteAsync() 메서드의 각 단계별 소요 시간을 측정:

```
┌─────────────────────────────────────────────────────────────┐
│ API 요청 흐름 분석                                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [1] entryRepository.findById()                             │
│      └─ DB SELECT 쿼리                                      │
│      └─ 예상: ~30-50ms                                      │
│      └─ 필요성: Entry 존재 확인, challengeId 획득            │
│                                                             │
│  [2] voteRepository.existsByEntryIdAndMemberId()  ← 병목 1  │
│      └─ DB SELECT 쿼리 (JOIN 포함)                          │
│      └─ 예상: ~50-80ms                                      │
│      └─ 문제: 매 요청마다 DB 조회                            │
│                                                             │
│  [3] rankingRedisService.incrementVote()                    │
│      └─ Redis ZINCRBY                                       │
│      └─ 예상: ~5-10ms                                       │
│      └─ 상태: 최적화됨 ✅                                    │
│                                                             │
│  [4] sqsTemplate.send()                         ← 주요 병목  │
│      └─ SQS SendMessage API 호출                            │
│      └─ 예상: ~200-400ms (네트워크 RTT 포함)                 │
│      └─ 문제: 동기 호출, 응답 대기                           │
│                                                             │
│  [5] rankingRedisService.getScore()                         │
│      └─ Redis ZSCORE                                        │
│      └─ 예상: ~5-10ms                                       │
│      └─ 상태: 최적화됨 ✅                                    │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│ 총합: 30 + 50 + 10 + 300 + 10 = ~400-550ms                  │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 핵심 통찰

```
질문: "비동기"라고 했는데 왜 SQS 전송이 동기인가?

분석:
1. SqsTemplate.send()는 기본적으로 동기 메서드
2. SQS에 메시지를 보내고 ACK를 받을 때까지 블로킹
3. 네트워크 RTT (서울 → SQS 엔드포인트) ~100-200ms
4. SQS 내부 처리 시간 ~50-100ms
5. 총 ~200-400ms 소요

→ "비동기 DB 저장"이지만 "동기 큐 전송"이었음!
```

## 2. 해결 전략: 진짜 비동기로 만들기

### 2.1 아키텍처 개선 방향

```
Before: 가짜 비동기 (SQS 전송은 동기)
────────────────────────────────────────────────────
Client → API → [DB체크] → [Redis] → [SQS 대기] → Response
                 50ms      10ms       300ms
                                        ↓
                                    Consumer → DB

총 응답시간: ~500ms


After: 진짜 비동기 (모든 외부 호출 비동기화)
────────────────────────────────────────────────────
Client → API → [Redis 체크] → [Redis 증가] → Response
                  5ms            5ms
                   ↓              ↓
              (Fire & Forget)  SQS → Consumer → DB

목표 응답시간: ~50ms
```

### 2.2 세 가지 최적화 포인트

| # | 현재 | 개선 | 예상 효과 |
|:--|:-----|:-----|:---------|
| 1 | DB 중복 체크 | Redis SET으로 체크 | -50ms |
| 2 | SQS 동기 전송 | 비동기 Fire & Forget | -300ms |
| 3 | Entry DB 조회 | Redis 캐시 또는 제거 | -30ms |

## 3. 최적화 1: 중복 투표 체크를 Redis로 이전

### 3.1 현재 문제

```java
// 현재 코드: DB 조회
if (voteRepository.existsByEntryIdAndMemberId(entryId, memberId)) {
    throw new GeneralException(ErrorStatus._BAD_REQUEST);
}
```

```sql
-- 실행되는 쿼리 (p6spy 로그)
SELECT v.vote_id
FROM vote v
LEFT JOIN entry e ON e.entry_id = v.entry_id
LEFT JOIN member m ON m.member_id = v.member_id
WHERE e.entry_id = 100 AND m.member_id = 128
LIMIT 1

→ 매 요청마다 DB 왕복
→ JOIN 연산 포함
→ Hot Spot에서 병목
```

### 3.2 해결: Redis SET 자료구조 활용

```
Redis 키 설계:
votes:{entryId} → SET of memberIds

예시:
votes:100 → {128, 129, 130, 131, ...}

연산:
- SADD votes:100 128  → 투표 추가 (O(1))
- SISMEMBER votes:100 128 → 중복 체크 (O(1))
- SREM votes:100 128  → 투표 취소 (O(1))
```

### 3.3 트레이드오프

```
장점:
- O(1) 조회 (DB는 인덱스 타도 O(log n))
- 네트워크 RTT 감소 (Redis는 EC2와 같은 VPC)
- DB 부하 감소

단점:
- 메모리 사용 증가
  → 100만 투표 × 8 bytes = 8MB (무시 가능)
- Redis 장애 시 데이터 손실 가능
  → DB가 Source of Truth, Redis는 캐시
- Eventual Consistency 복잡도 증가
  → 정합성 검증 배치로 해결
```

### 3.4 구현 계획

```java
// RankingRedisService에 추가
private static final String VOTES_KEY_PREFIX = "votes:";

public boolean hasVoted(Long entryId, Integer memberId) {
    String key = VOTES_KEY_PREFIX + entryId;
    return Boolean.TRUE.equals(
        redisTemplate.opsForSet().isMember(key, memberId.toString())
    );
}

public void recordVote(Long entryId, Integer memberId) {
    String key = VOTES_KEY_PREFIX + entryId;
    redisTemplate.opsForSet().add(key, memberId.toString());
}

public void removeVote(Long entryId, Integer memberId) {
    String key = VOTES_KEY_PREFIX + entryId;
    redisTemplate.opsForSet().remove(key, memberId.toString());
}
```

## 4. 최적화 2: SQS 비동기 전송 (Fire & Forget)

### 4.1 현재 문제

```java
// 현재 코드: 동기 전송
public void sendVote(VoteMessage message) {
    String payload = objectMapper.writeValueAsString(message);
    sqsTemplate.send(queueName, payload);  // 블로킹! 300ms 대기
}
```

### 4.2 해결: CompletableFuture 비동기 전송

```java
// 개선 코드: 비동기 전송
public void sendVoteAsync(VoteMessage message) {
    String payload = objectMapper.writeValueAsString(message);

    CompletableFuture.runAsync(() -> {
        try {
            sqsTemplate.send(queueName, payload);
            log.debug("[VoteProducer] Message sent async: {}", message.getVoteId());
        } catch (Exception e) {
            // 실패 시 로깅만 (이미 Redis에 반영됨)
            log.error("[VoteProducer] Async send failed: {}", e.getMessage());
            // TODO: 실패 메시지 재시도 큐 또는 DLQ 처리
        }
    });
}
```

### 4.3 Fire & Forget 패턴의 위험과 대응

```
질문: SQS 전송 실패하면 데이터 손실 아닌가?

분석:
1. Redis에는 이미 투표가 반영됨 (랭킹 OK)
2. SQS 전송 실패 → DB에만 저장 안 됨
3. DB는 영속 저장소 (Source of Truth)

위험:
- Redis ↔ DB 정합성 불일치

대응책:
1. 실패 시 로컬 큐에 저장 → 재시도
2. 정합성 검증 배치 (Phase 10-3)
3. DLQ 모니터링으로 실패 감지
```

### 4.4 구현 시 고려사항

## 5. 최적화 3: Entry 조회 최적화

### 5.1 현재 문제

```java
// 매 요청마다 Entry 조회
Entry entry = entryRepository.findById(entryId)
        .orElseThrow(() -> new GeneralException(ErrorStatus._NOTFOUND));

// 실제로 필요한 정보:
// 1. Entry 존재 여부
// 2. challengeId
// 3. entry.member.id (자기 투표 방지)
```

### 5.2 해결 방안

**옵션 A: Redis 캐시**
```
entry:meta:{entryId} → {challengeId, memberId}

장점: 조회 최적화
단점: 캐시 무효화 복잡
```

**옵션 B: challengeId를 요청에 포함**
```
POST /api/v1/challenges/{challengeId}/entries/{entryId}/votes

장점: DB 조회 불필요
단점: API 변경 필요
```

**옵션 C: Lazy 검증 (Consumer에서)**
```
API: Redis만 업데이트
Consumer: Entry 존재/유효성 검증 후 DB 저장

장점: API 최경량화
단점: 잘못된 투표도 일단 Redis에 반영
```

### 5.3 선택: 옵션 C (Lazy 검증)

```
이유:
1. 대부분의 요청은 유효함 (99%+)
2. 잘못된 투표는 Consumer에서 거부
3. API 응답은 최대한 빠르게

처리 흐름:
1. API: Redis 업데이트 (검증 없이)
2. SQS: 메시지 전송
3. Consumer: Entry 검증 → 실패 시 Redis 롤백

추가 구현:
- Consumer에서 실패 시 Redis 롤백 로직
- 롤백 실패 시 정합성 배치로 복구
```

## 6. 정합성 검증 배치 (Phase 10-3)

### 6.1 필요성

```
비동기 처리의 트레이드오프:
- 빠른 응답 ↔ Eventual Consistency
- Redis 선반영 ↔ DB 지연 저장

정합성 불일치 발생 시나리오:
1. SQS 전송 실패
2. Consumer 처리 실패
3. Redis 장애/재시작
4. 버그로 인한 중복/누락
```

### 6.2 검증 배치 설계

```java
@Scheduled(fixedRate = 300000)  // 5분마다
public void verifyVoteConsistency() {
    // 1. 활성 챌린지 조회
    List<Challenge> activeChallenges = challengeRepository.findByStatus(ACTIVE);

    for (Challenge challenge : activeChallenges) {
        // 2. Redis vs DB 비교
        List<Entry> entries = entryRepository.findByChallenge(challenge);

        for (Entry entry : entries) {
            int redisCount = rankingRedisService.getVoteCount(challenge.getId(), entry.getId());
            int dbCount = entry.getVoteCount();

            if (redisCount != dbCount) {
                log.warn("[Consistency] Mismatch: entry={}, redis={}, db={}",
                    entry.getId(), redisCount, dbCount);

                // 3. 불일치 처리 전략
                handleMismatch(entry, redisCount, dbCount);
            }
        }
    }
}
```

### 6.3 불일치 처리 전략

```
Source of Truth: DB (Vote 테이블)

불일치 발생 시:
1. Redis > DB: 아직 Consumer가 처리 안 함 (정상)
   → 대기 (30초 후 재검증)

2. Redis < DB: Redis 누락 (비정상)
   → DB 기준으로 Redis 동기화
   → 알람 발송 (잠재적 버그)

3. Redis = 0 but DB > 0: Redis 키 손실
   → DB에서 Redis 재구축
```

#### 7.2.2 Redis 호출 횟수 증가

```
Phase 10 (원본):
─────────────────────────────────────────────
요청당 Redis 호출: 2회
  1. ZINCRBY (투표 수 증가)
  2. ZSCORE (현재 점수 조회)

50 동시 요청 → 100 Redis 호출 (단일 커넥션)


Phase 10-2 (최적화):
─────────────────────────────────────────────
요청당 Redis 호출: 4회
  1. SISMEMBER (중복 체크)     ← 추가
  2. SADD (투표 기록)          ← 추가
  3. ZINCRBY (투표 수 증가)
  4. ZSCORE (현재 점수 조회)

50 동시 요청 → 200 Redis 호출 (단일 커넥션)

→ Redis 호출 2배 증가 + 단일 커넥션 병목 = 성능 악화
```

#### 7.2.3 네트워크 RTT 누적

```
App Server (43.200.83.214) ←──── 네트워크 ────→ Redis (43.201.243.102)
                                  RTT ~2-5ms

Phase 10:
  DB 쿼리 (로컬 같은 VPC) + Redis 2회 = ~50ms + 10ms = 60ms

Phase 10-2:
  Redis 4회 × RTT = 4 × 5ms = 20ms (단일 요청)
  BUT 50 동시 요청 → 직렬화로 인해 200 × 5ms = 1000ms+
```

---

### 7.3 왜 이런 설계를 했는가? (의도와 현실)

```
의도:
──────────────────────────────────────────────────
"DB 쿼리를 Redis로 대체하면 더 빠를 것이다"
  - Redis: 인메모리, O(1) 조회
  - DB: 디스크 I/O, 인덱스 탐색

가정:
  - Redis가 무조건 빠르다
  - 커넥션 모델을 고려하지 않음


현실:
──────────────────────────────────────────────────
"개별 연산 속도 < 커넥션 모델 영향"

  개별 속도:     Redis SISMEMBER (0.1ms) < DB SELECT (5ms)
  동시성 처리:   단일 커넥션 (직렬) vs 풀 30개 (병렬)

→ 개별 연산이 빨라도, 병렬 처리 능력이 더 중요!
```

## 8. 최적화 방안 (3가지)

### 8.1 방안 1: Lettuce 커넥션 풀 설정

#### 개념

```
현재: Lettuce 단일 커넥션 (멀티플렉싱)
─────────────────────────────────────────────────────
┌─────────────────────────────────────────────────┐
│                   Application                    │
├─────────────────────────────────────────────────┤
│  Thread 1 ──┐                                   │
│  Thread 2 ──┼──→ [Single Connection] ──→ Redis  │
│  Thread 3 ──┤         (직렬화)                  │
│  ...        │                                   │
│  Thread 50 ─┘                                   │
└─────────────────────────────────────────────────┘


개선: Lettuce 커넥션 풀링
─────────────────────────────────────────────────────
┌─────────────────────────────────────────────────┐
│                   Application                    │
├─────────────────────────────────────────────────┤
│  Thread 1 ──→ [Conn 1] ──┐                      │
│  Thread 2 ──→ [Conn 2] ──┼──→ Redis             │
│  Thread 3 ──→ [Conn 3] ──┤    (병렬 처리)       │
│  ...                     │                      │
│  Thread 50 → [Conn 10] ──┘                      │
└─────────────────────────────────────────────────┘
```

#### 구현 코드

```java
// RedisConfig.java
@Configuration
public class RedisConfig {

    @Bean
    public LettuceConnectionFactory redisConnectionFactory() {
        RedisStandaloneConfiguration config = new RedisStandaloneConfiguration();
        config.setHostName("43.201.243.102");
        config.setPort(6379);

        // 커넥션 풀 설정
        GenericObjectPoolConfig<Object> poolConfig = new GenericObjectPoolConfig<>();
        poolConfig.setMaxTotal(20);          // 최대 커넥션 수
        poolConfig.setMaxIdle(10);           // 최대 유휴 커넥션
        poolConfig.setMinIdle(5);            // 최소 유휴 커넥션
        poolConfig.setMaxWait(Duration.ofMillis(1000));

        LettucePoolingClientConfiguration clientConfig =
            LettucePoolingClientConfiguration.builder()
                .poolConfig(poolConfig)
                .build();

        return new LettuceConnectionFactory(config, clientConfig);
    }
}
```

#### 예상 효과


### 8.2 방안 2: Redis Pipeline 적용

#### 개념

```
현재: 개별 명령 전송 (Round Trip 4회)
─────────────────────────────────────────────────────
App                             Redis
 │                                │
 │ ─── SISMEMBER ───────────────→ │
 │ ←──────────────── response ─── │  RTT 1
 │                                │
 │ ─── SADD ────────────────────→ │
 │ ←──────────────── response ─── │  RTT 2
 │                                │
 │ ─── ZINCRBY ─────────────────→ │
 │ ←──────────────── response ─── │  RTT 3
 │                                │
 │ ─── ZSCORE ──────────────────→ │
 │ ←──────────────── response ─── │  RTT 4
 │                                │

총 RTT: 4 × 5ms = 20ms


개선: Pipeline (Round Trip 1회)
─────────────────────────────────────────────────────
App                             Redis
 │                                │
 │ ─── SISMEMBER ───┐             │
 │ ─── SADD ────────┼───────────→ │
 │ ─── ZINCRBY ─────┤             │
 │ ─── ZSCORE ──────┘             │
 │                                │
 │ ←─── responses (batch) ─────── │  RTT 1
 │                                │

총 RTT: 1 × 5ms = 5ms (75% 감소)
```

#### 구현 코드

```java
// RankingRedisService.java
public VoteResult processVotePipelined(Long entryId, Integer memberId, Long challengeId) {
    String votesKey = VOTES_KEY_PREFIX + entryId;
    String rankingKey = RANKING_KEY_PREFIX + challengeId;

    List<Object> results = redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
        // 1. 중복 체크
        connection.setCommands().sIsMember(
            votesKey.getBytes(),
            memberId.toString().getBytes()
        );
        // 2. 투표 기록
        connection.setCommands().sAdd(
            votesKey.getBytes(),
            memberId.toString().getBytes()
        );
        // 3. 랭킹 증가
        connection.zSetCommands().zIncrBy(
            rankingKey.getBytes(),
            1,
            entryId.toString().getBytes()
        );
        // 4. 현재 점수
        connection.zSetCommands().zScore(
            rankingKey.getBytes(),
            entryId.toString().getBytes()
        );
        return null;
    });

    Boolean alreadyVoted = (Boolean) results.get(0);
    Double currentScore = (Double) results.get(3);

    return new VoteResult(alreadyVoted, currentScore.intValue());
}
```

#### 예상 효과

```
Before: 4 RTT × 50 요청 = 200 RTT → 1.3s
After:  1 RTT × 50 요청 = 50 RTT → ~400ms (예상)
```

### 8.3 방안 3: 하이브리드 전략 (DB + Redis 역할 분리)

#### 개념

```

─────────────────────────────────────────────────────
Redis에 모든 부하 집중
  - 중복 체크 (SISMEMBER)
  - 투표 기록 (SADD)
  - 랭킹 (ZINCRBY, ZSCORE)

→ 단일 커넥션 병목


개선: 역할 분리
─────────────────────────────────────────────────────
┌─────────────────────────────────────────────────────┐
│                    Request Flow                      │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 중복 체크: DB (HikariCP 30 커넥션)              │
│     └─ 병렬 처리 가능                               │
│     └─ 인덱스 최적화로 ~5ms                         │
│                                                     │
│  2. 랭킹 업데이트: Redis (2 호출만)                  │
│     └─ ZINCRBY + ZSCORE                             │
│     └─ 기존과 동일                                  │
│                                                     │
│  3. DB 저장: SQS → Consumer                         │
│     └─ 비동기 처리                                  │
│                                                     │
└─────────────────────────────────────────────────────┘

장점:
- DB 커넥션 풀의 병렬 처리 능력 활용
- Redis 부하 50% 감소 (4회 → 2회)
- 각 컴포넌트의 강점 활용
```

```java
// VoteService.java - 하이브리드 전략
public VoteResponseDto.VoteResult voteHybrid(Long entryId, Integer memberId) {
    // 1. DB 중복 체크 (HikariCP 병렬 처리)
    if (voteRepository.existsByEntryIdAndMemberId(entryId, memberId)) {
        throw new GeneralException(ErrorStatus._BAD_REQUEST);
    }

    // 2. Entry 조회 (DB)
    Entry entry = entryRepository.findById(entryId)
            .orElseThrow(() -> new GeneralException(ErrorStatus._NOTFOUND));

    validateVote(entry, memberId);

    Long challengeId = entry.getChallenge().getId();

    // 3. Redis 랭킹만 업데이트 (2회 호출)
    rankingRedisService.incrementVote(challengeId, entryId);

    // 4. SQS 비동기 전송
    voteProducer.sendVote(VoteMessage.create(memberId, entryId, challengeId));

    // 5. 현재 점수 반환
    Double score = rankingRedisService.getScore(challengeId, entryId);

    return VoteResponseDto.VoteResult.builder()
            .entryId(entryId)
            .voteCount(score.intValue())
            .build();
}
```
