# 외부 서비스 장애 대응 전략

> PetStar 투표 시스템은 Redis, SQS 두 개의 외부 서비스에 의존한다.
> 각 서비스 장애 시 투표가 중단되지 않도록 Graceful Degradation 전략을 구현했다.

---

## 1. Fallback 체인 개요

```
정상 상태:
  Client → API → Redis(즉시) → SQS(비동기) → Response
                                  ↓
                              Consumer → DB

SQS 장애:
  Client → API → Redis(즉시) → [SQS 실패] → sync DB write → Response
  → Redis 랭킹 정상, DB 직접 저장으로 데이터 유실 없음

Redis 장애:
  Client → API → [Redis 실패] → Circuit Breaker → Pessimistic Lock → Response
  → DB Lock으로 정합성 보장, Redis는 복구 후 정합성 스케줄러가 동기화

둘 다 장애:
  Client → API → [Redis 실패] → Circuit Breaker → Pessimistic Lock → Response
  → votePessimistic()에서 Redis best-effort (실패해도 DB 저장 유지)
  → 정합성 스케줄러가 복구 후 Redis 동기화
```

## 2. 시나리오별 상세 분석

### 시나리오 1: SQS 네트워크 단절

```
상황: SQS 엔드포인트 불통 (네트워크 장애, AWS 장애 등)
영향 범위: voteAsync()에서 SQS 전송 실패
기존 문제: Fire & Forget 패턴으로 실패를 감지할 수 없었음 (dead code 버그)

해결 (Phase 11):
1. VoteProducer를 동기 전송으로 전환
2. 전송 실패 시 예외가 VoteService까지 전파
3. catch 블록에서 syncWriteToDb() 호출 — DB에 직접 저장
4. Redis에는 이미 반영되어 있으므로 롤백하지 않음

결과:
- 사용자 경험: 정상 (투표 성공 응답)
- 데이터 정합성: 보장 (Redis + DB 모두 반영)
- 응답 시간: ~800ms (DB 직접 저장으로 약간 증가)
```

### 시나리오 2: Redis 장애 (단독)

```
상황: Redis 서버 다운 또는 네트워크 불통
영향 범위: voteAsync()에서 Redis Pipeline 실패

해결 (기존 Circuit Breaker):
1. Redis 호출 실패 → Resilience4j Circuit Breaker 감지
2. 실패율 50% 초과 시 OPEN 상태 전환
3. voteFallback() → votePessimistic() 자동 전환
4. Pessimistic Lock으로 DB에 직접 저장
5. Redis 업데이트는 best-effort (실패해도 DB 저장 유지)
6. 30초 후 HALF_OPEN → Redis 복구 확인 → CLOSED 복귀

결과:
- 사용자 경험: 느려짐 (p95 1.73s) but 서비스 유지
- 데이터 정합성: DB 기준 보장
- 복구 후: 정합성 스케줄러(5분 주기)가 Redis를 DB 기준으로 동기화
```

### 시나리오 3: SQS + Redis 동시 장애

```
상황: 두 외부 서비스 모두 불통
영향 범위: 전체 비동기 경로 사용 불가

동작:
1. Redis 실패 → Circuit Breaker OPEN → votePessimistic() 전환
2. votePessimistic()에서 DB Lock으로 투표 저장
3. Redis 업데이트 시도 → 실패 → try-catch로 무시 (best-effort)
4. DB 저장은 성공 (Redis 무관)

결과:
- 사용자 경험: 느리지만 동작 (p95 1.73s)
- 데이터 정합성: DB 정확, Redis는 stale
- 복구 후: 정합성 스케줄러가 Redis를 DB 기준으로 전체 동기화
```

### 시나리오 4: Consumer 크래시

```
상황: SQS Consumer가 메시지 처리 중 다운
영향 범위: 처리 중이던 메시지 1건

SQS 재전달 메커니즘:
1. Consumer가 메시지를 수신하면 visibility timeout(30초) 시작
2. Consumer 크래시 → ACK 없음 → 30초 후 메시지 다시 visible
3. SQS가 다시 전달 → Consumer 재시작 후 처리
4. maxReceiveCount(3회) 초과 시 DLQ로 이동

결과:
- 최대 30초 지연 후 자동 재처리
- 재처리 시 idempotency 보장 (DB UK 제약조건)
- 3회 실패 시 DLQ에 보관 → DLQ 재처리 스케줄러가 처리
```

### 시나리오 5: 메시지 유실 (SQS 전송 성공, Consumer 미처리)

```
상황: SQS에 메시지 전송 성공했지만, 어떤 이유로 Consumer가 처리하지 못함

가능한 원인:
1. Consumer 장기간 다운 (message retention 초과)
2. DLQ에서도 retention 초과
3. 역직렬화 실패로 메시지 무시

방어선:
1. SQS message retention: 4일 (기본값)
2. DLQ message retention: 14일
3. 정합성 스케줄러: 5분마다 Redis vs DB 비교

스케줄러 동작:
- Redis score > DB voteCount → SQS 처리 지연으로 판단 → 30초 대기 후 재검증
- 재검증 후에도 불일치 → DB 기준으로 Redis 동기화 (ZADD)
- Redis score < DB voteCount → Redis 키 손실 → 즉시 DB 기준 동기화

한계:
- 스케줄러는 voteCount 수치만 비교, 개별 Vote 레코드는 검증하지 않음
- Redis에 반영됐지만 DB에 없는 개별 투표는 스케줄러가 복구 불가
- 이 경우 DB의 Vote 테이블 COUNT(*)와 entry.voteCount가 불일치할 수 있음

수용 이유:
- 사용자가 보는 랭킹은 Redis 기준 (정상)
- DB는 영속 저장소이므로 최종적으로 스케줄러가 정합
- 개별 투표 레코드 유실은 챌린지 결과에 영향 없음 (voteCount 기준)
```

### 시나리오 6: SQS 전송 성공, DB 저장 중 데드락

```
상황: Consumer에서 같은 Entry에 동시 UPDATE 시 데드락

현재 대응:
1. maxConcurrentMessages=1 (순차 처리)로 데드락 가능성 최소화
2. 데드락 발생 시 CannotAcquireLockException → 예외 throw
3. SQS가 visibility timeout 후 재전달
4. 3회 실패 시 DLQ로 이동

결과:
- 대부분 순차 처리로 데드락 없음
- 발생해도 SQS 재시도로 자동 복구
```

## 3. 방어선 요약

```
Layer 1: Redis (즉시 반영)
  → 장애 시 Circuit Breaker → DB fallback

Layer 2: SQS (비동기 전달)
  → 장애 시 syncWriteToDb() → DB 직접 저장

Layer 3: Consumer (DB 저장)
  → 크래시 시 SQS 재전달 (visibility timeout)
  → 반복 실패 시 DLQ 보관

Layer 4: DLQ 재처리 (30분 주기)
  → 실패 메시지 재처리 시도

Layer 5: 정합성 스케줄러 (5분 주기)
  → Redis ↔ DB 수치 비교
  → 불일치 시 DB 기준으로 Redis 동기화
  → 최종 안전망 (Source of Truth = DB)
```

## 4. 장애 시나리오 부하 테스트 결과

### 4.1 테스트 환경

```
인프라:
  - App Server: EC2 t3.small (2 vCPU, 2GB RAM)
  - DB: RDS db.t3.micro (MySQL 8.0)
  - Redis: EC2 t3.small에서 Docker 실행
  - SQS: AWS SQS Standard Queue (ap-northeast-2)

테스트 도구: k6
테스트 조건: 50 VUs, 30초, 투표 API
Redis connect-timeout: 2000ms (빠른 장애 감지를 위해 튜닝)
```

### 4.2 시나리오별 결과

| 시나리오 | 총 요청 | 성공률 | avg | p95 | Fallback 경로 |
|:---|:---|:---|:---|:---|:---|
| **정상** | 2,738 | 94.7% | 52ms | 138ms | Redis + SQS async |
| **SQS 장애** | 2,415 | 88.8% | 126ms | 337ms | Pessimistic Lock |
| **Redis 장애** | 1,850 | 82.8% | 320ms | 1,022ms | Circuit Breaker → Pessimistic Lock |
| **둘 다 장애** | 1,970 | 79.7% | 269ms | 792ms | DB-only 모드 |

> 실패분(5~20%)은 **중복 투표 400 에러**이며 서비스 장애가 아님

### 4.3 시나리오별 시뮬레이션 방법

```
SQS 장애:
  → spring.cloud.aws.sqs.enabled=false 로 SQS 비활성화
  → VoteProducer 빈 미생성 → voteAsync()에서 votePessimistic() fallback
  → SQS 없이 DB Lock으로 직접 저장

Redis 장애:
  → 모니터링 서버에서 docker stop petstar-redis
  → Redis 연결 실패 → Circuit Breaker OPEN 전환
  → voteFallback() → votePessimistic() 자동 전환
  → Redis 업데이트는 try-catch로 best-effort (실패해도 DB 저장 유지)

둘 다 장애:
  → SQS 비활성화 + Redis 중지
  → voteAsync() → VoteProducer null → votePessimistic()
  → votePessimistic() → DB Lock 저장 + Redis best-effort (실패 무시)
```

### 4.4 핵심 발견

```
1. 어떤 외부 서비스가 죽어도 투표는 계속 성공한다
   → SQS 장애: avg 52ms → 126ms (2.4배 느려지지만 동작)
   → Redis 장애: avg 52ms → 320ms (6배 느려지지만 동작)
   → 둘 다 장애: avg 52ms → 269ms (5배 느려지지만 동작)

2. Redis connect-timeout 튜닝이 Circuit Breaker 전환 속도에 결정적
   → 기본값(60s): Circuit Breaker 전환까지 각 요청 50초+ 대기
   → 2초 설정: 빠르게 실패 감지 → Circuit Breaker OPEN → 즉시 fallback
   → 교훈: 장애 감지 속도 = connect-timeout 설정에 의존

3. Graceful Degradation 실증
   → "외부 서비스 죽으면 서비스도 죽는다"가 아닌
   → "외부 서비스 죽으면 느려지지만 계속 동작한다"를 수치로 증명
```

## 5. 핵심 설계 원칙

### 5.1 "DB가 Source of Truth"

```
모든 장애 복구의 기준은 DB
- Redis: 캐시 역할 (유실 가능, 재구축 가능)
- SQS: 전달 매개체 (유실 시 정합성 스케줄러가 보완)
- DB: 최종 저장소 (유실 불가)
```

### 5.2 "Graceful Degradation > Complete Failure"

```
외부 서비스 장애 시:
✅ 느려지더라도 서비스 유지 (Pessimistic Lock fallback)
✅ 데이터 유실보다 지연 허용 (Eventual Consistency)
❌ 외부 서비스 장애가 전체 서비스 중단으로 이어지면 안 됨
```

### 5.3 "Best-effort + 후속 보정"

```
실시간 처리가 실패해도:
1. best-effort로 최선을 다하고 (try-catch + 로깅)
2. 주기적 배치로 후속 보정 (정합성 스케줄러)
3. 수동 개입 최소화 (DLQ 자동 재처리)
```
