# 비동기 투표 시스템 아키텍처 (Phase 10 최종)

> **최종 성과**: p95 1,730ms → 658ms (62% 개선)
> **전략**: DB 중복체크 + Redis 실시간 랭킹 + SQS 비동기 DB 저장
> **정합성**: Eventual Consistency + 5분 주기 검증 배치

---

## 3. Redis 역할 상세

### 3.1 Redis가 하는 일

```
┌────────────────────────────────────────────────────────────────┐
│                    Redis 역할 (이 시스템에서)                    │
├────────────────────┬───────────────────────────────────────────┤
│  실시간 랭킹        │ Sorted Set (ZSET)                        │
│                    │ 키: ranking:challenge:{challengeId}       │
│                    │ 멤버: entryId                             │
│                    │ 점수: voteCount                           │
│                    │                                           │
│                    │ ZINCRBY → 투표 시 점수 +1 (O(log N))     │
│                    │ ZREVRANGE → 랭킹 조회 (O(log N + M))     │
│                    │ ZSCORE → 특정 Entry 점수 조회 (O(1))      │
├────────────────────┼───────────────────────────────────────────┤
│  Pipeline 최적화    │ ZINCRBY + ZSCORE를 1회 네트워크 왕복으로   │
│                    │ Redis RTT: 10ms → 5ms (50% 감소)         │
├────────────────────┼───────────────────────────────────────────┤
│  Connection Pool   │ Lettuce + commons-pool2                  │
│                    │ max-active: 20, min-idle: 5              │
│                    │ 단일 커넥션 직렬화 → 풀링으로 병렬 전송     │
└────────────────────┴───────────────────────────────────────────┘
```

```
│  │  MySQL (멀티 스레드)     │    │  Redis (싱글 스레드)          │           │

│  │  Request 1 → Thread 1   │    │  Request 1 ─┐               │           │
│  │  Request 2 → Thread 2   │    │  Request 2 ─┤               │           │
│  │  Request 3 → Thread 3   │    │  Request 3 ─┼→ 1개씩 처리   │           │
│  │  ...       → ...        │    │  ...        ─┤               │           │
```

### 3.3 Pipeline 동작 원리

```
Before: 개별 명령 (네트워크 왕복 2회)
─────────────────────────────────────
App                         Redis Server
 │                              │
 │── ZINCRBY ranking:10 100 1 ─→│  처리
 │←── OK (Double) ──────────────│  RTT ① (~5ms)
 │                              │
 │── ZSCORE ranking:10 100 ────→│  처리
 │←── 42.0 (Double) ────────────│  RTT ② (~5ms)
 │                              │
 합계: RTT 2회 = ~10ms


After: Pipeline (네트워크 왕복 1회)
─────────────────────────────────────
App                         Redis Server
 │                              │
 │── ZINCRBY ranking:10 100 1 ─┐│
 │── ZSCORE ranking:10 100 ────┤│  배치 처리
 │                              ││
 │←── [OK, 42.0] ──────────────┘│  RTT ① (~5ms)
 │                              │
 합계: RTT 1회 = ~5ms (50% 감소)


구현 코드 (RankingRedisService):
─────────────────────────────────────
redisTemplate.executePipelined(new SessionCallback<>() {
    public Object execute(RedisOperations ops) {
        ops.opsForZSet().incrementScore(key, member, 1);  // ZINCRBY
        ops.opsForZSet().score(key, member);               // ZSCORE
        return null;
    }
});
// results[0] = ZINCRBY 결과, results[1] = ZSCORE 결과
```

### 3.4 Lettuce 커넥션 풀링

```
Before: Lettuce 기본 (단일 커넥션 멀티플렉싱)
───────────────────────────────────────────────
Thread 1 ──┐
Thread 2 ──┤
Thread 3 ──┼──→ [Single Connection] ──→ Redis
...        │         (직렬 전송)
Thread 50 ─┘

문제: 50개 스레드가 1개 커넥션을 공유 → 전송 직렬화 병목


After: commons-pool2 풀링 (max-active: 20)
───────────────────────────────────────────────
Thread 1 ──→ [Conn 1] ──┐
Thread 2 ──→ [Conn 2] ──┤
Thread 3 ──→ [Conn 3] ──┼──→ Redis (싱글 스레드)
...                      │
Thread 20 → [Conn 20] ──┘
Thread 21~50: Pool에서 빈 커넥션 대기 후 사용

효과: 클라이언트 → Redis 전송 구간의 직렬화 완화
한계: Redis 서버 자체는 싱글 스레드 (서버 처리량은 동일)

설정 (application-dev.yml):
spring.data.redis.lettuce.pool:
  max-active: 20      # 최대 커넥션
  max-idle: 10        # 최대 유휴
  min-idle: 5         # 최소 유휴 (Warm 상태 유지)
  max-wait: 1000ms    # 풀 고갈 시 최대 대기
```

## 4. 컴포넌트 상세

### 4.1 VoteService.voteAsync() — API 진입점

```java
// 파일: service/VoteService.java

public VoteResponseDto.VoteResult voteAsync(Long entryId, Integer memberId) {

    // 1. Entry 조회 + 검증 (DB 1회)
    Entry entry = entryRepository.findById(entryId)
            .orElseThrow(() -> new GeneralException(ErrorStatus._NOTFOUND));
    validateVote(entry, memberId);  // 챌린지 ACTIVE? 본인 Entry?
    Long challengeId = entry.getChallenge().getId();

    // 2. 중복 투표 체크 (DB 1회 - HikariCP 30개 병렬)
    if (voteRepository.existsByEntryIdAndMemberId(entryId, memberId)) {
        throw new GeneralException(ErrorStatus._BAD_REQUEST);
    }

    // 3. Redis 랭킹 업데이트 (Pipeline 1회: ZINCRBY + ZSCORE)
    int currentVoteCount = rankingRedisService
            .incrementVoteAndGetScore(challengeId, entryId);

    // 4. SQS 비동기 전송 (Fire & Forget)
    VoteMessage message = VoteMessage.create(memberId, entryId, challengeId);
    voteProducer.sendVote(message);

    // 5. 즉시 응답 (Redis의 실시간 득표수 반환)
    return VoteResponseDto.VoteResult.builder()
            .entryId(entryId)
            .voteCount(currentVoteCount)
            .build();
}
```

```
호출 비용 분석 (50 VUs Hot Spot):
─────────────────────────────────
① DB findById:          ~5ms  (인덱스 PK 조회, 빠름)
② DB exists:            ~10ms (UK 인덱스 조회, HikariCP 병렬)
③ Redis Pipeline:       ~5ms  (ZINCRBY + ZSCORE, 1회 왕복)
④ SQS Fire & Forget:    ~0ms  (비동기, 응답 안 기다림)
─────────────────────────────────
합계: ~20ms (단일 요청)
50 VUs 동시: 커넥션 풀 경합으로 p95 = 658ms
```

### 4.2 VoteProducer — SQS Fire & Forget

```java
// 파일: messaging/VoteProducer.java

// 핵심: CompletableFuture.runAsync()로 SQS 응답을 기다리지 않음
CompletableFuture.runAsync(() -> {
```

```
동기 전송 vs Fire & Forget 차이:
─────────────────────────────────
동기: API → SQS.send(300ms) → Response  (300ms 추가)
비동기: API → CompletableFuture → Response (즉시)
                    └→ SQS.send(300ms)   (백그라운드)

트레이드오프:
- 장점: API 응답에서 SQS 지연 완전 제거
- 단점: 전송 실패 시 Redis에는 반영됐지만 DB에 안 들어감
- 보완: VoteConsistencyScheduler가 5분마다 불일치 감지 → DB 기준 복구
```

### 4.3 VoteConsumer — SQS → DB 영속화

```java
// 파일: messaging/VoteConsumer.java

@SqsListener("${app.sqs.vote-queue-name:petstar-votes}")
@Transactional
public void handleVote(String payload) {
    VoteMessage message = objectMapper.readValue(payload, VoteMessage.class);

    // Idempotency: 이미 DB에 있으면 무시
    if (voteRepository.existsByEntryIdAndMemberId(entryId, memberId)) {
        return;  // 중복 메시지 안전 처리
    }

    // Entry/Member 검증 (실패 시 Redis 롤백)
    Entry entry = entryRepository.findById(entryId).orElse(null);
    if (entry == null) {
        rollbackRedis(message);  // ZINCRBY -1
        return;
    }

    // DB 저장
    voteRepository.save(vote);
    entryRepository.incrementVoteCount(entryId);  // Atomic UPDATE

    // 성공 시 SQS 자동 ACK (메시지 삭제)
}
```

```
SQS Consumer 설정 (SqsConfig.java):
────────────────────────────────────
maxConcurrentMessages: 1   ← 동시 처리 1개 (데드락 방지)
maxMessagesPerPoll: 1      ← 폴링당 1개

이유: 같은 Entry에 대한 UPDATE가 동시에 실행되면 데드락 발생
     순차 처리로 완전히 제거 (처리량 < API이지만 사용자 체감 없음)
```

### 4.4 VoteConsistencyScheduler — 정합성 보장

```java
// 파일: service/VoteConsistencyScheduler.java

@Scheduled(fixedRate = 300000)  // 5분마다
public void verifyConsistency() {
    // 활성 챌린지 → Entry별 Redis vs DB 비교
    for (Entry entry : entries) {
        int dbCount = entry.getVoteCount();
        int redisCount = rankingRedisService.getVoteCount(challengeId, entryId);

        if (redisCount > dbCount) {
            // SQS 처리 지연 가능성 → 30초 대기 후 재검증
            Thread.sleep(30000);
            // 여전히 불일치 → DB 기준 동기화
        } else if (redisCount < dbCount) {
            // 데이터 유실 → 즉시 DB 기준 동기화
            rankingRedisService.initEntry(challengeId, entryId, dbCount);
        }
    }
}
```

```
불일치 시나리오와 대응:
────────────────────────────────────────────────────────────

Case 1: Redis > DB (정상일 수 있음)
───────────────────────────────────
원인: SQS Consumer가 아직 처리 안 함
대응: 30초 대기 → 재검증 → 여전히 불일치 시 DB 기준으로 Redis 덮어쓰기

Case 2: Redis < DB (비정상)
───────────────────────────────────
원인: Redis 재시작, 키 만료, 장애
대응: 즉시 DB 기준으로 Redis 동기화 (ZADD로 점수 설정)

Case 3: Redis = DB (정상)
───────────────────────────────────
대응: 없음 (skip)

Source of Truth: 항상 DB (Vote 테이블)
```

## 5. 데이터 흐름 & 정합성 모델

### 5.1 3계층 정합성

```
┌─────────────────────────────────────────────────────────────────┐
│                      정합성 계층                                  │
├──────────┬──────────┬───────────────────────────────────────────┤
│ 계층     │ 지연시간  │ 설명                                      │
├──────────┼──────────┼───────────────────────────────────────────┤
│ Redis    │ 즉시     │ 투표 API 응답 시점에 랭킹 반영 완료         │
│          │ (~5ms)   │ 사용자가 보는 랭킹 = Redis 기준             │
├──────────┼──────────┼───────────────────────────────────────────┤
│ DB       │ 수초     │ SQS Consumer가 비동기로 저장                │
│          │ (~3s)    │ Source of Truth (최종 기준)                 │
├──────────┼──────────┼───────────────────────────────────────────┤
│ 검증     │ 5분      │ Scheduler가 Redis ↔ DB 비교                │
│          │          │ 불일치 시 DB 기준 동기화                    │
└──────────┴──────────┴───────────────────────────────────────────┘

시간축으로 보면:

t=0ms     API 호출
t=5ms     Redis 랭킹 반영 ✓
t=10ms    API 응답 반환 (사용자에게 결과 표시)
t=10ms    SQS 메시지 전송 시작 (백그라운드)
t=310ms   SQS 메시지 도착
t=500ms   Consumer: DB에 Vote 저장 완료 ✓
...
t=300000ms  Scheduler: 정합성 검증 ✓
```

### 5.2 실패 복구 체인

```
실패 지점별 복구 전략:
──────────────────────────────────────────────────────

① SQS 전송 실패 (Fire & Forget 실패)
   상태: Redis ✓, DB ✗
   복구: Scheduler가 5분 후 감지 → DB 기준 Redis 동기화
   결과: Redis 점수가 DB보다 1 높음 → Redis를 DB 기준으로 리셋

② Consumer 처리 실패 (검증 실패)
   상태: Redis ✓, DB ✗
   복구: Consumer가 Redis 롤백 (ZINCRBY -1)
   결과: Redis와 DB 모두 투표 미반영

③ Consumer 데드락
   상태: Redis ✓, DB ✗
   복구: SQS visibility timeout 후 자동 재시도
   결과: 재시도로 DB 저장 성공

④ Consumer 재시도 초과
   상태: Redis ✓, DB ✗
   복구: DLQ(Dead Letter Queue)로 이동 → 수동 처리
   최종 보험: Scheduler가 감지 → DB 기준 동기화

⑤ Redis 재시작/키 유실
   상태: Redis ✗, DB ✓
   복구: Scheduler가 감지 → DB 기준 Redis 재구축
```

### 6.2 각 Phase에서 바뀐 것

```
Phase 6 → Phase 10 (p95: 1,730ms → 516ms)
═════════════════════════════════════════════
변경: DB Lock → Redis 즉시 + SQS 비동기 DB 저장
핵심: "DB 저장을 API 응답에서 분리"

Phase 10 → Phase 10-2 (p95: 516ms → 1,300ms) ← 실패!
═════════════════════════════════════════════
변경: DB 중복체크 → Redis SISMEMBER로 전환
원인: Lettuce 단일 커넥션에 4개 Redis 호출 직렬화
교훈: "Redis 호출 횟수가 성능을 결정한다 (단일 커넥션 환경)"

Phase 10-2 → Phase 10-2b (p95: 1,300ms → 688ms)
═════════════════════════════════════════════
변경: 중복체크를 다시 DB로 + SQS를 Fire & Forget으로
핵심: "각 컴포넌트의 강점을 살리는 역할 분담"
      DB: 중복체크 (HikariCP 30 병렬)
      Redis: 랭킹만 (2회 호출로 축소)
      SQS: 비동기 (Fire & Forget으로 0ms)

Phase 10-2b → Phase 10-3 (p95: 688ms → 658ms)
═════════════════════════════════════════════
변경: Lettuce 풀링 + Redis Pipeline
핵심: Redis 2회 호출 → Pipeline 1회로 합침
개선폭 작은 이유: 전체 응답의 ~1.7%만 Redis (병목이 DB 쪽)
```

### 6.3 Redis 중복체크 재시도 & 실패 분석

```
Phase 10-3 이후 추가 시도: Redis SADD로 중복체크 전환
═════════════════════════════════════════════════════

시도 1: SADD 별도 호출 (p95: 863ms) ← 역전
───────────────────────────────────────
흐름: Redis SADD → DB findById → Redis Pipeline(ZINCRBY+ZSCORE)
비용: 2 Redis RT + 1 DB = 기존보다 Redis 호출 증가
원인: Redis 싱글스레드에 호출이 집중

시도 2: SADD+ZINCRBY+ZSCORE 통합 Pipeline (p95: 836ms) ← 여전히 역전
───────────────────────────────────────
흐름: DB findById → Redis Pipeline(SADD+ZINCRBY+ZSCORE)
비용: 1 DB + 1 Redis RT (이론상 최적)
실측: 44명 549~618ms (빠름), 6명 834~837ms (느림)

HikariCP 모니터링: idle 21/30 → DB가 놀고 있음!
───────────────────────────────────────
해석: DB에서 Redis로 일을 옮겼더니
      → DB 커넥션 21개가 유휴 (낭비)
      → Redis에 50개 요청이 한번에 몰림 (병목)

결론: DB 중복체크가 오히려 "자연스러운 부하 분산" 역할
      → HikariCP 30개 커넥션이 요청을 시간차로 분산
      → Redis에 도착하는 시점이 자연스럽게 분산됨
      → Redis 병목 완화

┌──────────────────────────────────────────────────────────┐
│  교훈: "느린 컴포넌트를 제거하면 빨라진다"는 항상 참이 아님  │
│  DB 체크가 Redis 앞에서 "자연스러운 속도 조절기" 역할       │
│  이를 제거하면 Redis에 트래픽이 한번에 몰려서 오히려 악화    │
└──────────────────────────────────────────────────────────┘
```
